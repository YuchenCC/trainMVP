## 2. 项目背景与目标

### 2.1 项目背景

随着企业系统数量增加，软件发布逐渐从单系统发布转向多系统协同发布。多个系统之间存在需求依赖、投产窗口、测试节奏、团队容量和版本范围等协同问题。传统需求管理工具通常以单系统、单需求、单任务为中心，缺少跨系统统一发布计划能力，导致发布管理中容易出现需求范围失控、依赖关系不透明、团队容量不可见、关键节点不统一等问题。

版本火车是一种面向多系统协同发布的管理模式，通过固定发布周期、统一关键节点、明确纳版机制和容量约束，实现多个系统在统一节奏下协同交付。

本项目建设“版本火车需求管理系统”，用于支撑需求从提出、评审、就绪、纳版、变更、投产到回滚/取消的全生命周期管理，并在纳版阶段引入AI辅助排期建议能力，提升发布计划编排效率和风险识别能力。

### 2.2 项目目标

MVP阶段目标是完成一个可演示、可落地、可继续扩展的版本火车需求管理系统，重点实现以下目标：

1. 建立统一需求池，支持需求录入、分类、评审、状态流转、依赖关系和已纳版需求变更管理。
2. 建立版本火车计划机制，支持版本火车创建、搭载系统管理、团队容量管理和关键节点管控。
3. 支持火车管理员基于需求优先级、工作量点数、团队容量和跨系统依赖进行纳版决策。
4. 引入AI排期建议能力，输出建议纳入、建议延期、依赖风险和容量预估结果。
5. 形成适合AI Coding实施的清晰需求基线，为后续概要设计、详细设计和开发任务拆解提供依据。

### 2.3 MVP范围

| 模块 | MVP范围 |
|---|---|
| 需求池管理 | 需求录入、分类、评审、状态流转、已纳版需求管理、变更回退、依赖关系管理、需求列表与详情 |
| 版本火车计划 | 版本火车创建、搭载系统配置、团队容量配置、关键节点管理、日历视图 |
| 纳版管理 | 手动纳版、AI辅助纳版、容量校验、依赖校验、移除、投产、回滚 |
| AI排期建议 | 基于容量、优先级、工作量和依赖关系生成纳版建议、延期建议和风险提示 |
| 权限控制 | 基于角色的功能权限和关键数据操作权限 |
| 操作审计 | 记录评审、纳版、移除、变更、取消、投产、回滚等关键操作 |

### 2.4 非MVP范围

以下内容不纳入MVP阶段：

1. 完整研发任务管理。
2. 测试用例管理和缺陷管理。
3. CI/CD流水线集成。
4. 企业级流程审批引擎。
5. 复杂甘特图和资源排班。
6. 多项目组合管理。
7. 复杂数据分析报表和BI看板。
8. 企业级消息通知中心。
9. 需求拆解为研发任务的自动化Agent。

## 3. 项目价值分析

### 3.1 功能价值

本系统面向多系统协同发布场景，通过版本火车机制将需求管理、发布计划、团队容量、关键节点和跨系统依赖统一纳入一个管理闭环，解决传统需求管理工具偏单系统、偏静态记录、缺少发布节奏管理的问题。

#### 3.1.1 解决多系统协同发布痛点

| 痛点 | 传统方式问题 | 系统解决方式 |
|---|---|---|
| 发布节奏不统一 | 各系统独立规划，缺少统一版本节奏，版本对齐困难 | 通过固定周期版本火车统一发布节奏 |
| 需求优先级混乱 | 排期依赖会议讨论和主观判断，缺少容量依据 | 结合优先级、工作量点数和团队容量进行纳版决策 |
| 跨系统依赖失控 | 依赖关系依赖人工沟通，容易遗漏且不可追溯 | 支持跨系统前置依赖配置，并在纳版时自动检查 |
| 团队容量不可见 | 无法量化每个系统团队的承载能力 | 通过工作量点数、可用容量、已分配容量和容量使用率实现可视化管理 |
| 关键节点不清晰 | 需求冻结、封板、投产缺少统一时间基准 | 自动推算统一纳版、统一封板、统一投产节点 |
| 纳版后变更失控 | 需求纳版后继续修改，导致范围、容量和测试边界变化 | 已纳版需求变更自动回退未就绪，重新评审和重新纳版 |

#### 3.1.2 管理效率提升

| 维度 | 传统方式 | 版本火车方式 | 预期提升 |
|---|---|---|---|
| 发布规划 | 人工协调，周期不固定 | 固定周期，关键节点自动推算 | 规划效率提升约60% |
| 需求排期 | 会议讨论，主观判断 | AI辅助排期 + 人工确认 | 排期准确率提升约40% |
| 容量管理 | 无量化估算 | 工作量点数 + 实时容量监控 | 资源利用率提升约30% |
| 依赖管理 | 口头沟通，不可追溯 | 系统关联 + AI自动检查 | 依赖风险降低约50% |
| 变更控制 | 纳版后变更依赖人工同步 | 已纳版变更回退 + 操作审计 | 降低版本范围失控风险 |

> 上述指标为MVP阶段目标化收益假设，用于竞赛展示和后续试点评估。正式落地后可结合实际版本数据进行校准。

### 3.2 AI能力价值

本项目的AI能力定位为“辅助决策”，不替代火车管理员的最终判断。AI能力嵌入纳版决策场景，基于结构化输入输出结果，重点提升排期效率、风险识别和决策解释性。

| AI能力 | 业务输入 | AI输出 | 人工确认点 | 价值 |
|---|---|---|---|---|
| AI排期建议 | 已就绪需求、优先级、工作量点数、团队容量 | 建议纳入本车需求清单 | 火车管理员确认纳版 | 降低排期决策成本 |
| 依赖风险识别 | 需求依赖关系、依赖需求状态、搭载系统范围 | 依赖未满足、依赖已取消、依赖系统未参车等风险 | 火车管理员判断是否强制纳版 | 降低跨系统发布风险 |
| 容量预估 | 各系统可用容量、已分配容量、候选需求点数 | 各系统容量使用率预估 | 火车管理员调整纳版范围 | 提升资源利用透明度 |
| 延期建议 | 容量不足、优先级较低、依赖未满足的需求 | 建议延期清单和原因说明 | 火车管理员确认延期 | 提升排期解释性 |

### 3.3 AI原生开发竞赛价值

本项目适合作为AI原生开发竞赛选题，原因包括：

1. **业务复杂度适中**：包含多角色、状态机、权限体系、容量计算和跨系统依赖，能够体现AI对复杂业务规则的理解与实现能力。
2. **前后端链路完整**：覆盖数据库设计、接口设计、后端业务逻辑、前端页面和日历视图，能够展示AI Coding的全链路工程能力。
3. **AI Agent场景明确**：AI排期建议不是泛化问答，而是基于结构化业务输入输出的辅助决策场景，具备可解释、可验证、可落地的特点。
4. **具备量化评估基础**：可从开发效率、代码质量、需求覆盖率、测试覆盖率、AI建议命中率等维度评估AI开发效果。
5. **可沉淀复用方法论**：项目可总结结构化PRD生成、数据建模、状态机实现、RBAC权限设计、AI Agent设计和AI辅助测试等经验。

火车创建员负责“谁参加这趟车”，包括版本火车创建、系统搭载、团队成员和容量配置。

火车管理员负责“这趟车怎么运行”，包括关键节点、纳版、移除、投产、回滚、容量控制和已纳版变更治理。

| 优先级 | P0、P1、P2、P3 |

| 字段 | 说明 |

|---|---|

| 产品经理 | 必填 |
| 项目经理 | 可选 |
| 技术经理 | 必填 |

## 6. 需求池管理

### 6.1 功能目标

需求池用于集中管理所有需求，覆盖待处理需求、未就绪需求、已就绪需求、已纳版需求、已投产需求和已取消需求。需求池支持需求录入、分类、评审、状态跟踪、已纳版需求变更回退、依赖关系维护和待纳版筛选。

### 6.2 需求录入规则

1. BA和产品经理均可新增需求。
2. 需求标题、需求描述、归属系统、优先级、工作量点数、业务归属人、产品经理为必填项。
3. 工作量点数范围为1-100。
4. 新建需求默认为草稿状态。
5. 需求可配置多个前置依赖，依赖需求可以跨系统。
6. 依赖关系作为纳版风险判断依据，不强制阻断需求录入。

### 6.3 依赖需求规则

1. 一个需求可关联多个前置依赖。
2. 依赖需求必须是系统内已存在需求。
3. 当前置依赖状态为“已纳版”或“已投产”时，视为依赖已满足。
4. 当前置依赖状态为“已取消”时，系统应给出风险预警。
5. 当前置依赖未纳入当前版本火车时，AI排期建议和手动纳版时应提示风险。
6. 存在依赖风险时允许火车管理员强制纳版，但必须进行风险提示和操作记录。

### 6.4 需求状态流转

#### 6.4.1 主流程

```text
草稿 → 待评审 → 已就绪 → 已纳版 → 已投产
```

#### 6.4.2 分支流程

```text
待评审 → 已拒绝 → 草稿
已纳版需求发生实质变更 → 未就绪 → 待评审 → 已就绪
已就绪 → 已纳版 → 已就绪
已纳版 → 已投产 → 已就绪
任意非取消状态 → 已取消
```

“未就绪”用于承接已纳版需求发生变更后的回退状态，表示该需求当前不再满足直接纳版或继续执行条件，需要补充信息、重新评估影响，并重新发起评审。

### 6.5 状态说明

| 状态 | 含义 | 可转换状态 |
|---|---|---|
| 草稿 | 初始录入状态，可编辑 | 待评审、已取消 |
| 待评审 | 已提交评审，等待确认 | 已就绪、已拒绝、已取消 |
| 未就绪 | 已纳版需求发生实质变更后的回退状态，需重新补充信息和评审 | 待评审、已取消 |
| 已就绪 | 评审通过，可进入版本规划 | 已纳版、已取消 |
| 已拒绝 | 评审未通过，需修改后重新提交 | 草稿、已取消 |
| 已纳版 | 已搭载到版本火车 | 未就绪、已就绪、已投产、已取消 |
| 已投产 | 已上线发布 | 已就绪、已取消 |
| 已取消 | 需求已取消或终止 | 无 |

### 6.6 状态操作规则

| 操作 | 状态变化 | 权限 | 关键规则 |
|---|---|---|---|
| 发起评审 | 草稿/未就绪 → 待评审 | 业务归属人BA | 校验必填项完整；记录操作人和时间 |
| 评审通过 | 待评审 → 已就绪 | 产品经理、项目经理 | 记录评审人、评审时间、评审意见 |
| 评审拒绝 | 待评审 → 已拒绝 | 产品经理、项目经理 | 拒绝原因必填，最多500字 |
| 重新编辑 | 已拒绝 → 草稿 | 业务归属人BA、产品经理 | 回到草稿后可继续编辑 |
| 取消需求 | 任意非取消状态 → 已取消 | 业务归属人BA、火车管理员 | 取消原因必填，最多500字；已纳版需求取消时自动从火车移除 |
| 已纳版需求变更 | 已纳版 → 未就绪 | 业务归属人BA、产品经理、火车管理员 | 变更原因必填，记录变更内容；从当前版本火车移除并释放容量；需重新评审后才能再次纳版 |

### 6.7 已纳版需求变更管理

已纳版需求仍属于需求池管理范围。为避免版本范围失控，已纳版需求发生实质变更时，必须触发状态回退。

#### 6.7.1 触发条件

以下字段发生变更时，视为实质变更：

1. 需求标题或需求描述发生影响范围的修改。
2. 优先级发生变化。
3. 工作量点数发生变化。
4. 归属系统发生变化。
5. 依赖需求新增、删除或变更。
6. 业务归属人或产品经理发生变化。
7. 火车管理员认定会影响本次版本范围、容量或投产风险的其他变更。

#### 6.7.2 流转规则

1. 已纳版需求发生实质变更后，状态由“已纳版”回退为“未就绪”。
2. 系统自动解除该需求与当前版本火车的纳版关系。
3. 系统释放该需求占用的系统容量。
4. 变更人必须填写变更原因，最多500字。
5. 系统记录变更前后内容、变更人、变更时间和变更原因。
6. 未就绪需求需补充完整信息后重新发起评审。
7. 重新评审通过后，状态变更为“已就绪”，可再次进入纳版。

#### 6.7.3 权限规则

| 操作 | 权限 |
|---|---|
| 发起已纳版需求变更 | 业务归属人BA、产品经理、火车管理员 |
| 确认回退未就绪 | 系统自动执行 |
| 重新发起评审 | 业务归属人BA |
| 再次纳版 | 火车管理员 |

#### 6.7.4 业务价值

已纳版需求变更回退机制能够保证版本火车范围受控，避免需求在纳版后被随意修改导致容量失真、依赖失控、测试范围扩大和投产风险上升。

### 6.8 需求池页面要求

#### 6.8.1 需求列表页

需求列表页用于查询和管理需求，支持：

1. 按系统切换需求列表。
2. 按状态、创建人、版本火车筛选。
3. 展示需求标题、状态、优先级、创建人、提出时间、所属版本火车等核心信息。
4. 根据当前用户角色和需求状态动态展示操作按钮。
5. 支持在需求池中查看和筛选“已纳版”需求，避免纳版后需求脱离需求池管理。

#### 6.8.2 需求详情页

需求详情页展示需求完整信息，支持：

1. 查看基础信息、需求描述、依赖需求、操作历史。
2. 根据权限执行编辑、发起评审、重新编辑、取消、已纳版变更等操作。
3. 依赖需求以状态标识展示：依赖已满足、依赖未满足、依赖已取消。
4. 点击依赖需求可跳转到对应详情。
5. 对已纳版需求，如发生需求描述、优先级、工作量点数、依赖关系、归属系统等实质变更，系统应提示该需求将回退为“未就绪”，并要求填写变更原因。

#### 6.8.3 需求评审页

需求评审页用于产品经理和项目经理处理待评审需求，支持：

1. 查看待评审需求列表。
2. 查看需求详情。
3. 填写评审意见。
4. 执行评审通过或评审拒绝。
5. 支持批量通过和批量拒绝；批量拒绝时需统一填写拒绝原因。

## 7. 版本火车计划管理

### 7.1 功能目标

版本火车计划模块用于创建和管理固定周期发布计划，支持搭载系统、配置容量、管控关键节点、执行纳版、投产和回滚。

### 7.2 版本火车创建规则

1. 火车创建员可创建版本火车。
2. 创建时需填写版本火车名称、开始时间、版本周期和描述。
3. 版本周期由超级管理员全局配置，支持2周、3周、4周。
4. 开始时间默认最近一个周一。
5. 结束时间根据开始时间和周期自动计算。
6. 同一系统同一时间只能参与一个版本火车。

### 7.3 搭载系统与容量规则

1. 火车创建员在创建版本火车时选择本车搭载系统。
2. 每个搭载系统必须配置业务BA、产品经理、技术经理、开发团队和测试负责人。
3. 每个搭载系统必须配置本期可用点数。
4. 已纳版需求点数计入对应系统的已分配点数。
5. 容量使用率达到90%及以上时应预警。
6. 容量已满或超出时，仍允许火车管理员强制纳版，但必须二次确认。
7. 已纳版需求因变更回退为未就绪时，系统自动释放对应容量。

### 7.4 关键节点规则

版本火车包含三个关键节点：统一纳版、统一封板、统一投产。

| 节点 | 含义 | 默认时间 |
|---|---|---|
| 统一纳版 | 确定进入本车的需求清单 | 周期过半前最后一个周五 |
| 统一封板 | UAT完成后控制变更 | 投产前一周周五 |
| 统一投产 | 版本火车正式发布上线 | 周期最后一天周五 |

不同周期默认节点：

| 周期 | 统一纳版 | 统一封板 | 统一投产 |
|---|---|---|---|
| 2周 | 第1周周五 | 第2周周五 | 第2周周五 |
| 3周 | 第2周周五 | 第3周周五 | 第3周周五 |
| 4周 | 第2周周五 | 第4周周五 | 第4周周五 |

节点管控规则：

1. 节点时间由系统自动推算，火车管理员可在±3天内微调。
2. 统一纳版前，可自由搭载和移除需求。
3. 统一纳版至统一封板之间，新增纳版需火车管理员确认。
4. 统一封板后，原则上不再接受新需求，仅支持紧急变更。
5. 到达统一投产节点时，系统自动将本车所有“已纳版”需求批量变更为“已投产”。

### 7.5 纳版管理

#### 7.5.1 手动纳版

火车管理员可将已就绪需求搭载到版本火车。

规则：

1. 需求状态必须为“已就绪”。“未就绪”需求不可直接纳版，必须重新评审通过后才能纳版。
2. 需求归属系统必须在当前版本火车搭载系统范围内。
3. 需求必须填写工作量点数。
4. 系统应校验依赖是否满足，未满足时提示风险但允许强制纳版。
5. 系统应校验容量是否充足，不足时提示风险但允许强制纳版。
6. 纳版成功后，需求状态变更为“已纳版”，并关联版本火车。
7. 纳版成功后，对应系统已分配点数增加。
8. 系统记录纳版时间和操作人。

#### 7.5.2 从火车移除

火车管理员可将已纳版需求从版本火车移除。

规则：

1. 需求状态必须为“已纳版”。
2. 移除原因必填，最多500字。
3. 移除后需求状态变更为“已就绪”。
4. 移除后释放对应系统容量。
5. 系统记录移除时间、操作人和移除原因。

#### 7.5.3 确认投产

火车管理员可手动确认需求投产，也可由系统在统一投产节点自动批量投产。

规则：

1. 需求状态必须为“已纳版”。
2. 投产后需求状态变更为“已投产”。
3. 系统记录投产时间和操作人。
4. 到达统一投产节点时，系统自动批量处理本车全部已纳版需求。

#### 7.5.4 回滚

火车管理员可将已投产需求回滚。

规则：

1. 需求状态必须为“已投产”。
2. 回滚原因必填，最多500字。
3. 回滚后需求状态变更为“已就绪”。
4. 系统记录回滚时间、操作人和回滚原因。

## 8. AI排期建议

### 8.1 功能定位

AI排期建议用于辅助火车管理员进行纳版决策，不直接替代人工决策。AI输出仅作为建议，最终纳版需由火车管理员确认。

### 8.2 触发方式

火车管理员在版本火车详情页点击“AI排期建议”按钮后触发。

### 8.3 输入数据

| 输入 | 说明 |
|---|---|
| 版本火车信息 | 当前版本火车ID、周期、搭载系统 |
| 团队容量 | 各系统可用点数、已分配点数、剩余点数 |
| 已就绪需求 | 需求标题、归属系统、优先级、工作量点数、依赖关系 |
| 依赖需求状态 | 依赖需求是否已纳版、已投产、已取消或未满足 |

### 8.4 建议逻辑

1. 优先检查需求依赖关系。
2. 依赖已取消的需求标记为风险。
3. 依赖未满足的需求建议延期或提示需联动纳入依赖需求。
4. 按优先级排序：P0 > P1 > P2 > P3。
5. 同优先级需求按工作量点数从小到大排序。
6. 按系统维度结合剩余容量进行分配。
7. 容量充足的需求建议纳入本车。
8. 容量不足的需求建议延期。
9. 若依赖需求所属系统未参与当前版本火车，应提示火车创建员补充搭载系统或调整计划。
10. 未就绪需求不纳入AI排期候选范围，仅在风险说明中提示需先完成评审。

### 8.5 输出结果

AI排期建议页应输出：

1. 建议纳入本车需求清单。
2. 建议延期需求清单及延期原因。
3. 依赖联动建议。
4. 依赖风险提示。
5. 各系统容量使用率预估。
6. AI总结建议。

### 8.6 人工确认

火车管理员可在AI建议结果中勾选部分或全部建议纳入需求，并点击确认纳版完成批量纳版。

### 8.7 AI能力边界

1. AI只提供排期建议，不自动变更需求状态。
2. AI不直接执行纳版、延期、取消、移除或回滚操作。
3. AI建议必须展示原因，便于火车管理员判断。
4. AI建议结果允许人工调整。
5. AI建议结果和人工确认结果均应记录，便于后续复盘。
