## test-strategy-planner 待调整内容

### P0 - 核心逻辑修复
| 调整项 | 说明 |
|-------|------|
| L1 单测判断逻辑 | 当前认为 Service 依赖 Prisma 就无 L1 单测，需识别纯函数（如 `calculateKeyDates`）和可 mock 场景（分页计算、状态机校验） |
| 章节顺序 | 策略应放在前面，遵循"结论 → 现状 → 依据 → 行动"逻辑 |

### P1 - 职责边界调整
| 调整项 | 说明 |
|-------|------|
| 删除门禁策略章节 | "单元测试门禁策略"不属于 test-strategy-planner 的职责，应交给 `unit-test-governance` |
| 合并重复章节 | 分层测试策略与覆盖矩阵重复，精简分层策略章节 |

### P2 - 文档结构优化
| 调整项 | 说明 |
|-------|------|
| 章节标题优化 | 使用更直观的标题，如"分层测试方案"替代"分层测试策略" |
| 合并同类章节 | 现有测试资产判断 → 合并到覆盖现状；后续执行建议 → 合并到 TODO 清单 |

### P3 - 内容增强
| 调整项 | 说明 |
|-------|------|
| 代码影响分析 | 每个代码文件列出影响的测试案例和必须覆盖原因 |
| 人工验证方案 | 明确哪些 UI 场景可用人工验证替代、需要什么证据 |
| 后续 Skill 调用链 | 明确交由哪个 skill 执行，形成完整的 skill 调用链 |

## 推荐的输出结构

```markdown
1. 分析范围（背景）
2. 分层测试方案（结论）← 核心输出，放在前面
3. 覆盖现状（现状统计）
4. 需求-案例覆盖矩阵（三方映射）
5. 代码影响测试矩阵（代码 → 案例映射）
6. 现有测试资产判断（可复用/不足）
7. TODO 清单与后续执行（优先级 + Skill 调用链）
8. 人工验证方案（UI 场景）
9. 风险提醒（潜在风险）
```
