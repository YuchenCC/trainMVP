## test-strategy-planner 待调整内容

### P0 - 核心逻辑修复
| 调整项 | 说明 |
|---|---|
| L1 单测判断逻辑 | 不能因为 Service 依赖 Prisma、HTTP、外部 SDK 就判断无 L1 单测价值；需识别纯函数（如 `calculateKeyDates`）、分页计算、字段映射、状态机校验、权限矩阵和可 mock 场景 |
| 章节顺序 | 策略应放在前面，遵循“结论 → 现状 → 依据 → 行动”逻辑，让项目经理、测试经理和研发负责人先看到怎么测、缺什么、风险在哪 |

### P1 - 职责边界与落地性
| 调整项 | 说明 |
|---|---|
| 单测门禁口径 | 不删除门禁提醒；planner 需要输出默认口径：增量覆盖率 > 80%、通过率 100%、L2/L3/人工验证不能抵扣，但最终是否通过交给 `unit-test-governance` 判定 |
| 代码影响分析 | 提升为 P1：研发要能看到“代码文件/模块 → 影响逻辑 → 对应测试案例 → 推荐测试层级 → 为什么必须覆盖” |
| 后续 Skill 调用链 | 提升为 P1：planner 是入口，需要明确下一步交给哪个 skill 执行，形成 L1/L2/L3/报告的闭环 |

### P2 - 文档结构优化
| 调整项 | 说明 |
|---|---|
| 报告精简 | 不单独输出大段“单元测试门禁策略”审计章节，门禁口径放入“分层测试方案”和“TODO 清单与后续执行” |
| 章节标题优化 | 使用更直观的标题，如“分层测试方案”替代“分层测试策略” |
| 合并同类章节 | 现有测试资产判断可合并到覆盖现状；后续执行建议可合并到 TODO 清单 |

### P3 - 内容增强
| 调整项 | 说明 |
|---|---|
| 人工验证方案 | 保留并增强；明确哪些低风险 UI 场景可用 SIT 人工验证替代、需要什么证据 |
| 轻量 evidence | 默认只创建 `evidence/{scope}/summary.md` 和公共附件目录；只有明确要求正式验收、审计、完整证据包时才按案例拆目录 |

## 推荐的输出结构

```markdown
1. 分析范围（背景）
2. 分层测试方案（结论，含单测门禁口径提醒）
3. 覆盖现状（现状统计，可合并现有测试资产判断）
4. 需求-案例覆盖矩阵（三方映射）
5. 代码影响测试矩阵（代码 → 案例映射）
6. 现有测试资产判断（可复用/不足；必要时可并入覆盖现状）
7. TODO 清单与后续执行（优先级 + Skill 调用链）
8. 人工验证方案（UI 场景 + evidence/summary.md）
9. 风险提醒（潜在风险）
```

## 关键口径

- planner 负责规划和路由，不负责执行测试，也不负责判定单测门禁最终是否通过。
- planner 必须提醒默认单测门禁：增量覆盖率 > 80%，通过率 100%，L2/L3/人工验证不能抵扣。
- 门禁是否通过、哪些代码必须补单测、现有单测是否可运行，交给 `unit-test-governance`。
- 代码影响分析和后续 Skill 调用链不是可选装饰项，是让研发真正能落地的核心输出。
- 人工验证用于替代低风险 UI 自动化，不替代 L1 单测和关键 L2 API 契约测试。
