# AI智能纳版能力设计与可解释性

**版本号**: v1.0  
**日期**: 2026-05-28  
**对应 Issue**: ISSUE-007  
**竞赛定位**: 需求智能分析与 AI 辅助决策亮点

---

## 一、文档目标

本文档说明智能纳版能力的业务目标、输入输出、规则引擎、AI 解释层、人工确认机制和测试场景。该能力是项目在 AI 原生研发能力攻关赛道中的重点展示内容之一。

## 二、业务目标

火车管理员在班次纳版时，需要同时考虑：

- 候选需求是否已就绪。
- 需求优先级是否合理。
- 当前班次各系统剩余容量是否足够。
- 需求之间是否存在前置依赖。
- 依赖是否已纳版、已投产、未满足或已取消。
- 是否存在需要人工确认的风险。

智能纳版的目标是将这些判断结构化，并通过 AI 生成可解释建议，降低人工排期成本。

## 三、设计原则

| 原则 | 说明 |
|------|------|
| 规则先行 | 纳入和延期结果由确定性规则产生 |
| AI解释 | AI 负责解释、总结和表达风险 |
| 人工确认 | 火车管理员确认后才真正改变需求状态 |
| 风险可见 | 容量和依赖风险必须显式展示 |
| 过程可审计 | 确认纳版后写入状态日志和容量变化 |

## 四、端到端流程

```text
火车管理员选择候选需求
  -> 点击 AI 智能纳版
  -> 后端读取班次容量和候选需求
  -> 构造 AI/规则输入
  -> 规则引擎排序和容量分配
  -> AI 生成解释与总结
  -> 前端展示建议列表、容量分析、风险警告
  -> 火车管理员选择确认项
  -> 后端执行纳版
  -> 更新需求状态、子状态、班次容量快照、状态日志
```

## 五、输入模型

智能纳版输入包含：

| 字段 | 说明 |
|------|------|
| `scheduleId` | 目标班次 ID |
| `requirementIds` | 候选需求 ID 列表 |
| `trainSchedule` | 班次名称、总容量、已用容量、剩余容量 |
| `selectedRequirements` | 需求编号、标题、优先级、点数、系统、状态、依赖 |

示例输入结构：

```json
{
  "trainSchedule": {
    "name": "2026年Q2第1班",
    "totalCapacity": 100,
    "usedCapacity": 30,
    "remainingCapacity": 70
  },
  "selectedRequirements": [
    {
      "reqCode": "REQ-2026-0101",
      "title": "用户登录优化",
      "priority": "P0",
      "storyPoints": 8,
      "system": "用户中心",
      "status": "READY",
      "dependencies": []
    }
  ]
}
```

## 六、输出模型

输出包含：

| 字段 | 说明 |
|------|------|
| `success` | 建议生成是否成功 |
| `summary` | AI 总结 |
| `capacityAnalysis` | 容量是否足够、超出点数 |
| `suggestions` | 每个需求的建议结果、排序、理由 |
| `warnings` | 容量、依赖、循环依赖等风险 |

建议结果通常包括：

- `recommended`：建议纳入。
- `not_recommended`：建议延期或不纳入。
- `optional`：可人工判断。

## 七、规则引擎

### 7.1 优先级排序

优先级顺序：

```text
P0 -> P1 -> P2 -> P3
```

同优先级内按工作量点数降序排列，优先让大需求进入排期判断。

### 7.2 容量分配

采用“可装则装”策略：

1. 按排序后的候选需求逐个判断。
2. 如果剩余容量足够，则建议纳入并扣减剩余容量。
3. 如果容量不足，则建议延期。
4. 某个大需求放不下，不阻塞后续小需求继续判断。

该策略避免“高优先级大需求占不下导致后续所有需求都无法被建议”的低效情况。

### 7.3 依赖处理

| 依赖状态 | 处理方式 |
|----------|----------|
| 已投产 | 依赖满足 |
| 已纳版 | 依赖基本满足 |
| 已就绪但未纳入本班次 | 提示 warning 风险 |
| 草稿/待评审/已拒绝 | 提示 high 风险 |
| 已取消 | 提示 critical 风险 |
| 循环依赖 | 数据一致性错误，应阻断保存或生成明确警告 |

依赖联动只提示人工评估，不自动纳入依赖闭包。

### 7.4 状态校验

只有 `READY` 状态需求可正常纳版。非 `READY` 状态需求应被标记为不推荐，并说明原因。

## 八、AI解释层

AI 的职责是：

- 用自然语言解释为什么建议纳入或延期。
- 总结容量使用情况。
- 提醒依赖风险。
- 帮助火车管理员理解排序和取舍。

AI 不做以下事情：

- 不直接修改需求状态。
- 不绕过规则引擎。
- 不自动确认风险。
- 不自动纳入依赖需求。

这种设计使 AI 能力可控、可解释、可审计，适合竞赛中强调“安全可信的 AI 原生研发”。

## 十、测试场景

| 场景 | 期望 |
|------|------|
| 容量充足 | 全部候选需求 recommended |
| 容量刚好 | 全部可纳入，无超限 |
| 容量不足 | 部分 not_recommended，并给出 exceededBy |
| 剩余容量为 0 | 全部不推荐 |
| P0/P1/P2/P3 混合 | 按优先级排序 |
| 同优先级 | 按点数降序 |
| 简单依赖 | 被依赖方排在依赖方之前 |
| 链式依赖 | 拓扑顺序正确 |
| 依赖已取消 | 生成 critical 依赖风险 |
| 非 READY 需求 | 不推荐纳版 |

## 十一、竞赛演示建议

推荐准备四条演示路线：

1. **正常纳版**：容量充足，AI 给出全部纳入建议。
2. **容量超限**：剩余容量不足，AI 解释哪些需求延期。
3. **依赖排序**：A 依赖 B，B 排在 A 前。
4. **依赖风险**：依赖已取消，AI 给出 critical 风险但不自动阻断人工决策。

## 十二、加分点表达

该能力可对应赛道加分点：

- 需求智能分析：自动分析需求优先级、点数和依赖。
- 安全研发：AI 不直接执行高风险业务变更。
- 非功能性需求：可解释、可审计、可控。
- AI 原生研发：AI 能力嵌入真实业务流程，而不是独立演示组件。
