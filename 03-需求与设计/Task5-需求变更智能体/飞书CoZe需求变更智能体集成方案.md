## 二、CoZe 智能体配置

### 2.1 智能体基础设置

| 配置项 | 设置值 | 说明 |
|-------|-------|------|
| 智能体名称 | 需求变更助手 | - |
| 头像 | 📋 | 使用文档/剪贴板图标 |
| 描述 | 自动分析对话中的需求变更，生成确认消息并同步到版本火车系统 | - |
| 可见范围 | 仅自己 | 个人号使用 |

### 2.2 人设与回复逻辑（系统提示词）

```
你是「需求变更助手」，专门帮助产品经理分析和确认需求变更。

## 核心能力
1. 分析对话内容，识别需求变更点（新增/修改/删除/范围/时间/优先级）
2. 对比历史需求，明确变更前后差异
3. 评估变更影响（工作量、进度、风险）
4. 生成结构化的变更摘要和确认消息

## 输入格式
用户会提供以下信息：
- 对话内容：产品/业务沟通记录
- 需求背景：当前迭代/版本信息
- 历史需求：已有的需求清单（可选）

## 输出要求
必须输出以下三部分内容：

### 1️⃣ 结构化摘要（JSON格式）
```json
{
  "change_id": "CR-YYYYMMDD-XXX",
  "version": "版本号",
  "changes": [
    {
      "type": "新增|修改|删除|范围|时间|优先级",
      "item": "需求项名称",
      "before": "变更前状态",
      "after": "变更后状态",
      "impact": "影响评估",
      "workload": "工作量变化(+/-X人天)"
    }
  ],
  "total_impact": {
    "workload": "总工作量变化",
    "schedule": "进度影响",
    "risk_level": "高|中|低"
  },
  "risks": [
    {
      "type": "技术|进度|资源|沟通",
      "level": "高|中|低",
      "description": "风险描述",
      "suggestion": "应对建议"
    }
  ]
}
```

### 2️⃣ 确认消息（Markdown格式）
生成可直接发送给需求提出方的确认消息，包含：
- 变更内容总结（带✅标记）
- 影响说明（工作量、进度）
- 风险提示（⚠️标记）
- 确认请求

### 3️⃣ 变更日志条目
按版本火车系统格式生成变更记录

## 注意事项
- 如果未检测到变更，输出 {"no_change": true, "reason": "原因"}
- 影响评估使用"预计"、"可能"等不确定措辞
- 风险提醒要具体，给出应对建议
```

### 2.3 工作流设计

```
┌─────────────────────────────────────────────────────────────┐
│                        CoZe 工作流                           │
└─────────────────────────────────────────────────────────────┘

[开始]
  │
  ▼

  │

                │

需要申请的权限：

## 四、版本火车系统API对接

### 4.1 API接口设计

#### 4.1.1 创建变更单

```http
POST /api/v1/change-requests
Content-Type: application/json
Authorization: Bearer {token}

{
  "version_id": "版本ID",
  "change_id": "CR-20240527-001",
  "title": "变更标题",
  "description": "变更描述",
  "type": "新增|修改|删除|范围|时间|优先级",
  "status": "待确认",
  "proposer": {
    "name": "提出人姓名",
    "department": "部门"
  },
  "impact": {
    "workload_days": 2.5,
    "schedule_delay_days": 1,
    "affected_modules": ["模块A", "模块B"]
  },
  "risks": [
    {
      "type": "技术",
      "level": "中",
      "description": "风险描述"
    }
  ],
  "created_at": "2024-05-27T14:30:00Z"
}
```

#### 4.1.2 更新需求状态

```http
PUT /api/v1/requirements/{requirement_id}
Content-Type: application/json
Authorization: Bearer {token}

{
  "status": "已变更",
  "change_id": "CR-20240527-001",
  "updated_fields": {
    "field_name": "新值"
  }
}
```

#### 4.1.3 获取历史需求

```http
GET /api/v1/versions/{version_id}/requirements
Authorization: Bearer {token}
```

### 4.2 CoZe插件配置

在CoZe中创建「版本火车系统」插件：

```yaml
插件名称: 版本火车系统
插件描述: 与版本火车管理系统交互，创建变更单和更新需求状态

    参数:
      - 名称: version_id
        类型: string
        描述: 版本ID
        必填: true
      - 名称: change_data
        类型: object
        描述: 变更数据（JSON格式）
        必填: true
        
  - 名称: update_requirement
    描述: 更新需求状态
    参数:
      - 名称: requirement_id
        类型: string
        描述: 需求ID
        必填: true
      - 名称: update_data
        类型: object
        描述: 更新数据
        必填: true
        
  - 名称: get_requirements
    描述: 获取版本下的需求列表
    参数:
      - 名称: version_id
        类型: string
        描述: 版本ID
        必填: true
```

## 五、完整使用流程

### 5.1 触发方式

#### 方式一：飞书个人会话触发（推荐）

```
用户 ──→ 飞书个人会话 ──→ 发送对话内容 ──→ CoZe智能体分析
```

**操作步骤：**
1. 与业务方/研发团队沟通后，复制对话内容
2. 在飞书个人会话中@需求变更助手
3. 粘贴对话内容，附加版本信息
4. 智能体分析并输出结果

**输入格式示例：**
```
@需求变更助手

【版本】V2.5.0 迭代3
【对话内容】
业务方：这个报表功能还需要加一个导出Excel的功能
产品：这个之前没排进去，现在加的话可能要延期
业务方：延期多久？
产品：大概2天吧
业务方：那按钮颜色改成红色，突出一点
产品：这个简单，半天搞定
业务方：对了，短信通知先不做了，预算不够
产品：那正好，省1天工作量
```

#### 方式二：飞书群聊触发

```
群聊对话 ──→ 用户@机器人 ──→ CoZe智能体分析 ──→ 群内推送结果
```

### 5.2 智能体处理流程

```
┌─────────────────────────────────────────────────────────────┐
│                        处理流程                              │
└─────────────────────────────────────────────────────────────┘

Step 1: 解析输入
  ├── 提取版本信息
  ├── 提取对话内容
  └── 提取历史需求（如有）

Step 2: 变更分析
  ├── 检测变更点
  ├── 对比历史需求
  ├── 评估影响
  └── 识别风险

Step 3: 生成输出
  ├── JSON结构化数据
  ├── Markdown确认消息
  └── 变更日志条目

Step 4: 同步操作
  ├── 发送飞书个人确认消息
  ├── 发送飞书群聊摘要卡片
  └── 调用版本火车API创建变更单

Step 5: 返回结果
  └── 汇总处理结果给用户
```

## 六、部署配置

### 6.1 CoZe 智能体配置步骤

1. **登录 CoZe 平台** (https://www.coze.cn)
2. **创建智能体**
   - 点击「创建智能体」
   - 选择「从零开始创建」
   - 填写名称：需求变更助手
3. **配置人设与回复逻辑**
   - 粘贴上面的系统提示词
4. **添加插件**
   - 添加「版本火车系统」插件（需先创建）
   - 添加「飞书」插件（用于发送消息）
5. **配置工作流**
   - 创建新工作流
   - 按照上面的工作流设计配置节点
6. **发布智能体**
   - 点击「发布」
   - 选择发布到飞书

### 6.2 飞书应用配置步骤

1. **创建应用**
   - 登录飞书开放平台
   - 创建企业自建应用
2. **配置权限**
   - 申请所需的API权限
   - 发布版本并通过审核
3. **获取凭证**
   - 记录 App ID 和 App Secret
   - 用于CoZe插件调用
4. **配置事件订阅**
   - 设置请求地址
   - 验证加密密钥

### 6.3 版本火车系统配置

1. **开放API接口**
   - 确保系统提供RESTful API
   - 支持变更单的CRUD操作
2. **配置API认证**
   - 生成API Token
   - 配置IP白名单
3. **提供接口文档**
   - 接口地址
   - 请求/响应格式
   - 错误码说明

## 八、进阶功能

### 8.1 智能推荐

基于历史数据，智能体可以：
- 推荐类似变更的处理方案
- 预测变更可能带来的连锁影响
- 建议最佳沟通话术

### 8.2 数据分析

统计指标：
- 变更频率趋势
- 变更类型分布
- 平均延期天数
- 风险发生概率

### 8.3 自动化升级

- 变更确认后自动更新项目计划
- 高风险变更自动通知相关负责人
- 定期生成变更报告

## 九、注意事项

1. **数据安全**
   - API通信使用HTTPS
   - Token定期更换
   - 敏感信息脱敏处理

2. **异常处理**
   - 网络异常时重试机制
   - API失败时人工介入流程
   - 数据一致性校验

3. **用户体验**
   - 处理时间控制在5秒内
   - 提供清晰的操作反馈
   - 支持人工修正和补充
