## US-T: T仪表盘

### 用户故事

> **作为** 版本火车系统的各类角色用户（BA、PM、项目经理、技术经理、测试经理、火车管理员）
> **我希望** 登录后首先看到一个聚合我角色相关的需求状态、待办事项和关键节点的仪表盘
> **以便** 快速了解当前工作全貌，无需逐个模块翻找

### 模块定位

T仪表盘是「角色工作台」，不按角色分页面路由，所有角色访问同一个 `/dashboard`，数据内容根据角色动态变化。后端全部只读聚合查询，无副作用。

### 重点功能

#### 1. 角色自适应统计卡片

根据当前用户角色，展示不同维度的需求状态统计。点击卡片跳转需求池对应筛选视图。

| 角色 | 核心卡片 | 说明 |
|------|----------|------|
| BA | 我的草稿、评审中、已就绪、已纳版 | 聚焦个人需求进度 |
| PM | 待评审、已就绪、已纳版 | 聚焦评审和就绪状态 |
| 项目经理 | 待评审、紧急变更待审批 | 聚焦审批决策 |
| 技术经理 | 待评审、待开发完成 | 聚焦开发进度 |
| 测试经理 | 紧急变更待审批、待 SIT 通过 | 聚焦测试进度 |
| 火车管理员 | 待纳版、待投产、变更率 | 聚焦火车运营 |

**统计维度**：
- 草稿数 / 待评审数 / 已就绪数 / 已纳版数 / 已投产数 / 已取消数
- 支持系统筛选器联动（顶部 `SystemSelector`）
- 点击卡片跳转 `/requirements?status={状态}`

#### 2. 角色待办列表

按角色展示不同待办区，每个区显示对应状态的需求列表（标题、系统、优先级、创建时间），点击跳转需求详情。

```
BA 待办区：
  - 「审核拒绝」：需重新编辑后提交
  - 「需重新提交」：变更通过后需重新编辑提交审核

PM 待办区：
  - 「待评审需求」

项目经理待办区：
  - 「待评审需求」
  - 「紧急变更待审批」（含通过/驳回按钮）

技术经理待办区：
  - 「待评审需求」
  - 「待开发完成」（含标记开发完成按钮）

测试经理待办区：
  - 「紧急变更待审批」
  - 「待 SIT 通过」（含标记 SIT 通过按钮）

火车管理员待办区：
  - 「待纳版需求」
  - 「待投产需求」
```

#### 3. 关键节点倒计时

自动检测当前进行中班次的里程碑日期，展示 14 天内即将到来的关键节点：

| 节点类型 | 颜色 | 说明 |
|----------|------|------|
| 纳版截止 | 橙 | 班次 boardingDate 倒计时 |
| SIT 开始 | 蓝 | 班次 sitDate 倒计时 |
| UAT 开始 | 紫 | 班次 uatDate 倒计时 |
| 封板 | 红 | 班次 lockdownDate 倒计时 |
| 投产 | 绿 | 班次 releaseDate 倒计时 |

按剩余天数升序排列，过期不显示。显示所属火车名称和剩余天数。

#### 4. 班次变更率监控

选择一个进行中班次后，展示该班次的变更统计：
- 已纳版需求总数
- 发生过变更的需求数（含普通变更和紧急变更）
- 变更率百分比
- 紧急变更率

> 数据来源：`requirementService.getStats({ scheduleId })` → `changeStats`

#### 5. AI 智能纳版入口

火车管理员可在仪表盘直接触发 AI 智能纳版（弹窗），选择目标火车 → AI 排期建议 → 批量确认纳版。

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR-T1 | 同一页面路由 `/dashboard`，不按角色分拆路由 | 路由匹配 |
| BR-T2 | 统计数据根据当前用户角色和系统筛选动态计算 | 页面加载/筛选变更 |
| BR-T3 | 关键节点仅展示当前日期之后 14 天内的 | 页面加载 |
| BR-T4 | 班次变更率仅在选择具体班次后展示 | 班次选择变更 |
| BR-T5 | AI 智能纳版入口仅 TRAIN_ADMIN/SUPER_ADMIN 可见 | 页面渲染 |

### 验收标准

- [ ] 所有角色登录后默认进入 `/dashboard`
- [ ] 统计卡片数据与角色匹配，点击跳转正确
- [ ] 待办列表数据准确，跳转详情正确
- [ ] 关键节点倒计时计算正确，14天范围过滤生效
- [ ] 系统筛选器联动全部仪表盘数据
- [ ] 班次变更率正常加载和显示
- [ ] 火车管理员可见 AI 智能纳版按钮

### 页面路由

| 路由 | 页面 | 入口 |
|------|------|------|
| `/dashboard` | DashboardPage | 侧边栏「仪表盘」、登录默认跳转 |

## US-C: 月历视图

### 用户故事

> **作为** PM 或火车管理员
> **我希望** 在一个双月日历视图中看到所有版本火车的班次时间线和关键里程碑
> **以便** 直观掌握多个火车的并行节奏，提前识别排期冲突

### 模块定位

月历视图提供「时间轴」视角，以双月日历为载体，将抽象的班次日期转化为可视化色条和里程碑标记，支持按火车和系统维度筛选。

### 重点功能

#### 1. 双月并行日历

左右并排展示两个月历（当前月 + 次月），支持月份切换：
- 左月历 ← → 右月历各自独立切换
- 「回到当月」按钮一键复位
- 日期格内渲染班次色条和事件标记

#### 2. 班次时间线色条

每个班次在日历日期格内渲染为连续彩色横条：
- **颜色**：按班次在火车内的序号轮询分配（8 色调色板）
- **起止**：班次的 `startDate` ~ `endDate` 跨度
- **重叠处理**：多个班次日期重叠时自动分行（`rowIndex`），不遮挡
- **悬浮提示**：鼠标悬停显示火车名、班次名、起止日期

```
┌──────────────────────────────────────────────────────┐
│  六 月                          七 月                │
│  Mo Tu We Th Fr Sa Su   Mo Tu We Th Fr Sa Su        │
│  ┌───────蓝色色条────────┐                            │
│  1  2  3  4  5  6  7    30 1  2  3  4  5  6         │
│  └──────────────────────┘ ┌────绿色色条──────┐       │
│  8  9  10 11 12 13 14  7  8  9  10 11 12 13        │
│                          └──────────────────┘       │
└──────────────────────────────────────────────────────┘
```

#### 3. 里程碑事件标记

在班次色条上标记关键日期节点（带颜色和图标）：

| 事件 | 显示文本 | 颜色 | 图标 |
|------|----------|------|------|
| 纳版截止 | 纳版 | `#1677ff` 蓝 | ⬆ |
| 封板 | 封板 | `#fa8c16` 橙 | ⚠ |
| 投产 | 投产 | `#52c41a` 绿 | ✓ |

#### 4. 火车与系统筛选

- **火车下拉**：切换查看不同火车的班次（默认加载全部火车）
- **系统筛选**：按搭载系统过滤（仅显示涉及该系统的班次）
- 筛选后日历自动刷新

#### 5. 班次进度侧边信息

日历下方展示当前所选火车/系统的班次进度列表：
- 班次名称、所属火车
- 进度条（已过天数 / 总天数）
- 当前状态标签（计划中 / 进行中 / 已完成）

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR-C1 | 日历默认展示当前月 + 次月双月视图 | 页面加载 |
| BR-C2 | 班次色条按 `startDate ~ endDate` 跨度渲染，跨月自动跨越月历边界 | 数据绑定 |
| BR-C3 | 多班次重叠时 `rowIndex` 递增分行，最多 4 行 | 布局计算 |
| BR-C4 | 里程碑仅展示「纳版截止」「封板」「投产」三种，按各自属性日期标记 | 数据绑定 |
| BR-C5 | 火车筛选切换时日历和进度列表同步更新 | 筛选变更 |

### 验收标准

- [ ] 双月日历正确渲染（当前月 + 次月）
- [ ] 月份切换正常（上/下月按钮、回到当月）
- [ ] 班次色条颜色区分明显，起止日期正确
- [ ] 重叠班次自动分行显示
- [ ] 里程碑标记位置和颜色正确
- [ ] 火车筛选和系统筛选联动正确
- [ ] 悬浮提示信息完整（火车名 + 班次名 + 日期范围）

### 页面路由

| 路由 | 页面 | 组件 | 入口 |
|------|------|------|------|
| `/calendar` | CalendarPage | CalendarView | 侧边栏「月历视图」 |

## US-R: 需求池

### 用户故事

> **作为** 版本火车系统的需求管理用户
> **我希望** 在一个集中的需求池中完成需求录入、编辑、评审、跟踪和纳版全生命周期管理
> **以便** 需求的流转状态清晰可追溯，不同角色各司其职

### 模块定位

需求池是「需求全生命周期管理中心」，承载从草稿到投产的完整需求状态机，覆盖 BA、PM、项目经理、技术经理、测试经理、火车管理员的多角色协作。

### 重点功能

#### 1. 需求列表与多维筛选

分页表格展示需求列表，支持：

| 筛选维度 | 控件 | 说明 |
|----------|------|------|
| 系统筛选 | Select 下拉 | 按搭载系统过滤（BA 默认筛选所属系统） |
| 状态多选 | Select 多选 | DRAFT / PENDING_REVIEW / READY / ONBOARDED / RELEASED / CANCELLED |
| 关键词搜索 | Input 输入框 | 匹配需求标题 |
| 排序 | 表头点击 | 创建时间 / 优先级 / 工作量 / 状态 |

表格列：标题、系统、优先级、状态、子状态、工作量、BA、创建时间、操作

#### 2. 需求生命周期状态机

```
草稿 ──发起评审──→ 待评审 ──通过评审──→ 已就绪 ──纳版──→ 已纳版 ──投产──→ 已投产
  ↑                  │                    │                    │
  │                  │ 驳回               │ 需求变更            │ 回滚
  │                  ↓                    ↓                    ↓
  └──重新提交─── 已驳回             草稿（退回）        已就绪（退回）
```

需求状态含义：

| 状态 | 英文 | 含义 | 可执行操作 |
|------|------|------|------------|
| 草稿 | DRAFT | 需求刚创建，信息待完善 | 编辑、发起评审 |
| 待评审 | PENDING_REVIEW | 已提交，等待 PM/技术经理评审 | 通过评审、驳回 |
| 已就绪 | READY | 评审通过，可被纳版 | 需求变更 |
| 已纳版 | ONBOARDED | 已进入版本火车 | 子状态变更、需求变更（非封板）、投产 |
| 已投产 | RELEASED | 已完成上线发布 | 回滚 |
| 已取消 | CANCELLED | 终止，不再进入规划 | 无 |

#### 3. 操作权限矩阵

| 操作 | BA (归属人) | BA (非归属人) | PM | 项目经理 | 技术经理 | 测试经理 | 火车管理员 | 超管 |
|------|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|
| 录入需求 | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |
| 编辑草稿 | ✓ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| 发起评审 | ✓ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| 通过评审 | ✗ | ✗ | ✓ | ✓ | ✓ | ✗ | ✗ | ✓ |
| 驳回 | ✗ | ✗ | ✓ | ✓ | ✓ | ✗ | ✗ | ✓ |
| 重新提交 | ✓ (归属人) | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| 需求变更 | ✓ (归属人) | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ | ✓ |
| 取消需求 | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ | ✓ |

#### 4. AI 需求审查

BA 在草稿/编辑页面点击「AI 审查」按钮，触发自动化质量检查：

- **评分构成**：本地规则 20% + Coze AI 80%
- **审查维度**：标题质量、描述完整性、验收条件明确性、工作量合理性
- **返回结果**：总评分、问题列表、优化建议、AI 优化标题/描述、AI 建议验收条件
- **展示位置**：审查结果在需求详情页以独立卡片展示

#### 5. 子状态流转（已纳版后）

已纳版需求通过子状态跟踪开发测试进度：

```
DEV_IN_PROGRESS → DEV_COMPLETE → SIT_IN_PROGRESS → SIT_PASS → UAT_IN_PROGRESS → UAT_PASS → FROZEN
                                                                                                    ↓
                                                                                              RELEASED (投产)
```

| 子状态 | 操作角色 | 下一步 |
|--------|----------|--------|
| DEV_IN_PROGRESS | 技术经理 → DEV_COMPLETE | 标记开发完成 |
| DEV_COMPLETE | 测试经理 → SIT_IN_PROGRESS | 开始 SIT |
| SIT_IN_PROGRESS | 测试经理 → SIT_PASS | SIT 通过 |
| SIT_PASS | 测试经理 → UAT_IN_PROGRESS | 开始 UAT |
| UAT_IN_PROGRESS | 测试经理 → UAT_PASS | UAT 通过 |
| UAT_PASS | 火车管理员 → FROZEN | 封板 |
| FROZEN | 火车管理员 → RELEASED | 统一投产 |

#### 6. 需求变更管理

- **已就绪状态变更**：需求退回草稿，需重新评审
- **已纳版（非封板）变更**：退出版本火车、释放容量、退回草稿
- **已纳版（封板）紧急变更**：需项目经理或火车管理员审批，通过后退出火车、释放容量

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR-R1 | 仅草稿状态可编辑需求字段 | 编辑提交 |
| BR-R2 | 仅归属人或超管可发起评审 | 按钮点击 |
| BR-R3 | PM/技术经理/项目经理/超管可评审（通过/驳回） | 按钮点击 |
| BR-R4 | 驳回后仅归属人可重新编辑提交 | 按钮点击 |
| BR-R5 | 已纳版非封板需求变更退回草稿 + 退出火车 + 释放容量 | 变更提交 |
| BR-R6 | 封板后变更需紧急变更审批流程 | 变更提交 |
| BR-R7 | 子状态仅已纳版需求可操作，按角色权限流转 | 按钮点击 |
| BR-R8 | AI 审查通过后无法代替人工评审 | 业务校验 |

### 验收标准

- [ ] 需求列表分页加载，筛选/搜索/排序功能正常
- [ ] 需求录入表单字段校验完整（标题/系统/优先级/工作量必填）
- [ ] 状态机流转严格按规则执行（草稿→待评审→已就绪→已纳版→已投产）
- [ ] 权限矩阵控制正确：不同角色只能操作授权范围内的按钮
- [ ] 需求变更正确退回状态并释放火车容量
- [ ] 子状态流转按角色顺序，不可越级
- [ ] AI 审查返回评分和建议并正确展示
- [ ] 需求详情页展示完整信息（基本信息 + 状态日志 + 变更记录 + 审查结果）

### 页面路由

| 路由 | 页面 | 入口 |
|------|------|------|
| `/requirements` | RequirementsPage（列表） | 侧边栏「需求池」 |
| `/requirements/new` | RequirementCreatePage | 需求池「新增需求」按钮 |
| `/requirements/:id` | RequirementDetailPage | 列表行点击 |
| `/requirements/:id/edit` | RequirementEditPage | 详情页「编辑」按钮 |

## US-V: 版本火车

### 用户故事

> **作为** 火车管理员
> **我希望** 管理版本火车的创建、搭载系统配置、班次编排和纳版搭载
> **以便** 有序组织多系统的发布节奏，控制容量和交付范围

### 模块定位

版本火车是「发布计划容器」，包含火车本身（搭载系统、团队容量）+ 班次（时间窗口、关键节点）+ 纳版（需求进车、容量占用）。路由上有两个入口：`/trains`（火车管理）和 `/schedules`（班次列表）。

### 重点功能

#### 1. 版本火车 CRUD

| 操作 | 路由 | 说明 |
|------|------|------|
| 火车列表 | `/trains` | 展示名称、搭载系统数、容量、班次数 |
| 创建火车 | `/trains/new` | 设置名称、描述、搭载系统及团队容量 |
| 火车详情 | `/trains/:id` | 搭载系统列表、容量使用率、班次入口 |
| 编辑火车 | `/trains/:id/edit` | 修改名称/描述/搭载系统配置（乐观锁） |
| 取消火车 | 列表操作 | 仅「计划中」且无纳版需求的火车可取消 |

**搭载系统配置**（创建/编辑时）：
- 选择参与系统（排除已在其他进行中火车的系统）
- 设置团队容量点数（1-500 整数）
- 配置团队成员

**容量使用率**（火车详情）：
- 每个系统的「已用点数 / 总容量点数」百分比进度条
- 超出容量时进度条变红警告

#### 2. 班次管理

每个版本火车可创建多个班次，班次是时间维度的发布窗口。

| 字段 | 说明 |
|------|------|
| 班次名称 | 如「2026年6月班次」 |
| 开始日期 | 班次时间窗口起始 |
| 结束日期 | 班次时间窗口结束 |
| 纳版截止 | 需求进入本班次的最后日期 |
| 封板日期 | 需求冻结、不再接受变更的日期 |
| 投产日期 | 统一投产上线日期 |
| SIT 日期 | SIT 测试开始日期 |
| UAT 日期 | UAT 测试开始日期 |
| 自定义关键日期 | 额外里程碑（名称 + 日期） |

班次关键节点可通过两种方式设置：
- **自动计算**：输入开始/结束日期，按配置规则自动推算纳版截止、封板、投产日期
- **手动设置**：逐项手动填写每个关键日期

**班次状态**：PLANNING（计划中） → IN_PROGRESS（进行中） → COMPLETED（已完成）

**班次列表**（`/schedules`）：
- 按火车筛选或全局查看
- 表格展示：班次名称、所属火车、起止日期、状态、搭载系统数、需求数
- 操作：查看详情、编辑班次

#### 3. 火车详情页集成

火车详情页（`/trains/:id`）聚合以下信息：
- 火车基本信息（名称、描述、创建时间）
- 搭载系统卡片（系统名、容量、已用/总量进度条、成员标签）
- 班次列表入口
- 已纳版需求列表（该火车下所有班次的需求汇总）

#### 4. 纳版搭载

火车管理员将已就绪需求纳入指定火车的指定班次：

- **前置校验**：需求状态为 READY（已就绪）、需求所属系统已搭载到该火车
- **容量校验**：系统剩余容量 >= 需求工作量点数，否则告警但允许强制纳版（软约束）
- **依赖检查**：检查前置依赖需求状态，若依赖未就绪/未纳版则标记「依赖风险」并提示
- **批量纳版**：AI 智能纳版支持批量选择需求一键纳版

#### 5. 火车周期重叠校验

创建/编辑火车时校验起止日期是否与现有火车重叠（已取消的火车不参与比较）：

- 两个火车的起止日期闭区间存在交集 → 判定重叠
- 重叠时警告提示，但不强制阻止（允许管理员判断业务合理性）

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR-V1 | 仅 TRAIN_ADMIN / SUPER_ADMIN 可创建/编辑/取消火车 | 操作前 |
| BR-V2 | 搭载系统不可与其它进行中火车的系统冲突 | 添加系统 |
| BR-V3 | 容量点数范围 1-500 正整数 | 提交 |
| BR-V4 | 取消火车前校验：状态为 PLANNING 且无纳版需求 | 取消操作 |
| BR-V5 | 编辑火车使用乐观锁（version 字段）防并发冲突 | 编辑提交 |
| BR-V6 | 纳版需求前：需求状态为 READY、系统已搭载、容量充足（软约束） | 纳版操作 |
| BR-V7 | 封板后需求的紧急变更需审批 | 变更操作 |
| BR-V8 | 火车周期重叠为软警告，不强制阻止 | 创建/编辑火车 |

## 模块间关系

```
                    ┌─────────────┐
                    │  需求池      │
                    │  录入/评审   │
                    │  已就绪 ──────┼──── 纳版 ────┐
                    └─────────────┘               │
                                                  ▼
┌──────────────┐                        ┌──────────────┐
│  T仪表盘     │                        │  版本火车     │
│  角色聚合    │◄── 统计/待办 ─────────│  火车/班次    │
│  关键节点    │                        │  纳版搭载     │
└──────────────┘                        └──────┬───────┘
       ▲                                       │
       │         进度/里程碑                    │
       └────────── 月历视图 ◄──────────────────┘
                   双月时间线
```

- **需求池 → 版本火车**：已就绪需求通过纳版进入火车的指定班次
- **版本火车 → T仪表盘**：火车管理员仪表盘聚合纳版/投产待办、变更率
- **版本火车 → 月历视图**：班次起止日期和里程碑渲染到双月日历
- **T仪表盘 → 需求池**：统计卡片和待办列表点击均跳转需求池对应筛选视图
