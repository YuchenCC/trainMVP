## 二、需求池管理

### 2.1 功能概述

需求池是管理所有待处理需求的中心仓库，支持业务BA录入、分类、评审和优先级管理。

### 2.2 用户场景

| 角色 | 场景 | 目标 |
|------|------|------|
| 业务BA | 录入新需求 | 作为业务归属人录入业务需求 |
| 产品经理 | 录入新需求 | 辅助BA录入和编辑需求 |
| 业务BA/产品经理 | 管理需求状态 | 跟踪需求处理进度 |
| 项目经理/产品经理 | 评审需求 | 对需求进行评审，决定是否通过 |
| 火车管理员 | 查看待规划需求 | 为版本火车选择需求 |

### 2.3 功能详细设计

#### 2.3.1 需求录入

| 需求项 | 详细说明 |
|--------|----------|
| 需求标题 | 必填，简洁描述需求核心内容 |
| 需求描述 | 必填，详细说明需求背景、目标和价值 |
| 归属系统 | 必填，选择参与当前版本火车的系统 |
| 优先级 | 必填，选择：P0/P1/P2/P3 |
| 工作量点数 | 必填，正整数（1-100），表示该需求的开发工作量大小 |
| 业务归属人 | 必填，选择该系统的业务BA |
| 产品经理 | 必填，选择该系统的产品经理 |
| 创建人 | 必填，实际创建该需求的用户，默认当前用户 |
| 提出时间 | 系统自动填充 |
| 依赖需求 | 可选，关联其他系统的需求作为前置依赖（支持跨系统） |

**依赖需求规则**：
- 一个需求可配置多个依赖需求，支持跨系统依赖
- 依赖需求需选择已存在的需求（草稿及以上状态均可关联）
- 依赖关系为"前置依赖"：依赖需求未完成前，本需求不建议纳版
- 依赖需求状态为"已投产"或"已纳版"时，视为依赖已满足
- 依赖需求状态为"已取消"时，系统提示预警

#### 2.3.2 需求分类

| 分类维度 | 可选值 | 说明 |
|----------|--------|------|
| 需求类型 | 新功能/优化/缺陷 | 按需求性质分类 |
| 来源渠道 | 业务提出/用户反馈/数据分析/竞品分析 | 按需求来源分类 |

#### 2.3.3 需求状态流转

**需求状态流转图：**

```

                                 │
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
          ▼                      │                      │
     ┌─────────┐                 │                      │
     │  草稿   │                 │                      │
     └────┬────┘                 │                      │
          │ 发起评审              │                      │
          ▼                      │                      │
     ┌─────────┐    拒绝    ┌─────────┐               │
     │ 待评审  │───────────►│ 已拒绝  │               │

     │ 已纳版  │─────────────────┘                    │

          │                                           │
          │ 子状态流转：                               │
          │ ┌──────────┐  开发完成   ┌──────────┐     │
          │ │ 开发中   │───────────►│ SIT测试  │     │

          │                              ▼            │
          │                        ┌──────────┐       │
          │                        │ UAT测试  │       │
          │                        └────┬─────┘       │
          │                             │ UAT通过     │
          │                             ▼             │
          │                        ┌──────────┐       │
          │                        │  封板    │───────┤ 紧急变更（需审批）
          │                        └────┬─────┘       │
          │                             │ 封板后投产   │
          ▼                             ▼             │
     ┌─────────┐                                       │
     │ 已投产  │                                       │
     └────┬────┘                                       │
          │                                            │
          ▼                                            │
     ┌─────────┐◄──────────────────────────────────────┐
     │ 已取消  │         投产前任何状态都可取消（终态）    │
     └─────────┘                                       │
                                                       │
                                                       │ 回滚（已投产→已就绪）
```

> **图例说明**：
> - 主线：草稿 → 待评审 → 已就绪 → 已纳版 → 已投产 → 已取消
> - 已纳版子状态：开发中 → SIT测试 → UAT测试 → 封板
> - **从火车移除**：已纳版 → 已就绪（管理员操作）
> - **需求变更**：已就绪、已纳版（开发中/SIT测试/UAT测试）→ 草稿
> - **紧急变更**：封板 → 草稿（需审批）
> - **回滚**：已投产 → 已就绪（管理员操作）

**主线流程**：草稿 → 待评审 → 已就绪 → 已纳版（开发中→SIT测试→UAT测试→封板）→ 已投产

**分支流程**：
- 待评审 → 已拒绝（可重新编辑后再次提交）
- 已就绪 → 已纳版，已纳版 → 已就绪（从火车移除）
- 已纳版子状态流转：开发中 → SIT测试 → UAT测试 → 封板
- **已就绪/已纳版（非封板）→ 草稿（需求变更）：需求变更内容后退回草稿，需重新发起评审**
- **封板状态 → 草稿（紧急变更）：需额外审批通过后方可变更**
- 任何状态 → 已取消（已投产状态除外，已投产只能回滚）

| 状态 | 说明 | 可转换状态 |
|------|------|------------|
| **草稿** | 初始录入状态，可编辑修改 | 待评审、已取消 |
| **待评审** | BA已发起评审，等待产品经理或项目经理确认 | 已就绪、已拒绝、已取消 |
| **已就绪** | 产品经理或项目经理评审通过，可进入版本火车规划 | 已纳版、已取消、草稿（需求变更） |
| **已拒绝** | 产品经理或项目经理评审未通过，需说明原因 | 草稿（重新编辑）、已取消 |
| **已纳版** | 已搭载到版本火车，含4个子状态 | 已就绪（从火车移除）、已投产、已取消、草稿（需求变更/紧急变更） |
| 开发中 | 需求已纳版，进入开发阶段 | SIT测试、已就绪（从火车移除）、草稿（需求变更） |
| SIT测试 | 开发完成，进入技术测试阶段 | UAT测试、已就绪（从火车移除）、草稿（需求变更） |
| UAT测试 | SIT通过，进入业务验收测试阶段 | 封板、已就绪（从火车移除）、草稿（需求变更） |
| 封板 | UAT通过，代码冻结，仅支持紧急变更 | 已投产、草稿（紧急变更，需审批） |
| **已投产** | 需求已上线发布 | 已就绪（回滚时） |
| **已取消** | 需求已取消，终态 | - |

| 状态变更 | 触发功能 | 操作入口 | 操作权限 |
|----------|----------|----------|----------|
| 草稿 → 待评审 | 发起评审 | 需求详情页/列表页操作按钮 | 业务归属人（BA） |
| 待评审 → 已就绪 | 评审通过 | 评审页面 | 产品经理/项目经理 |
| 待评审 → 已拒绝 | 评审拒绝 | 评审页面（需填写原因） | 产品经理/项目经理 |
| 已拒绝 → 草稿 | 重新编辑 | 需求详情页操作按钮 | 业务归属人（BA）/产品经理 |
| 已就绪 → 已纳版（开发中） | 纳版搭载 | 版本火车页面 | 火车管理员 |
| 已纳版 → 已就绪 | 从火车移除 | 版本火车页面 | 火车管理员 |
| 开发中 → SIT测试 | 开发完成 | 需求详情页/列表页操作按钮 | 技术经理 |
| SIT测试 → UAT测试 | SIT通过 | 需求详情页/列表页操作按钮 | 测试经理 |
| UAT测试 → 封板 | UAT通过 | 需求详情页/列表页操作按钮 | 业务归属人（BA） |
| 封板 → 已投产 | 确认投产 | 版本火车页面 | 火车管理员 |
| 已就绪/已纳版（非封板）→ 草稿 | 需求变更 | 需求详情页/列表页操作按钮 | 业务归属人（BA）/火车管理员 |
| 封板 → 草稿 | 紧急变更 | 需求详情页/列表页操作按钮（需审批） | 业务归属人（BA）/火车管理员 |
| 任意状态 → 已取消 | 取消需求 | 需求详情页/列表页操作按钮（需填写原因） | 业务归属人（BA）/火车管理员 |

##### 2.3.4.1 发起评审

**功能描述**：业务归属人（BA）将草稿状态的需求发起评审

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"草稿"，当前用户为该需求的业务归属人（BA） |
| 输入 | 需求ID |
| 输出 | 需求状态变更为"待评审" |
| 业务规则 | 1. 校验需求必填项是否完整<br>2. 校验当前用户是否为该需求的业务归属人（BA）<br>3. 记录发起评审时间和操作人<br>4. 发起评审后需求进入待评审状态，等待产品经理或项目经理确认 |
| 界面交互 | 点击"发起评审"按钮 → 校验通过 → 状态变更 → 提示"已发起评审" |

##### 2.3.4.2 评审通过

**功能描述**：评审人通过需求评审

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"待评审"，当前用户为评审人（产品经理、项目经理或技术经理） |
| 输入 | 需求ID、评审意见（可选） |
| 输出 | 需求状态变更为"已就绪" |
| 业务规则 | 1. 校验当前用户是否有评审权限（产品经理、项目经理或技术经理）<br>2. 记录评审人、评审时间<br>3. 保存评审意见 |
| 界面交互 | 点击"通过"按钮 → 状态变更 → 提示"评审通过" |

##### 2.3.4.3 评审拒绝

**功能描述**：评审人拒绝需求评审

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"待评审"，当前用户为评审人（产品经理、项目经理或技术经理） |
| 输入 | 需求ID、拒绝原因（必填） |
| 输出 | 需求状态变更为"已拒绝" |
| 业务规则 | 1. 校验当前用户是否有评审权限（产品经理、项目经理或技术经理）<br>2. 拒绝原因必填，最多500字<br>3. 记录评审人、评审时间、拒绝原因 |
| 界面交互 | 点击"拒绝"按钮 → 弹窗填写拒绝原因 → 确认 → 状态变更 → 提示"已拒绝" |

##### 2.3.4.4 重新编辑

**功能描述**：将已拒绝的需求重新编辑

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已拒绝"，当前用户为业务归属人（BA）或产品经理 |
| 输入 | 需求ID |
| 输出 | 需求状态变更为"草稿" |
| 业务规则 | 1. 校验当前用户是否为业务归属人（BA）或产品经理<br>2. 状态变更为草稿后可编辑修改 |
| 界面交互 | 点击"重新编辑"按钮 → 状态变更 → 进入编辑页面 |

##### 2.3.4.5 取消需求

**功能描述**：取消需求

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态不为"已取消"且不为"已投产"，当前用户为业务归属人（BA）或火车管理员 |
| 输入 | 需求ID、取消原因（必填） |
| 输出 | 需求状态变更为"已取消" |
| 业务规则 | 1. 校验当前用户是否为业务归属人（BA）或火车管理员（产品经理不能取消需求）<br>2. 取消原因必填，最多500字<br>3. 记录取消时间、操作人、取消原因<br>4. 已纳版的需求取消时，自动从版本火车移除<br>5. 已投产的需求不能取消，只能通过回滚操作撤回 |
| 界面交互 | 点击"取消"按钮 → 弹窗填写取消原因 → 确认 → 状态变更 → 提示"已取消" |

##### 2.3.4.6 需求变更（普通变更）

**功能描述**：已就绪或已纳版（非封板）状态的需求如需变更内容，退回草稿状态，需重新发起评审

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已就绪"或"已纳版（开发中/SIT测试/UAT测试）"，当前用户为业务归属人（BA）或火车管理员 |
| 输入 | 需求ID、变更原因（必填） |
| 输出 | 需求状态变更为"草稿"；如已纳版则同时从版本火车移除 |
| 业务规则 | 1. 校验当前用户是否为业务归属人（BA）或火车管理员<br>2. 变更原因必填，最多500字<br>3. 记录变更时间、操作人、变更原因<br>4. 已纳版的需求变更时，自动从版本火车移除，释放团队容量<br>5. 变更后需求状态统一为"草稿"，可编辑修改<br>6. 修改完成后需重新发起评审，走完整评审流程<br>7. 如需再次纳版，需重新走纳版流程 |
| 界面交互 | 点击"需求变更"按钮 → 弹窗填写变更原因 → 确认 → 状态变更 → 提示"已退回草稿，可重新编辑后发起评审" |

##### 2.3.4.7 紧急变更（封板状态）

**功能描述**：封板状态的需求如需变更内容，需提交紧急变更申请，审批通过后方可变更

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已纳版（封板）"，当前用户为业务归属人（BA）或火车管理员 |
| 输入 | 需求ID、紧急变更原因（必填）、紧急程度（P0紧急/P1较紧急） |
| 输出 | 提交审批 → 审批通过后状态变更为"草稿"，从版本火车移除 |
| 业务规则 | 1. 校验当前用户是否为业务归属人（BA）或火车管理员<br>2. 紧急变更原因必填，最多500字，需说明为何封板后仍需变更<br>3. 紧急程度必选：P0紧急（影响上线）或P1较紧急<br>4. 提交后进入审批流程，审批人为项目经理或火车管理员<br>5. 审批通过后：需求状态变更为"草稿"，从版本火车移除，释放团队容量<br>6. 审批拒绝后：维持封板状态，记录拒绝原因<br>7. 变更后需重新发起评审，走完整评审和纳版流程 |
| 界面交互 | 点击"紧急变更"按钮 → 弹窗填写变更原因和紧急程度 → 提交审批 → 审批人收到通知 → 审批通过/拒绝 → 状态变更/维持 |

##### 2.3.4.8 开发完成（进入SIT测试）

**功能描述**：开发中状态的需求完成开发，进入SIT测试阶段

| 项目 | 说明 |
|------|------|

| 输入 | 需求ID |
| 输出 | 需求子状态变更为"SIT测试" |
| 业务规则 | 1. 校验当前用户是否为技术经理<br>2. 记录开发完成时间<br>3. 子状态变更为"SIT测试" |
| 界面交互 | 点击"开发完成"按钮 → 确认 → 状态变更 → 提示"已进入SIT测试" |

##### 2.3.4.9 SIT通过（进入UAT测试）

| 输出 | 需求子状态变更为"UAT测试" |
| 业务规则 | 1. 校验当前用户是否为测试经理<br>2. 记录SIT通过时间<br>3. 子状态变更为"UAT测试" |
| 界面交互 | 点击"SIT通过"按钮 → 确认 → 状态变更 → 提示"已进入UAT测试" |

##### 2.3.4.10 UAT通过（封板）

**功能描述**：UAT测试通过，进入封板状态

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已纳版（UAT测试）"，当前用户为业务归属人（BA） |
| 输入 | 需求ID、UAT验收结果（可选） |
| 输出 | 需求子状态变更为"封板" |
| 业务规则 | 1. 校验当前用户是否为业务归属人（BA）<br>2. 记录UAT通过时间<br>3. 子状态变更为"封板"，代码冻结<br>4. 封板后仅支持紧急变更（需审批） |

> **说明**: 纳版搭载、从火车移除、确认投产功能详见第三章功能详细设计（3.4.5-3.4.8）。

## 三、版本火车计划（日历）

### 3.1 功能概述

版本火车计划模块以日历视图为核心，支持版本火车的创建、管理和关键节点管控。

### 3.2 用户场景

| 角色 | 场景 | 目标 |
|------|------|------|
| 火车管理员 | 创建版本火车并定义成员 | 确定哪些系统参与本次版本火车 |
| 火车管理员 | 创建火车时刻表 | 设置需求冻结、规划会等关键节点时间 |
| 火车管理员 | 管理需求搭载 | 将已就绪的需求搭载到版本火车 |
| 业务BA | 查看版本火车日历 | 了解发版计划和进度 |

### 3.3 角色职责

| 角色 | 职责范围 | 说明 |
|------|----------|------|
| **火车管理员** | 管理版本火车全生命周期 | 创建版本火车、配置搭载系统及各系统的人员（BA、产品经理、技术经理、测试经理等）、管理关键节点、需求搭载/移除/投产 |

> **说明**: 火车管理员负责版本火车的全生命周期管理，包括"谁来参加"和"什么时候做什么"。

### 3.4 功能详细设计

#### 3.4.1 版本火车创建

**操作权限**: 火车管理员

| 字段 | 说明 |
|------|------|
| 版本火车名称 | 必填，如"2026年Q2第1车" |
| 开始时间 | 必填，版本火车起始日期（周一） |
| 版本周期 | 系统自动填充，由超级管理员全局配置（可选：2周/3周/4周） |
| 结束时间 | 系统自动计算 = 开始时间 + 周期天数 - 1 |
| 描述 | 可选，版本火车背景和目标说明 |
| 状态 | 系统自动：计划中/进行中/已完成/已取消 |

**版本周期规则**：
- 版本火车周期由 **超级管理员** 全局配置，可选：**2周（14天）**、**3周（21天）**、**4周（28天）**
- 开始时间默认为最近一个周一，支持手动选择
- 结束时间根据配置的周期自动计算
- 关键节点时间根据开始时间和周期自动推算（见3.4.3）
- 同一系统同一时间只能参与一个版本火车

#### 3.4.2 搭载系统与团队容量管理

**操作权限**: 火车管理员

**核心概念**: 每个版本火车可搭载多个系统，每个系统需配置团队容量，容量决定该系统在本版本火车中能承接的需求工作量上限。

##### 系统成员配置

| 字段 | 说明 |
|------|------|
| 系统名称 | 必填，如"用户中心"、"订单系统"、"支付系统" |
| 业务BA | 必填，该系统的业务负责人 |
| 产品经理 | 必填，该系统的产品负责人 |
| 项目经理 | 可选，该系统的项目经理 |
| 技术经理 | 必填，该系统的技术负责人 |
| 开发团队 | 必填，该系统的开发人员列表 |
| 测试负责人 | 必填，该系统的测试负责人 |

##### 团队容量配置

| 字段 | 说明 |
|------|------|
| 系统名称 | 自动填充 |
| 本期可用点数 | 必填，该系统在本版本火车中的总可用工作量点数 |
| 已分配点数 | 系统自动计算，已纳版需求的工作量点数总和 |
| 剩余可用点数 | 系统自动计算 = 可用点数 - 已分配点数 |
| 容量使用率 | 系统自动计算 = 已分配点数 / 可用点数 × 100% |

**容量规则**：
- 火车管理员在创建版本火车时，为每个搭载系统配置本期可用点数
- 已纳版需求的工作量点数累计计入已分配点数
- 当已分配点数 ≥ 可用点数时，系统提示容量已满
- 容量已满时仍可手动纳版，但需二次确认
- 容量使用率以进度条形式展示，≥90% 显示红色预警

##### 需求工作量点

| 字段 | 说明 |
|------|------|
| 工作量点数 | 必填，正整数，表示该需求的开发工作量大小 |
| 填写时机 | 需求状态变更为"已就绪"前必须填写 |
| 填写权限 | 业务归属人（BA）或产品经理 |

**工作量点规则**：
- 工作量点数为必填项，范围 1-100
- 工作量点数在需求评审通过前必须填写，否则无法发起评审
- 工作量点数一旦填写，已就绪需求改点数可由项目经理或技术经理修改，不需要重新评审；已纳版需求改点数必须走需求变更。”
- 修改工作量点数时，记录修改人和修改时间

#### 3.4.3 关键节点管控

**操作权限**: 火车管理员

版本火车包含以下3个关键节点，时间根据版本开始日期和周期自动推算：

| 节点名称 | 节点说明 | 默认时间（相对于开始时间） |
|----------|----------|---------------------------|
| **统一纳版** | 确定进入本次版本火车的需求清单，已就绪的需求在此节点前完成纳版 | 周期过半前最后一个周五 |
| **统一封板** | UAT测试完成，之后严格控制需求变更 | 投产前一周周五 |
| **统一投产** | 版本火车正式发布上线 | 周期最后一天（周五） |

**不同周期节点时间对照**：

| 周期 | 统一纳版 | 统一封板 | 统一投产 |
|------|----------|----------|----------|
| 2周（14天） | 第1周周五（第5天） | 第2周周五（第12天） | 第2周周五（第14天） |
| 3周（21天） | 第2周周五（第12天） | 第3周周五（第19天） | 第3周周五（第21天） |
| 4周（28天） | 第2周周五（第12天） | 第4周周五（第26天） | 第4周周五（第28天） |

**节点时间线示意**（以4周周期、5月5日周一开始为例）：

```
第1周          第2周          第3周          第4周
5/5  5/9       5/12 5/16      5/19 5/23      5/26 5/30
│              │              │              │
├──────────────┼──────────────┼──────────────┤
│  需求准备    │  统一纳版    │  开发+测试   │  统一投产  │
│              │  5/16(周五)  │  统一封板    │  5/30(周五) │
│              │              │  5/23(周五)  │            │
└──────────────┴──────────────┴──────────────┴────────────┘
```

**节点管控规则**：
- 节点时间由火车管理员在创建版本火车时确认，支持调整
- 统一纳版节点前：可自由搭载/移除需求
- 统一纳版到统一封板之间：纳版需火车管理员审批
- 统一封板后：不再接受新需求搭载，仅支持紧急变更（需额外审批）
- 到达统一投产节点：系统自动将所有"已纳版"状态的需求批量变更为"已投产"

#### 3.4.4 智能纳版（AI Agent排期建议）

**功能描述**: 根据团队容量、需求工作量点数和优先级，由AI Agent生成纳版排期建议，辅助火车管理员决策。

| 项目 | 说明 |
|------|------|
| 触发条件 | 火车管理员点击"AI排期建议"按钮 |
| 输入 | 版本火车ID、各系统团队容量、所有已就绪需求（含工作量点数和优先级） |
| 输出 | AI生成的纳版建议清单 |
| 操作权限 | 火车管理员 |

**AI排期建议逻辑**：

```
输入:
  - 各系统可用容量（点数）
  - 已就绪需求列表（含工作量点数、优先级、归属系统、依赖需求）

处理逻辑:
  1. 依赖检查（优先）：
     - 遍历所有已就绪需求，检查其依赖需求是否已满足
     - 依赖需求状态为"已纳版"或"已投产" → 依赖已满足 ✅
     - 依赖需求状态为"已取消" → 标记为"⚠️ 依赖已取消，建议确认" ⚠️
     - 依赖需求未纳入任何版本火车 → 标记为"⏳ 依赖未满足，建议延期或同时纳入" ⏳
  2. 按优先级排序：P0 > P1 > P2 > P3
  3. 同优先级按工作量点数从小到大排序（优先承接小需求）
  4. 按系统维度逐个分配：
     - 从优先级最高的需求开始
     - 跳过依赖未满足的需求（标记为建议延期）
     - 检查该需求归属系统的剩余容量
     - 若容量充足 → 纳入建议清单
     - 若容量不足 → 标记为"建议延期至下一车"
  5. 依赖联动建议：
     - 若需求A依赖需求B，且B未纳入本车 → 建议同时将B纳入本车
     - 若需求B归属系统不在本车搭载列表中 → 提示"依赖需求B归属系统未参与本车"
  6. 输出建议结果

输出:
  - ✅ 建议纳入本车需求清单（含各系统容量使用情况）
  - ⏳ 建议延期需求清单（含延期原因：容量不足/依赖未满足）
  - 🔗 依赖联动建议（建议同时纳入的依赖需求）
  - ⚠️ 依赖风险提示（依赖已取消/依赖系统未参与本车）
  - 📊 各系统容量使用率预估
```

**AI建议结果展示**：

| 归属系统 | 建议纳入需求数 | 总点数 | 可用容量 | 使用率 | 延期需求数 |
|----------|---------------|--------|----------|--------|-----------|
| 用户中心 | 5 | 32 | 40 | 80% | 1 |
| 订单系统 | 3 | 25 | 30 | 83% | 2 |
| 支付系统 | 4 | 28 | 28 | 100% | 0 |

> **重要**: AI排期建议仅供参考，实际纳版由火车管理员人工确认。

#### 3.4.5 纳版搭载

**功能描述**：火车管理员确认将已就绪的需求搭载到版本火车

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已就绪"，当前用户为火车管理员 |
| 输入 | 需求ID、版本火车ID |
| 输出 | 需求状态变更为"已纳版"，关联版本火车，扣减系统容量 |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 需求归属系统必须在版本火车的搭载系统列表中<br>3. 需求工作量点数必须已填写<br>4. **依赖校验**：检查依赖需求是否已满足（已纳版/已投产），未满足时按风险等级分级提示（warning：依赖未纳入本车；high：依赖状态为草稿/待评审/已拒绝；critical：依赖已取消），允许强制纳版但需填写确认说明<br>5. 纳版后扣减对应系统的可用容量<br>6. 若系统容量不足，弹窗预警但允许强制纳版<br>7. 记录纳版时间和操作人 |
| 界面交互 | 版本火车页面 → 选择需求 → 点击"确认纳版" → 校验容量 → 状态变更 → 提示"纳版成功" |

**纳版模式**：
- **AI辅助纳版**: 点击"AI排期建议" → 查看建议 → 勾选确认 → 批量纳版
- **手动纳版**: 直接勾选需求 → 点击"确认纳版" → 逐个/批量纳版
