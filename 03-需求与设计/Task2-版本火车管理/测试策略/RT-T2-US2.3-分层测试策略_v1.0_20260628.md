## 二、L1 单元测试分析

### 1.1 可单测识别结果

本模块代码中，班次相关业务逻辑集中在 `train.service.ts`，但所有函数都直接依赖 Prisma 数据库操作（`prisma.trainSchedule.findMany`、`prisma.$transaction` 等），没有独立可测试的纯函数。

**无 L1 单测覆盖点原因**：
- `listAllSchedules` → 依赖 Prisma `findMany` + 分页计算
- `listTrainSchedules` → 依赖 Prisma `findMany`
- `updateTrainScheduleStatus` → 依赖 Prisma `update` + 状态机校验（但校验逻辑内联在函数中）
- 日期校验逻辑 → 内联在函数中，无独立工具函数

**结论**：L1 单测不适用于本模块，建议通过 L2 API 自动化覆盖。

## 四、L3 Playwright 测试策略

### 4.1 核心用户旅程

**Journey 1: 班次列表查询与火车切换**
```
进入 /schedules → 默认加载全局班次 → 选择火车 → 班次列表刷新 → 清空火车 → 恢复全局列表
```

**Journey 2: 班次状态变更**
```
进入 /schedules → 选择火车 → 点击计划中班次"开始" → 班次变为进行中 → 点击"封板" → 班次变为封板
```

**Journey 3: 新增班次**
```
进入 /schedules → 选择火车 → 点击新增班次 → 填写班次名称/日期 → 提交 → 班次出现在列表
```

### 4.2 页面元素验证

| 测试点 | 验证内容 |
|-------|---------|
| 火车下拉框 | 数据加载、下拉展开、选项选择 |
| 班次表格 | 列字段渲染（名称/状态/日期/系统数/已纳版） |
| 分页器 | 总数显示、页码切换 |
| 操作按钮 | 状态对应按钮显示（开始/封板/投产/取消/编辑） |
| 详情跳转 | 点击行跳转 /trains/:trainId/schedules/:scheduleId |

### 4.3 优先级规划

**P0（优先实现）**:
