## US2.1 版本火车创建

**作为** 火车管理员  
**我希望** 创建新的版本火车，设置火车名称、开始/结束时间、描述  
**以便** 启动新的版本火车周期

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR2.1.1 | 版本火车名称必填，长度 2-100 字符 | 提交时 |
| BR2.1.2 | 开始时间必填，火车管理员自由选择（建议选周一） | 提交时 |
| BR2.1.3 | 结束时间必填，火车管理员自由选择（建议选周五），且必须晚于开始时间 | 提交时 |
| BR2.1.4 | 描述可选，最多 2000 字符 | 提交时 |
| BR2.1.5 | 系统自动生成关键节点时间（统一纳版、统一封板、统一投产） | 创建时 |
| BR2.1.6 | 关键节点时间根据实际周期长度自动计算，确保为周五 | 创建时 |
| BR2.1.7 | 版本火车初始状态为「计划中」 | 创建时 |
| BR2.1.8 | 乐观锁：version 字段用于并发控制 | 更新时 |

### 关键节点时间计算规则

| 节点 | 计算方式 |
|------|----------|
| **统一纳版日** | 周期过半前的最后一个周五 |
| **统一封板日** | 投产前一周的周五（即结束时间前 7 天的周五） |
| **统一投产日** | 结束时间（火车管理员指定） |

**计算逻辑**：
1. 计算实际周期天数 = 结束时间 - 开始时间 + 1
2. 根据周期长度计算各节点位置
3. 确保节点日期为周五（若不是，往前调整到最近的周五）

### 验收条件

- [ ] AC2.1.1: 火车管理员点击「创建版本火车」，打开创建表单
- [ ] AC2.1.2: 表单包含名称、开始时间、结束时间、描述字段
- [ ] AC2.1.3: 必填字段为空时提交，显示对应校验错误提示
- [ ] AC2.1.4: 结束时间早于开始时间时，提示「结束时间必须晚于开始时间」
- [ ] AC2.1.5: 提交成功后，版本火车状态为「计划中」，自动生成关键节点时间
- [ ] AC2.1.6: 提交成功后跳转到版本火车详情页
- [ ] AC2.1.7: 非授权角色看不到「创建版本火车」按钮

## US2.2 搭载系统与团队容量管理

**作为** 火车管理员  
**我希望** 为版本火车配置搭载系统及每个系统的团队容量  
**以便** 控制各系统在本版本火车中的需求工作量上限

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR2.2.1 | 搭载系统从全局系统列表中选择，一个系统同一时间只能参与一个版本火车 | 提交时 |
| BR2.2.2 | 每个系统必须配置：业务BA、产品经理、技术经理、测试负责人、开发团队 | 提交时 |
| BR2.2.3 | 每个系统必须配置「本期可用点数」，正整数，范围 1-500 | 提交时 |
| BR2.2.4 | 系统自动计算：已分配点数 = 该系统已纳版需求的工作量总和 | 实时 |
| BR2.2.5 | 系统自动计算：剩余可用点数 = 本期可用点数 - 已分配点数 | 实时 |
| BR2.2.6 | 系统自动计算：容量使用率 = 已分配点数 / 本期可用点数 × 100% | 实时 |
| BR2.2.7 | 容量使用率 ≥ 90% 时显示红色预警 | 显示时 |
| BR2.2.8 | 支持动态添加/移除搭载系统（但系统已有纳版需求时不能移除） | 操作时 |
| BR2.2.9 | 支持调整本期可用点数（调整后自动重新计算剩余点数和使用率） | 更新时 |

### 验收条件

- [ ] AC2.2.1: 版本火车详情页「搭载系统」标签页显示已配置的系统列表
- [ ] AC2.2.2: 点击「添加系统」，弹出系统选择表单
- [ ] AC2.2.3: 系统选择表单中，排除已参与其他进行中/计划中版本火车的系统
- [ ] AC2.2.4: 添加系统时，必须填写业务BA、产品经理、技术经理、测试负责人、开发团队、本期可用点数
- [ ] AC2.2.5: 本期可用点数输入非 1-500 的整数时，提示「本期可用点数必须为 1-500 的正整数」
- [ ] AC2.2.6: 添加成功后，该系统显示在列表中，容量使用率为 0%
- [ ] AC2.2.7: 当系统已有纳版需求时，「移除」按钮禁用或提示无法移除
- [ ] AC2.2.8: 调整本期可用点数后，剩余可用点数和容量使用率自动更新
- [ ] AC2.2.9: 容量使用率 ≥ 90% 时，进度条显示为红色并加 ⚠️ 标识

## US2.5 纳版搭载（含依赖校验、容量校验）

**作为** 火车管理员  
**我希望** 将已就绪的需求纳版到版本火车  
**以便** 确定本次版本火车的需求范围

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR2.5.1 | 前置条件：需求状态为「已就绪」 | 操作时 |
| BR2.5.2 | 需求归属系统必须在版本火车的搭载系统列表中 | 校验时 |
| BR2.5.3 | 需求工作量点数必须已填写 | 校验时 |
| BR2.5.4 | **依赖校验**：检查依赖需求状态，按风险等级分级提示 | 校验时 |
| BR2.5.5 | 依赖风险等级：已纳版/已投产→无风险；已就绪未纳版→warning；草稿/待评审/已拒绝→high；已取消→critical | 计算时 |
| BR2.5.6 | 存在依赖风险时允许强制纳版，但需填写确认说明 | 提交时 |
| BR2.5.7 | **容量校验**：纳版后扣减对应系统的可用容量 | 提交时 |
| BR2.5.8 | 若系统容量不足（剩余可用点数 < 需求点数），弹窗预警但允许强制纳版 | 提交时 |
| BR2.5.9 | 纳版成功后，需求状态变更为「已纳版」，关联版本火车 | 变更时 |
| BR2.5.10 | 纳版成功后，记录操作历史：操作时间、操作人、操作类型 | 变更时 |
| BR2.5.11 | 使用 Prisma 事务保证数据一致性 | 执行时 |

### 依赖风险提示

| 风险等级 | 依赖状态 | 提示内容 | 是否允许纳版 |
|----------|----------|----------|-------------|
| 无风险 | 已纳版 / 已投产 | 依赖已满足 | 是，无需确认 |
| warning | 已就绪（未纳版） | 依赖需求已就绪但未纳入本车，纳版后可能影响开发进度 | 是，需确认 |
| high | 草稿 / 待评审 / 已拒绝 | 依赖需求状态为[XX]，纳版后可能阻塞开发 | 是，需填写说明 |
| critical | 已取消 | 依赖需求已取消，请重新评估需求依赖 | 是，需填写说明 |

### 验收条件

- [ ] AC2.5.1: 版本火车详情页「纳版管理」标签页显示待纳版需求列表（已就绪状态）
- [ ] AC2.5.2: 待纳版需求列表支持多选，点击「确认纳版」打开纳版确认弹窗
- [ ] AC2.5.3: 纳版确认弹窗显示：已选需求列表、容量影响、依赖风险
- [ ] AC2.5.4: 若需求归属系统不在搭载系统列表中，提示「需求归属系统[XX]未配置到本火车」
- [ ] AC2.5.5: 若存在依赖风险，显示对应风险等级标识和提示文字
- [ ] AC2.5.6: 若容量不足，显示 ⚠️ 容量预警但仍可点击「确认纳版」
- [ ] AC2.5.7: 确认纳版后，需求状态变更为「已纳版」，火车ID关联成功
- [ ] AC2.5.8: 确认纳版后，对应系统的已分配点数增加，剩余点数减少
- [ ] AC2.5.9: 操作历史记录「纳版搭载」条目
