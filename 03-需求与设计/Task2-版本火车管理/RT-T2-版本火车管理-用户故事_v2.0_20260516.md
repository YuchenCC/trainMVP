## 架构变更说明

### 变更前（v1.x）：
- 1 火车 → 1 班次（一一对应）
- 需求直接关联到火车

### 变更后（v2.0）：
- 1 火车 → 多班次（一对多）
- 班次为独立实体
- 需求关联到班次，而非火车
- 班次创建时保存容量配置的快照
- 已完成班次变为只读

## US2.2 火车班次创建

**作为** 火车管理员  
**我希望** 为已创建的版本火车创建新班次，设置班次名称（可选）、开始/结束时间，系统自动计算关键节点并保存容量快照  
**以便** 确定本次火车的具体发版周期

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR2.2.1 | 班次名称可选，不填则自动生成（格式：[火车名]第[序号]班 | 提交时 |
| BR2.2.2 | 开始时间必填，火车管理员自由选择（建议选周一） | 提交时 |
| BR2.2.3 | 结束时间必填，火车管理员自由选择（建议选周五），且必须晚于开始时间 | 提交时 |
| BR2.2.4 | 系统自动生成关键节点时间（统一纳版、统一封板、统一投产） | 创建时 |
| BR2.2.5 | 关键节点时间根据实际周期长度自动计算，确保为周五 | 创建时 |
| BR2.2.6 | 班次创建时，保存当前火车配置的容量快照（历史记录） | 创建时 |
| BR2.2.7 | 同一火车下的班次时间可以重叠（支持多班次并行） | 创建时 |
| BR2.2.8 | 班次创建后状态为「计划中」 | 创建时 |
| BR2.2.9 | 班次关键节点时间支持调整 | 操作时 |
| BR2.2.10 | 乐观锁：version 字段用于并发控制 | 更新时 |

### 关键节点时间计算规则

| 节点 | 计算方式 |
|------|----------|
| **统一纳版日** | 周期过半前的最后一个周五 |
| **统一封板日** | 投产前一周的周五（即结束时间前 7 天的周五） |
| **统一投产日** | 结束时间（火车管理员指定） |

**计算逻辑**：
1. 计算实际周期天数 = 结束时间 - 开始时间 + 1
2. 根据周期长度计算各节点位置
3. 确保节点日期为周五（若不是，往前调整到最近的周五）

### 验收条件

- [ ] AC2.2.1: 班次列表页显示「创建班次」按钮（仅火车管理员可见）
- [ ] AC2.2.2: 点击「创建班次」，打开班次创建表单
- [ ] AC2.2.3: 表单包含班次名称（可选）、开始时间、结束时间字段
- [ ] AC2.2.4: 必填字段为空时提交，显示对应校验错误提示
- [ ] AC2.2.5: 结束时间早于开始时间时，提示「结束时间必须晚于开始时间」
- [ ] AC2.2.6: 提交成功后，班次状态为「计划中」
- [ ] AC2.2.7: 提交成功后，系统自动计算并显示关键节点时间
- [ ] AC2.2.8: 提交成功后，系统自动保存当前容量配置作为快照
- [ ] AC2.2.9: 关键节点时间显示在班次详情页中

## US2.5 纳版搭载到班次（含依赖校验、容量校验）

**作为** 火车管理员  
**我希望** 将已就绪的需求纳版到具体班次  
**以便** 确定本次班次的需求范围

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR2.5.1 | 前置条件：需求状态为「已就绪」，班次状态不为「已完成」 | 操作时 |
| BR2.5.2 | 纳版流程：先根据需求归属系统查询相关火车，再查询火车下的有效班次，用户选择具体班次 | 操作时 |
| BR2.5.3 | 需求归属系统必须在班次所属火车的搭载系统列表中 | 校验时 |
| BR2.5.4 | 需求工作量点数必须已填写 | 校验时 |
| BR2.5.5 | **依赖校验**：检查依赖需求状态，按风险等级分级提示 | 校验时 |
| BR2.5.6 | 依赖风险等级：已纳版/已投产→无风险；已就绪未纳版→warning；草稿/待评审/已拒绝→high；已取消→critical | 计算时 |
| BR2.5.7 | 存在依赖风险时允许强制纳版，但需填写确认说明 | 提交时 |
| BR2.5.8 | **容量校验**：纳版后扣减对应班次的系统可用容量 | 提交时 |
| BR2.5.9 | 若系统容量不足（剩余可用点数 < 需求点数），弹窗预警但允许强制纳版 | 提交时 |
| BR2.5.10 | 纳版成功后，需求状态变更为「已纳版」，关联班次ID | 变更时 |
| BR2.5.11 | 纳版成功后，班次状态自动变更为「进行中」（如果之前是计划中） | 变更时 |
| BR2.5.12 | 纳版成功后，记录操作历史：操作时间、操作人、操作类型 | 变更时 |
| BR2.5.13 | 使用 Prisma 事务保证数据一致性 | 执行时 |
| BR2.5.14 | 乐观锁：version 字段用于并发控制 | 更新时 |

### 依赖风险提示

| 风险等级 | 依赖状态 | 提示内容 | 是否允许纳版 |
|----------|----------|----------|-------------|
| 无风险 | 已纳版 / 已投产 | 依赖已满足 | 是，无需确认 |
| warning | 已就绪（未纳版） | 依赖需求已就绪但未纳入本班次，纳版后可能影响开发进度 | 是，需确认 |
| high | 草稿 / 待评审 / 已拒绝 | 依赖需求状态为[XX]，纳版后可能阻塞开发 | 是，需填写说明 |
| critical | 已取消 | 依赖需求已取消，请重新评估需求依赖 | 是，需填写说明 |

## US2.8 回滚

**作为** 火车管理员  
**我希望** 将已投产的需求回滚  
**以便** 处理发版后的问题

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR2.8.1 | 前置条件：需求状态为「已投产」，班次状态不为「已完成」 | 操作时 |
| BR2.8.2 | 回滚原因必填，最多 500 字符 | 提交时 |
| BR2.8.3 | 回滚成功后，需求状态变更为「已就绪」 | 变更时 |
| BR2.8.4 | 回滚成功后，记录操作历史：操作时间、操作人、操作类型、回滚原因 | 变更时 |
| BR2.8.5 | 乐观锁：version 字段用于并发控制 | 更新时 |

### 验收条件

- [ ] AC2.8.1: 已投产需求列表中，每个需求显示「回滚」按钮（班次未完成时）
- [ ] AC2.8.2: 点击「回滚」，弹出回滚原因填写弹窗
- [ ] AC2.8.3: 回滚原因为空时提交，提示「请输入回滚原因」
- [ ] AC2.8.4: 确认回滚后，需求状态变更为「已就绪」
- [ ] AC2.8.5: 操作历史记录「回滚」条目，包含回滚原因

## US2.9 完成班次

**作为** 火车管理员  
**我希望** 手动确认关闭班次  
**以便** 结束班次的生命周期

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR2.9.1 | 前置条件：当前用户为火车管理员，当前日期不早于统一投产节点，该班次下不存在仍处于「已纳版」状态的需求 | 操作时 |
| BR2.9.2 | 所有需求必须为「已投产」或「已取消」或「已就绪」（已移除） | 校验时 |
| BR2.9.3 | 不满足条件时，提示具体原因（例如：「还有 X 个需求未投产」） | 提示时 |
| BR2.9.4 | 完成成功后，班次状态变更为「已完成」 | 变更时 |
| BR2.9.5 | 班次完成后变为只读，不可编辑 | 变更后 |
| BR2.9.6 | 完成成功后，记录操作历史：操作时间、操作人、操作类型 | 变更时 |
| BR2.9.7 | 乐观锁：version 字段用于并发控制 | 更新时 |

### 验收条件

- [ ] AC2.9.1: 进行中状态的班次详情页显示「完成班次」按钮
- [ ] AC2.9.2: 点击「完成班次」，弹出确认弹窗
- [ ] AC2.9.3: 若当前日期早于统一投产节点，提示「当前日期未到统一投产节点[XX]，无法完成」
- [ ] AC2.9.4: 若存在未投产的需求，提示「还有 X 个需求未投产，请先处理」
- [ ] AC2.9.5: 确认完成后，班次状态变更为「已完成」
- [ ] AC2.9.6: 班次完成后，编辑按钮隐藏或禁用，需求不再支持编辑
- [ ] AC2.9.7: 操作历史记录「完成班次」条目

## US2.10 BA 默认系统筛选（Task 1 增强）

**作为** 业务BA  
**我希望** 进入需求列表页时，默认显示我归属系统的需求  
**以便** 快速查看我负责的需求

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR2.10.1 | 触发条件：当前用户角色为 BA，进入需求列表页 | 加载时 |
| BR2.10.2 | 查询该 BA 在版本火车搭载系统中作为业务归属人的系统列表 | 查询时 |
| BR2.10.3 | 取第一个系统 ID 作为 systemId 默认值 | 加载时 |
| BR2.10.4 | 前端系统下拉框默认选中该系统 | 渲染时 |
| BR2.10.5 | BA 可手动切换系统下拉框查看其他系统需求 | 操作时 |
| BR2.10.6 | 若无归属系统时，显示全部系统需求（回退到 MVP 行为） | 加载时 |

### 验收条件

- [ ] AC2.10.1: BA 用户进入需求列表页，系统下拉框默认选中其第一个归属系统
- [ ] AC2.10.2: 需求列表只显示默认选中系统的需求
- [ ] AC2.10.3: BA 可手动切换系统下拉框，查看其他系统需求
- [ ] AC2.10.4: 若无归属系统，系统下拉框默认选中「全部」，显示所有系统需求
