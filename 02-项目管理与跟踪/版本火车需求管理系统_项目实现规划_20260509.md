| 项目 | 内容 |

## 二、技术架构

### 2.1 技术选型（全栈JS方案）

| 层级 | 技术选型 | 版本 | 说明 |
|------|----------|------|------|
| **前端框架** | React 18 + TypeScript | 18.x | AI生成代码准确率最高 |
| **UI组件库** | Ant Design 5 | 5.x | 企业级中后台首选 |
| **状态管理** | Zustand | 4.x | 轻量级，AI易生成 |
| **构建工具** | Vite | 5.x | 快速开发体验 |
| **后端框架** | Fastify + TypeScript | 4.x | 高性能Node.js框架 |
| **ORM** | Prisma | 5.x | 类型安全，AI友好 |
| **数据库** | PostgreSQL | 16 | 开发阶段使用 |
| **缓存** | Redis | 7 | 会话/热点数据缓存 |
| **AI大模型** | DeepSeek V3 | - | 国产大模型，性价比高 |
| **部署** | Docker Compose | - | 本地开发+演示部署 |

## 四、并行任务规划

> **核心思路**：将项目拆为5个任务，每个任务按 **设计 → 前后端并行开发 → 各自单测 → 模块集成测试** 的流水线执行。Task 1和Task 2可并行，完成后进入Task 3，最后统一集成测试。

### 4.1 角色任务定义

每个功能模块内部按以下阶段执行，**后端和前端并行开发**：

```
┌──────────┐
│  设计     │  接口设计、类型定义、页面原型
│ (Design) │
└────┬─────┘
     │
     ├──────────────────────────────┐
     ▼                              ▼
┌──────────┐                   ┌──────────┐
│ 后端开发  │                   │ 前端开发  │  基于接口文档+Mock数据并行
│ (Backend) │                   │(Frontend)│
└────┬─────┘                   └────┬─────┘
     │                              │
     ▼                              ▼
┌──────────┐                   ┌──────────┐
│ 后端单测  │                   │ 前端单测  │  开发完成立即测试
│ (UT-Back)│                   │(UT-Front)│
└────┬─────┘                   └────┬─────┘
     │                              │
     └──────────┬───────────────────┘
                ▼
         ┌──────────┐
         │ 模块集成  │  前后端联调、全流程验证
         │ (Integ)  │
         └──────────┘
```

| 角色 | 说明 | 产出 | 并行策略 |
|------|------|------|----------|
| **设计** | 接口设计、数据模型、页面原型 | API文档、类型定义、页面线框 | 先行，产出供前后端同时使用 |
| **后端开发** | API实现、业务逻辑、数据库操作 | 后端代码 | 与前端并行 |
| **前端开发** | 页面组件、交互逻辑、状态管理 | 前端代码 | 与后端并行，用Mock数据 |
| **后端单测** | API单元测试、业务逻辑测试 | 测试用例 | 后端开发完成后立即执行 |
| **前端单测** | 组件测试、交互逻辑测试 | 测试用例 | 前端开发完成后立即执行 |
| **模块集成测试** | 前后端联调、模块内全流程验证 | 集成测试报告 | 前后端单测都通过后执行 |

### 5.1 核心表结构

```prisma
// 用户表
model User {
  id          String   @id @default(cuid())
  username    String   @unique
  displayName String
  email       String   @unique
  role        Role     @default(BA)
  ssoId       String?  // SSO对接ID
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  
  // 关联
  ownedRequirements  Requirement[]  @relation("BAOwner")
  pmRequirements     Requirement[]  @relation("PM")
  createdRequirements Requirement[] @relation("Creator")
}

enum Role {
  BA           // 业务BA
  PM           // 产品经理
  PROJECT_MGR  // 项目经理
  TECH_MGR     // 技术经理
  TRAIN_CREATOR // 火车创建员
  TRAIN_ADMIN   // 火车管理员
  SUPER_ADMIN   // 超级管理员
}

// 系统表
model System {
  id          String   @id @default(cuid())
  name        String   @unique
  description String?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  
  // 关联
  requirements   Requirement[]
  trainSystems   TrainSystem[]
  members        SystemMember[]
}

// 系统成员表
model SystemMember {
  id       String     @id @default(cuid())
  systemId String
  userId   String
  role     SystemRole
  system   System     @relation(fields: [systemId], references: [id])
  user     User       @relation(fields: [userId], references: [id])
  
  @@unique([systemId, userId, role])
}

enum SystemRole {
  BA        // 业务BA
  PM        // 产品经理
  PROJECT_MGR // 项目经理
  TECH_MGR  // 技术经理
  DEV       // 开发
  QA        // 测试
}

// 需求表
model Requirement {
  id            String           @id @default(cuid())
  title         String
  description   String           @db.Text
  status        ReqStatus        @default(DRAFT)
  subStatus     ReqSubStatus?    // 已纳版时的子状态
  priority      Int              @default(3) // 1-5
  storyPoints   Int              @default(0)
  dependencies  Json?            // 依赖需求ID列表
  
  // 关联
  systemId      String
  system        System           @relation(fields: [systemId], references: [id])
  baOwnerId     String           // 业务归属人
  baOwner       User             @relation("BAOwner", fields: [baOwnerId], references: [id])
  pmId          String?          // 产品经理
  pm            User?            @relation("PM", fields: [pmId], references: [id])
  creatorId     String           // 创建人
  creator       User             @relation("Creator", fields: [creatorId], references: [id])
  trainId       String?          // 所属火车
  train         Train?           @relation(fields: [trainId], references: [id])
  
  createdAt     DateTime         @default(now())
  updatedAt     DateTime         @updatedAt
  
  // 状态变更记录
  statusLogs    StatusLog[]
}

enum ReqStatus {
  DRAFT       // 草稿
  PENDING     // 待评审
  READY       // 已就绪
  REJECTED    // 已拒绝
  ONBOARDED   // 已纳版
  RELEASED    // 已投产
  CANCELLED   // 已取消
}

enum ReqSubStatus {
  DEV_IN_PROGRESS  // 开发中
  SIT_TESTING      // SIT测试
  UAT_TESTING      // UAT测试
  FROZEN           // 封板
}

// 状态变更记录
model StatusLog {
  id           String   @id @default(cuid())
  requirementId String
  requirement  Requirement @relation(fields: [requirementId], references: [id])
  fromStatus   ReqStatus?
  toStatus     ReqStatus
  reason       String?  @db.Text
  operatorId   String
  createdAt    DateTime @default(now())
}

// 版本火车表
model Train {
  id          String      @id @default(cuid())
  name        String
  startDate   DateTime
  endDate     DateTime
  cycleWeeks  Int         @default(4) // 2/3/4周
  status      TrainStatus @default(PLANNING)
  description String?     @db.Text
  
  // 关键节点（自动计算）
  boardingDate DateTime?  // 统一纳版日
  lockdownDate DateTime?  // 统一封板日
  releaseDate  DateTime?  // 统一投产日
  
  createdById String
  createdBy   User        @relation(fields: [createdById], references: [id])
  createdAt   DateTime    @default(now())
  updatedAt   DateTime    @updatedAt
  
  // 关联
  requirements   Requirement[]
  trainSystems   TrainSystem[]
}

enum TrainStatus {
  PLANNING    // 计划中
  IN_PROGRESS // 进行中
  COMPLETED   // 已完成
  CANCELLED   // 已取消
}

// 火车-系统关联表（容量管理）
model TrainSystem {
  id             String   @id @default(cuid())
  trainId        String
  systemId       String
  capacityPoints Int      @default(0) // 可用容量
  usedPoints     Int      @default(0) // 已用容量
  
  train          Train    @relation(fields: [trainId], references: [id])
  system         System   @relation(fields: [systemId], references: [id])
  
  @@unique([trainId, systemId])
}

// 紧急变更审批记录
model EmergencyChange {
  id             String   @id @default(cuid())
  requirementId  String
  requirement    Requirement @relation(fields: [requirementId], references: [id])
  urgency        UrgencyLevel
  reason         String   @db.Text
  status         ApprovalStatus @default(PENDING)
  approverId     String?
  approvedAt     DateTime?
  createdAt      DateTime @default(now())
}

enum UrgencyLevel {
  P0 // 紧急（影响上线）
  P1 // 较紧急
}

enum ApprovalStatus {
  PENDING  // 待审批
  APPROVED // 已通过
  REJECTED // 已拒绝
}
```

### 9.3 人工审核机制

> **设计和代码提交两个关键节点必须由用户人工审核通过后才能继续**

#### 审核关卡

```
设计完成                    代码完成
    │                          │
    ▼                          ▼
┌──────────┐              ┌──────────┐
│ AI提交设计 │              │ AI提交代码 │
│ 供审核    │              │ 供审核    │
└────┬─────┘              └────┬─────┘
     │                          │
     ▼                          ▼
┌──────────┐              ┌──────────┐
│ ⏸️ 暂停    │              │ ⏸️ 暂停    │
│ 等待审核  │              │ 等待审核  │
└────┬─────┘              └────┬─────┘
     │                          │
     ▼                          ▼
┌──────────┐              ┌──────────┐
│ 用户审核  │              │ 用户审核  │
│ ✅通过    │              │ ✅通过    │
│ ❌打回修改│              │ ❌打回修改│
└────┬─────┘              └────┬─────┘
     │                          │
     ▼                          ▼
  通过→继续开发              通过→git commit
  打回→修改后重新提交        打回→修改后重新提交
```

#### 审核流程说明

| 审核节点 | 触发时机 | 审核内容 | 通过后动作 |
|----------|----------|----------|------------|
| **设计审核** | 每个Task设计阶段完成后 | 接口设计、数据模型、页面原型、类型定义 | 进入前后端并行开发 |
| **代码审核** | 每个提交节点完成后 | 代码质量、功能完整性、测试通过率 | git commit + push |

#### AI执行规则

1. **设计阶段**：AI完成设计后主动提交给用户审核，**不自动进入开发**
2. **代码阶段**：AI完成一个提交节点的代码后主动提交给用户审核，**不自动commit**
3. **打回处理**：用户打回时，AI根据反馈修改后重新提交审核
4. **审核超时**：无超时机制，AI始终等待用户确认

### 9.4 AI协作规范

1. **分步骤生成**：复杂功能拆分成小任务，逐个生成
2. **先设计后编码**：设计完成后必须等用户审核通过，再开始编码
3. **增量验证**：每完成一个功能点立即测试验证
4. **保留上下文**：关键决策写入文档，避免AI遗忘
5. **主动提交审核**：设计和代码两个节点主动暂停，等待用户确认
6. **同步更新PRD**：所有设计变更、代码变更都必须同步更新PRD文档，并在PRD的版本记录中追加变更历史（版本号+日期+变更内容）
7. **过程文档归档**：每个Task的设计方案、测试案例必须输出为独立文档（MD格式），作为过程文档交付
