### 2.2 核心交付内容

#### T1设计方案 Grill 审核（17项决策）

对 Task 1 需求池管理设计方案进行系统性审核，逐项确认并做出以下关键决策：

| 编号 | 决策点 | 最终方案 |
|------|--------|----------|
| 1 | 删除方式 | 软删除（deletedAt字段），保留审计数据 |
| 2 | 并发控制 | 乐观锁（version字段），前端提示冲突后重新加载 |
| 3 | 需求编号 | 自动生成 REQ-{年份}-{4位序号}，全局唯一不可修改 |
| 4 | 批量操作 | 支持批量发起评审、批量取消、批量修改优先级 |
| 5 | 批量冲突处理 | 部分成功模式：成功的执行，失败的返回失败列表 |
| 6 | 并发批量冲突 | 批量操作中单个冲突不影响其他，冲突项返回具体原因 |
| 7 | 依赖风险分级 | A级（阻塞）/ B级（影响）/ C级（提示），依赖关系上标注 |
| 8 | 依赖风险传递 | 不递归传递，仅查直接依赖（传递性检查记入待办） |
| 9 | 评审通过指定PM | 评审通过时必须指定产品经理 |
| 10 | 需求变更范围 | 已就绪/已纳版（非封板）可变更回草稿 |
| 11 | 变更后编号 | 需求编号不变，版本号+1 |
| 12 | 取消需求 | 需填写取消原因，已投产不可取消 |
| 13 | 紧急变更审批 | 仅项目经理+火车管理员可审批 |
| 14 | 子状态确认 | 开发完成→技术经理，SIT通过→测试经理，UAT通过→BA |
| 15 | 需求描述格式 | TipTap富文本编辑器，存储时XSS过滤（sanitize-html） |
| 16 | XSS防护策略 | 存储时过滤（写入前清洗），非渲染时过滤 |
| 17 | 状态变更通知 | 记入待办（站内通知 + 红点提示，MVP后续实现） |

## 三、工作过程回顾

### 3.1 阶段划分

```
T1设计方案Grill审核（约1.5小时）
  ├─ 逐项审核设计方案，确认17个决策点
  ├─ 关键决策：软删除 vs 硬删除 → 软删除
  ├─ 关键决策：乐观锁 vs 悲观锁 → 乐观锁
  ├─ 关键决策：需求编号自动生成
  ├─ 关键决策：批量操作部分成功模式
  ├─ 关键决策：富文本XSS存储时过滤
  └─ 关键决策：通知机制记入待办（MVP后续）

文档更新（约1小时）
  ├─ 设计方案更新至v1.1（17项决策落地）
  ├─ PRD更新至v1.1（新增批量操作、需求编号、软删除、乐观锁）
  └─ 创建待办事项文档

收尾（约0.5小时）
  ├─ 工作小结编写
  └─ 提交推送到远端
```

### 3.2 关键决策演进

| 决策点 | 初始方案 | 最终方案 | 理由 |
|--------|----------|----------|------|
| 删除方式 | 硬删除 | 软删除 | 保留审计数据，符合业务合规需求 |
| 并发控制 | 无 | 乐观锁 | MVP阶段冲突概率低，实现简单 |
| 需求编号 | 无 | 自动生成 | 便于追踪和引用，全局唯一 |
| 批量操作 | 无 | 部分成功模式 | 提升效率，单个失败不影响整体 |
| XSS防护 | 渲染时过滤 | 存储时过滤 | 安全前置，从源头阻断 |
| 通知机制 | 本次实现 | 记入待办 | 降低MVP范围，聚焦核心功能 |

## 四、交流过程复盘与改进建议

### 4.1 做得好的地方

1. **Grill审核机制有效**：系统性逐项审核设计方案，发现并解决了17个潜在问题，避免了编码阶段的返工
2. **决策记录完整**：每个决策点都有明确的选择和理由，便于后续回溯
3. **待办管理规范**：将非MVP范围的功能记入待办，保持当前迭代聚焦

### 4.2 可以改进的地方

#### 1. Grill审核耗时较长
- **现象**：17个决策点逐一确认，整体耗时约1.5小时
- **改进**：审核前先按优先级分组，低风险决策批量确认，高风险决策逐项讨论

#### 2. 设计方案初稿不够完善
- **现象**：grill审核中才发现软删除、乐观锁、需求编号等基础设计缺失
- **改进**：设计方案初稿应覆盖CRUD基础模式（删除策略、并发控制、编号规则）
