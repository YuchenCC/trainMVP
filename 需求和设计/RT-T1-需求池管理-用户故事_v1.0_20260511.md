## US1.1 需求录入

**作为** 业务BA（或产品经理、火车管理员）  
**我希望** 录入一条新需求，填写标题、描述、归属系统、优先级、工作量点数、业务归属人等信息  
**以便** 需求进入需求池进行后续评审和规划

### 业务规则

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR1.1.1 | 需求标题必填，长度 2-100 字符 | 提交时 |
| BR1.1.2 | 需求描述必填，支持富文本（TipTap），后端存储时做 XSS 白名单过滤（允许 p/br/b/i/u/ul/ol/li/h1-h6/a/img），支持图片上传（单张≤2MB，仅 jpg/png/gif，本地文件存储） | 提交时 |
| BR1.1.3 | 归属系统必填，从全局系统字典中选择（无论是否有版本火车） | 提交时 |
| BR1.1.4 | 优先级必填，P0/P1/P2/P3 四选一 | 提交时 |
| BR1.1.5 | 工作量点数必填，正整数，范围 1-100 | 提交时 |
| BR1.1.6 | 业务归属人必填，选择该系统下的 BA 角色 | 提交时 |
| BR1.1.7 | 产品经理可选，不填时评审通过前可后续指定 | 提交时 |
| BR1.1.8 | 创建人自动填充为当前登录用户 | 创建时 |
| BR1.1.9 | 提出时间自动填充为当前时间 | 创建时 |
| BR1.1.10 | 需求编号自动生成，格式 `REQ-{系统码}-{年份}-{序号}`（如 REQ-UC-2026-0001），按系统递增，跨年重置，全局唯一 | 创建时 |
| BR1.1.11 | 需求类型可选：新功能/优化/缺陷 | 提交时 |
| BR1.1.12 | 来源渠道可选：业务提出/用户反馈/数据分析/竞品分析 | 提交时 |
| BR1.1.13 | 新建需求初始状态为"草稿" | 创建时 |
| BR1.1.14 | 需求描述存储为 HTML，服务端入库前做 XSS 过滤（strip_tags 或白名单策略） | 入库时 |
| BR1.1.15 | 乐观锁：需求记录包含 version 字段，更新时校验版本号防止并发覆盖 | 更新时 |

### 验收条件

- [ ] AC1.1.1: BA 用户点击"新增需求"，可打开需求录入表单
- [ ] AC1.1.2: 表单包含所有必填字段（标题、描述、归属系统、优先级、工作量点数、业务归属人）
- [ ] AC1.1.3: 归属系统下拉框只显示已参与版本火车的系统
- [ ] AC1.1.4: 业务归属人下拉框只显示对应系统的 BA 角色
- [ ] AC1.1.5: 必填字段为空时提交，表单显示对应校验错误提示
- [ ] AC1.1.6: 工作量点数输入非 1-100 的整数时，提示"工作量点数必须为 1-100 的正整数"
- [ ] AC1.1.7: 提交成功后，需求状态为"草稿"，自动生成需求编号
- [ ] AC1.1.8: 提交成功后跳转到需求详情页
- [ ] AC1.1.9: 需求描述中输入 `<script>alert(1)</script>`，保存后查看详情页不执行脚本
- [ ] AC1.1.10: 产品经理角色也可录入需求，创建人记录为该产品经理
- [ ] AC1.1.11: 非授权角色（技术经理、测试经理）看不到"新增需求"按钮

| 编号 | 规则 | 校验时机 |
|------|------|----------|
| BR1.2.1 | 仅草稿状态的需求可编辑 | 操作时 |
