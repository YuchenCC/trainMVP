## 二、需求池管理

### 2.1 功能概述

需求池是管理所有待处理需求的中心仓库，支持业务BA录入、分类、评审和优先级管理。

### 2.2 用户场景

| 角色        | 场景      | 目标             |
| --------- | ------- | -------------- |
| 业务BA      | 录入新需求   | 作为业务归属人录入业务需求  |
| 产品经理      | 录入新需求   | 辅助BA录入和编辑需求    |
| 业务BA/产品经理 | 管理需求状态  | 跟踪需求处理进度       |
| 项目经理/产品经理 | 评审需求    | 对需求进行评审，决定是否通过 |
| 火车管理员     | 查看待规划需求 | 为版本火车选择需求      |

### 2.3 功能详细设计

#### 2.3.1 需求录入

| 需求项   | 详细说明                                       |
| ----- | ------------------------------------------ |
| 需求标题  | 必填，简洁描述需求核心内容，1-200字                              |
| 需求描述  | 必填，详细说明需求背景、目标和价值（TipTap富文本编辑器，后端存储时sanitize-html过滤 + 前端渲染时DOMPurify二次过滤，双重XSS防护） |
| 归属系统  | 必填，选择参与当前版本火车的系统（草稿状态可修改，修改时联动清空业务归属人和产品经理）                           |
| 优先级   | 必填，选择：P0/P1/P2/P3                          |
| 工作量点数 | 必填，正整数（1-100），表示该需求的开发工作量大小                |
| 业务归属人 | 必填，选择该系统的业务BA（联动归属系统过滤该系统的BA成员）                              |
| 产品经理  | 可选，评审通过时指定（联动归属系统过滤该系统的PM成员）                                 |
| 创建人   | 必填，实际创建该需求的用户，默认当前用户                       |
| 提出时间  | 系统自动填充                                     |
| 依赖需求  | 可选，关联其他系统的需求作为前置依赖（支持跨系统，编辑时全量替换依赖列表）                  |

**依赖需求规则**：

- 一个需求可配置多个依赖需求，支持跨系统依赖
- 依赖需求需选择已存在的需求（草稿及以上状态均可关联）
- 依赖关系为"前置依赖"：依赖需求未完成前，本需求不建议纳版
- 依赖需求状态为"已投产"或"已纳版"时，视为依赖已满足，显示✅标识
- 依赖需求状态为"已取消"时，系统提示⚠️预警（仅预警，不阻断操作）
- 依赖管理采用全量替换模式：编辑需求时提交完整依赖列表，后端全量替换，不做增量操作
- 添加依赖时自动进行循环依赖检测（DFS算法，按需查询数据库），存在循环则阻断

**v1.2 增强**：

- **系统列表动态加载**：归属系统下拉从 `GET /api/systems` 动态获取，不在前端硬编码
- **人员动态加载**：选择归属系统后，通过 `GET /api/systems/:id/users` 动态获取该系统成员，BA/PM下拉框按角色过滤
- **依赖需求搜索**：通过 `GET /api/requirements/search?q=` 关键词搜索已有需求（模糊匹配编号和标题），300ms防抖，点击添加，不再随机选择
- **实施文件**：
  - 后端：`apps/server/src/modules/systems/`（系统+成员查询）
  - 后端：`apps/server/src/modules/requirements/service.ts`（searchRequirements）
  - 前端：`apps/web/src/services/system.ts`（系统服务API）
  - 前端：`apps/web/src/components/requirements/RequirementForm.tsx`（动态表单）

**草稿编辑权限**：

- 草稿状态下，BA/PM/PROJECT_MGR 都可编辑需求（不限归属人）
- 非草稿状态下，仅业务归属人（BA）、产品经理（PM）可操作自己关联的需求

**需求编号规则**：

- 格式：REQ-{年份}-{4位序号}，如 REQ-2026-0001
- 创建时自动生成，并发冲突时自动重试最多3次

#### 2.3.2 需求分类

| ---- | ------------------- | ------- |
| 需求类型 | 新功能/优化/缺陷           | 按需求性质分类 |

> **图例说明**：
>
> - 主线：草稿 → 待评审 → 已就绪 → 已纳版 → 已投产 → 已取消
> - 已纳版子状态：开发中 → SIT测试 → UAT测试 → 封板
> - **从火车移除**：已纳版 → 已就绪（管理员操作）
> - **需求变更**：已就绪、已纳版（开发中/SIT测试/UAT测试）→ 草稿
> - **紧急变更**：封板 → 草稿（需审批）
> - **回滚**：已投产 → 已就绪（管理员操作）

**主线流程**：草稿 → 待评审 → 已就绪 → 已纳版（开发中→SIT测试→UAT测试→封板）→ 已投产

**分支流程**：

- 待评审 → 已拒绝（可重新编辑后再次提交）
- 已就绪 → 已纳版，已纳版 → 已就绪（从火车移除）
- 已纳版子状态流转：开发中 → SIT测试 → UAT测试 → 封板 （实际应关联至研发协同软需任务拆分，MVP版本用子状态模拟）
- **已就绪/已纳版（非封板）→ 草稿（需求变更）：需求变更内容后退回草稿，需重新发起评审**
- **封板状态 → 草稿（紧急变更）：需额外审批通过后方可变更**
- 任何状态 → 已取消（已投产状态除外，已投产只能回滚）

| 状态      | 说明                      | 可转换状态                            |
| ------- | ----------------------- | -------------------------------- |
| **草稿**  | 初始录入状态，可编辑修改            | 待评审、已取消                          |
| **待评审** | BA已发起评审，等待产品经理或项目经理确认   | 已就绪、已拒绝、已取消                      |
| **已就绪** | 产品经理或项目经理评审通过，可进入版本火车规划 | 已纳版、已取消、草稿（需求变更）                 |
| **已拒绝** | 产品经理或项目经理评审未通过，需说明原因    | 草稿（重新编辑）、已取消                     |
| **已纳版** | 已搭载到版本火车，含4个子状态         | 已就绪（从火车移除）、已投产、已取消、草稿（需求变更/紧急变更） |
| 开发中     | 需求已纳版，进入开发阶段            | SIT测试、已就绪（从火车移除）、草稿（需求变更）        |
| SIT测试   | 开发完成，进入技术测试阶段           | UAT测试、已就绪（从火车移除）、草稿（需求变更）        |
| UAT测试   | SIT通过，进入业务验收测试阶段        | 封板、已就绪（从火车移除）、草稿（需求变更）           |
| 封板      | UAT通过，代码冻结，仅支持紧急变更      | 已投产、草稿（紧急变更，需审批）                 |
| **已投产** | 需求已上线发布                 | 已就绪（回滚时）                         |
| **已取消** | 需求已取消，终态                | -                                |

| 已拒绝 → 草稿         | 重新编辑  | 需求详情页操作按钮            | 业务归属人（BA）/产品经理  |
| 已就绪 → 已纳版（开发中）   | 纳版搭载  | 版本火车页面               | 火车管理员           |

##### 2.3.4.1 发起评审

**功能描述**：业务归属人（BA）将草稿状态的需求发起评审

| 项目   | 说明                                                                                    |
| ---- | ------------------------------------------------------------------------------------- |
| 前置条件 | 需求状态为"草稿"，当前用户为BA/PM/PROJECT_MGR                                                          |
| 输入   | 需求ID                                                                                  |
| 输出   | 需求状态变更为"待评审"                                                                          |
| 业务规则 | 1. 校验需求必填项是否完整2. 校验当前用户是否有操作权限（BA/PM/PROJECT_MGR，草稿状态不限归属人）3. 记录发起评审时间和操作人4. 发起评审后需求进入待评审状态，等待产品经理或项目经理确认 |
| 界面交互 | 点击"发起评审"按钮 → 校验通过 → 状态变更 → 提示"已发起评审"                                                  |

##### 2.3.4.2 评审通过

**功能描述**：评审人通过需求评审

| 项目   | 说明                                                     |
| ---- | ------------------------------------------------------ |
| 前置条件 | 需求状态为"待评审"，当前用户为评审人（产品经理、项目经理或技术经理）                    |
| 输入   | 需求ID、评审意见（可选）                                          |
| 输出   | 需求状态变更为"已就绪"                                           |
| 业务规则 | 1. 校验当前用户是否有评审权限（产品经理、项目经理或技术经理）2. 记录评审人、评审时间3. 保存评审意见 |
| 界面交互 | 点击"通过"按钮 → 状态变更 → 提示"评审通过"                             |

##### 2.3.4.3 评审拒绝

**功能描述**：评审人拒绝需求评审

| 项目   | 说明                                                                 |
| ---- | ------------------------------------------------------------------ |
| 前置条件 | 需求状态为"待评审"，当前用户为评审人（产品经理、项目经理或技术经理）                                |
| 输入   | 需求ID、拒绝原因（必填）                                                      |
| 输出   | 需求状态变更为"已拒绝"                                                       |
| 业务规则 | 1. 校验当前用户是否有评审权限（产品经理、项目经理或技术经理）2. 拒绝原因必填，最多500字3. 记录评审人、评审时间、拒绝原因 |
| 界面交互 | 点击"拒绝"按钮 → 弹窗填写拒绝原因 → 确认 → 状态变更 → 提示"已拒绝"                          |

##### 2.3.4.4 重新编辑

**功能描述**：将已拒绝的需求重新编辑

| 项目   | 说明                                         |
| ---- | ------------------------------------------ |
| 前置条件 | 需求状态为"已拒绝"，当前用户为BA/PM/PROJECT_MGR             |
| 输入   | 需求ID                                       |
| 输出   | 需求状态变更为"草稿"                                |
| 业务规则 | 1. 校验当前用户是否为BA/PM/PROJECT_MGR2. 状态变更为草稿后可编辑修改 |
| 界面交互 | 点击"重新编辑"按钮 → 状态变更 → 进入编辑页面                 |

##### 2.3.4.5 取消需求

**功能描述**：取消需求（草稿状态免填原因，非草稿状态需填原因）

| 项目   | 说明                                                                                                                      |
| ---- | ----------------------------------------------------------------------------------------------------------------------- |
| 前置条件 | 需求状态不为"已取消"且不为"已投产"，当前用户为业务归属人（BA）、项目经理（PROJECT_MGR）或火车管理员                                                                                |
| 输入   | 需求ID、取消原因（草稿状态免填，非草稿状态必填）                                                                                                           |
| 输出   | 需求状态变更为"已取消"                                                                                                            |
| 业务规则 | 1. 校验当前用户是否为业务归属人（BA）、项目经理（PROJECT_MGR）或火车管理员（产品经理不能取消需求）2. 草稿状态取消免填原因，直接变更状态3. 非草稿状态取消原因必填，最多500字4. 记录取消时间、操作人、取消原因5. 已纳版的需求取消时，自动从版本火车移除6. 已投产的需求不能取消，只能通过回滚操作撤回 |
| 界面交互 | 草稿：点击"取消"按钮 → 确认 → 状态变更 → 提示"已取消"；非草稿：点击"取消"按钮 → 弹窗填写取消原因 → 确认 → 状态变更 → 提示"已取消"                                                                               |

##### 2.3.4.6 需求变更（普通变更）

**功能描述**：已就绪或已纳版（非封板）状态的需求如需变更内容，退回草稿状态，需重新发起评审

| 项目   | 说明                                                                                                                                                         |
| ---- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 前置条件 | 需求状态为"已就绪"或"已纳版（开发中/SIT测试/UAT测试）"，当前用户为业务归属人（BA）或火车管理员                                                                                                     |
| 输入   | 需求ID、变更原因（必填）                                                                                                                                              |
| 输出   | 需求状态变更为"草稿"；如已纳版则同时从版本火车移除                                                                                                                                 |
| 业务规则 | 1. 校验当前用户是否为业务归属人（BA）或火车管理员2. 变更原因必填，最多500字3. 记录变更时间、操作人、变更原因4. 已纳版的需求变更时，自动从版本火车移除，释放团队容量5. 变更后需求状态统一为"草稿"，可编辑修改6. 修改完成后需重新发起评审，走完整评审流程7. 如需再次纳版，需重新走纳版流程 |
| 界面交互 | 点击"需求变更"按钮 → 弹窗填写变更原因 → 确认 → 状态变更 → 提示"已退回草稿，可重新编辑后发起评审"                                                                                                   |

##### 2.3.4.7 紧急变更（封板状态）

**系统切换规则**：

- 下拉框显示所有参与当前版本火车的系统
- 切换系统后，列表显示该系统下的所有需求
- 新增需求时，自动归属到当前选中的系统

### 2.4.6 批量操作

需求列表支持勾选多个需求后执行批量操作：

| 批量操作    | 说明                   | 操作权限     |
| ------- | -------------------- | -------- |
| 批量发起评审  | 将多个草稿状态需求一次性发起评审     | BA、火车管理员 |
| 批量取消    | 将多个需求一次性取消，需填写统一取消原因 | BA、火车管理员 |
| 批量修改优先级 | 将多个需求统一设置同一优先级       | BA、火车管理员 |

**批量操作规则**：

- 仅对符合前置条件的需求执行操作，不符合的跳过并在结果中提示
- 批量操作结果返回成功数和失败数

### 2.4.7 需求编号

- 系统自动生成需求编号，格式：`REQ-{年份}-{4位序号}`（如 REQ-2026-0001）
- 需求编号全局唯一，创建时自动赋值，不可修改
- 列表页和详情页均显示需求编号

### 2.4.8 需求删除

- 需求采用软删除策略（设置deletedAt字段），不物理删除
- 仅草稿状态可删除
- 软删除后列表不显示，但审计日志保留完整

### 2.4.9 并发编辑

- 需求编辑采用乐观锁机制（version字段）
- 提交时校验版本号，冲突时返回409提示"需求已被他人修改，请刷新后重试"
