## 三、版本火车计划（日历）

### 3.1 功能概述

版本火车计划模块以日历视图为核心，支持版本火车的创建、管理和关键节点管控。

### 3.2 用户场景

| 角色 | 场景 | 目标 |
|------|------|------|
| 火车管理员 | 创建版本火车并定义成员 | 确定哪些系统参与本次版本火车 |
| 火车管理员 | 创建火车时刻表 | 设置需求冻结、规划会等关键节点时间 |
| 火车管理员 | 管理需求搭载 | 将已就绪的需求搭载到版本火车 |
| 业务BA | 查看版本火车日历 | 了解发版计划和进度 |

### 3.3 角色职责

| 角色 | 职责范围 | 说明 |
|------|----------|------|
| **火车管理员** | 管理版本火车全生命周期 | 创建版本火车、配置搭载系统及各系统的人员（BA、产品经理、技术经理、测试经理等）、管理关键节点、需求搭载/移除/投产 |

> **说明**: 火车管理员负责版本火车的全生命周期管理，包括"谁来参加"和"什么时候做什么"。

### 3.4 功能详细设计

#### 3.4.1 版本火车创建

**操作权限**: 火车管理员

| 字段 | 说明 |
|------|------|
| 版本火车名称 | 必填，如"2026年Q2第1车" |
| 开始时间 | 必填，火车管理员自由选择（建议选周一） |
| 结束时间 | 必填，火车管理员自由选择（建议选周五） |
| 描述 | 可选，版本火车背景和目标说明 |
| 状态 | 系统自动：计划中/进行中/已完成/已取消 |

**版本火车周期规则**：
- 开始时间和结束时间由 **火车管理员** 自由选择，不再限制固定周期
- 建议开始时间选周一，结束时间选周五，便于关键节点计算
- 关键节点时间根据实际周期长度自动推算（见3.4.3）
- 同一系统同一时间只能参与一个版本火车

#### 3.4.2 搭载系统与团队容量管理

**操作权限**: 火车管理员

**核心概念**: 每个版本火车可搭载多个系统，每个系统需配置团队容量，容量决定该系统在本版本火车中能承接的需求工作量上限。

##### 系统成员配置

| 字段 | 说明 |
|------|------|
| 系统名称 | 必填，如"用户中心"、"订单系统"、"支付系统" |
| 业务BA | 必填，该系统的业务负责人 |
| 产品经理 | 必填，该系统的产品负责人 |
| 项目经理 | 可选，该系统的项目经理 |
| 技术经理 | 必填，该系统的技术负责人 |
| 开发团队 | 必填，该系统的开发人员列表 |
| 测试负责人 | 必填，该系统的测试负责人 |

##### 团队容量配置

| 字段 | 说明 |
|------|------|
| 系统名称 | 自动填充 |
| 本期可用点数 | 必填，该系统在本版本火车中的总可用工作量点数 |
| 已分配点数 | 系统自动计算，已纳版需求的工作量点数总和 |
| 剩余可用点数 | 系统自动计算 = 可用点数 - 已分配点数 |
| 容量使用率 | 系统自动计算 = 已分配点数 / 可用点数 × 100% |

**容量规则**：
- 火车管理员在创建版本火车时，为每个搭载系统配置本期可用点数
- 已纳版需求的工作量点数累计计入已分配点数
- 当已分配点数 ≥ 可用点数时，系统提示容量已满
- 容量已满时仍可手动纳版，但需二次确认
- 容量使用率以进度条形式展示，≥90% 显示红色预警

##### 需求工作量点

| 字段 | 说明 |
|------|------|
| 工作量点数 | 必填，正整数，表示该需求的开发工作量大小 |
| 填写时机 | 需求状态变更为"已就绪"前必须填写 |
| 填写权限 | 业务归属人（BA）或产品经理 |

**工作量点规则**：
- 工作量点数为必填项，范围 1-100
- 工作量点数在需求评审通过前必须填写，否则无法发起评审
- 工作量点数一旦填写，已就绪需求改点数可由项目经理或技术经理修改，不需要重新评审；已纳版需求改点数必须走需求变更。”
- 修改工作量点数时，记录修改人和修改时间

#### 3.4.3 关键节点管控

**操作权限**: 火车管理员

版本火车包含以下3个关键节点，时间根据版本开始日期和周期自动推算：

| 节点名称 | 节点说明 | 默认时间（相对于开始时间） |
|----------|----------|---------------------------|
| **统一纳版** | 确定进入本次版本火车的需求清单，已就绪的需求在此节点前完成纳版 | 周期过半前最后一个周五 |
| **统一封板** | UAT测试完成，之后严格控制需求变更 | 投产前一周周五 |
| **统一投产** | 版本火车正式发布上线 | 周期最后一天（周五） |

**不同周期节点时间对照**：

| 周期 | 统一纳版 | 统一封板 | 统一投产 |
|------|----------|----------|----------|
| 2周（14天） | 第1周周五（第5天） | 第2周周五（第12天） | 第2周周五（第14天） |
| 3周（21天） | 第2周周五（第12天） | 第3周周五（第19天） | 第3周周五（第21天） |
| 4周（28天） | 第2周周五（第12天） | 第4周周五（第26天） | 第4周周五（第28天） |

**节点时间线示意**（以4周周期、5月5日周一开始为例）：

```
第1周          第2周          第3周          第4周
5/5  5/9       5/12 5/16      5/19 5/23      5/26 5/30
│              │              │              │
├──────────────┼──────────────┼──────────────┤
│  需求准备    │  统一纳版    │  开发+测试   │  统一投产  │
│              │  5/16(周五)  │  统一封板    │  5/30(周五) │
│              │              │  5/23(周五)  │            │
└──────────────┴──────────────┴──────────────┴────────────┘
```

**节点管控规则**：
- 节点时间由火车管理员在创建版本火车时确认，支持调整
- 统一纳版节点前：可自由搭载/移除需求
- 统一纳版到统一封板之间：纳版需火车管理员审批
- 统一封板后：不再接受新需求搭载，仅支持紧急变更（需额外审批）
- 到达统一投产节点：系统自动将所有"已纳版"状态的需求批量变更为"已投产"

#### 3.4.5 纳版搭载

**功能描述**：火车管理员确认将已就绪的需求搭载到版本火车

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已就绪"，当前用户为火车管理员 |
| 输入 | 需求ID、版本火车ID |
| 输出 | 需求状态变更为"已纳版"，关联版本火车，扣减系统容量 |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 需求归属系统必须在版本火车的搭载系统列表中<br>3. 需求工作量点数必须已填写<br>4. **依赖校验**：检查依赖需求是否已满足（已纳版/已投产），未满足时按风险等级分级提示（warning：依赖未纳入本车；high：依赖状态为草稿/待评审/已拒绝；critical：依赖已取消），允许强制纳版但需填写确认说明<br>5. 纳版后扣减对应系统的可用容量<br>6. 若系统容量不足，弹窗预警但允许强制纳版<br>7. 记录纳版时间和操作人 |
| 界面交互 | 版本火车页面 → 选择需求 → 点击"确认纳版" → 校验容量 → 状态变更 → 提示"纳版成功" |

**纳版模式**：
- **AI辅助纳版**: 点击"AI排期建议" → 查看建议 → 勾选确认 → 批量纳版
- **手动纳版**: 直接勾选需求 → 点击"确认纳版" → 逐个/批量纳版

#### 3.4.6 从火车移除

**功能描述**：将已纳版的需求从版本火车移除

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已纳版"，当前用户为火车管理员 |
| 输入 | 需求ID、移除原因（必填） |
| 输出 | 需求状态变更为"已就绪"，释放系统容量 |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 移除原因必填，最多500字<br>3. 移除后释放对应系统的工作量点数容量<br>4. 记录移除时间、操作人、移除原因 |
| 界面交互 | 点击"移除"按钮 → 弹窗填写移除原因 → 确认 → 状态变更 → 释放容量 → 提示"已移除" |

#### 3.4.7 确认投产

**功能描述**：确认需求已投产上线

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已纳版"，当前用户为火车管理员 |
| 输入 | 需求ID |
| 输出 | 需求状态变更为"已投产" |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 到达统一投产节点时，系统自动将所有"已纳版"需求批量变更为"已投产"<br>3. 也支持手动逐个确认投产<br>4. 记录投产时间和操作人 |
| 界面交互 | 点击"确认投产"按钮 → 状态变更 → 提示"已投产" |

#### 3.4.8 回滚

**功能描述**：将已投产的需求回滚

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已投产"，当前用户为火车管理员 |
| 输入 | 需求ID、回滚原因（必填） |
| 输出 | 需求状态变更为"已就绪" |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 回滚原因必填，最多500字<br>3. 记录回滚时间、操作人、回滚原因 |
| 界面交互 | 点击"回滚"按钮 → 弹窗填写回滚原因 → 确认 → 状态变更 → 提示"已回滚" |

#### 3.4.9 完成版本火车

**功能描述**：火车管理员手动确认关闭版本火车，使其生命周期结束

| 项目 | 说明 |
|------|------|
| 前置条件 | 当前用户为火车管理员，当前日期不早于统一投产节点，该版本火车不存在仍处于已纳版状态的需求 |
| 输入 | 版本火车ID |
| 输出 | 版本火车状态变更为"已完成" |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 校验当前日期不早于统一投产节点<br>3. 校验该版本火车下不存在仍处于已纳版状态的需求（所有需求均已投产或已取消/已移除）<br>4. 不满足条件时提示具体原因<br>5. 记录完成时间和操作人 |
| 界面交互 | 版本火车详情页 → 点击"完成版本火车"按钮 → 校验条件 → 确认弹窗 → 状态变更 → 提示"版本火车已完成" |

## 四、Task 1 增强功能（T2 阶段实现）

> 以下功能属于 Task 1（需求池管理）的增强，因依赖 Task 2 的搭载系统数据，延后到 T2 阶段实现。

### 4.1 BA 默认系统筛选

**功能描述**：BA 角色进入需求列表页时，默认显示其第一个归属系统的需求，而非全部系统。

**背景**：MVP 阶段（T1）所有角色默认查看全部系统的需求。T2 阶段利用版本火车的搭载系统配置，实现 BA 按归属系统自动过滤。

| 项目 | 说明 |
|------|------|
| 触发条件 | 当前用户角色为 BA，进入需求列表页 |
| 默认行为 | 自动选中该 BA 在搭载系统中第一个归属系统的需求 |
| 数据来源 | 版本火车的搭载系统配置（BA 字段） |
| 切换行为 | BA 可手动切换系统下拉框查看其他系统需求 |
| 无归属系统时 | 显示全部系统需求（回退到 MVP 行为） |

**实现要点**：
- 需求列表页加载时，判断当前用户角色
- 若为 BA，查询该用户作为业务归属人的搭载系统列表
- 取第一个系统 ID 作为 systemId 默认值
- 前端系统下拉框默认选中该系统
