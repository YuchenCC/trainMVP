## 二、功能概述

版本火车计划模块以日历视图为核心，支持版本火车的创建、管理和关键节点管控。

---

## 三、功能详细设计

### 3.1 版本火车创建

**操作权限**: 火车管理员

| 字段 | 说明 |
|------|------|
| 版本火车名称 | 必填，如"2026年Q2第1车" |
| 开始时间 | 必填，版本火车起始日期（周一） |
| 版本周期 | 系统自动填充，由超级管理员全局配置（2周/3周/4周） |
| 结束时间 | 系统自动计算 = 开始时间 + 周期天数 - 1 |
| 描述 | 可选，版本火车背景和目标说明 |
| 状态 | 系统自动：规划中/进行中/已完成/已取消 |

**版本周期规则**：
- 版本火车周期由超级管理员全局配置
- 开始时间默认为最近一个周一，支持手动选择
- 结束时间根据配置的周期自动计算
- 关键节点时间根据开始时间和周期自动推算

### 3.2 搭载系统与团队容量管理

**操作权限**: 火车管理员

#### 系统成员配置

| 字段 | 说明 |
|------|------|
| 系统名称 | 必填，如"用户中心"、"订单系统" |
| 业务BA | 必填，该系统的业务负责人 |
| 产品经理 | 必填，该系统的产品负责人 |
| 项目经理 | 可选，该系统的项目经理 |
| 技术经理 | 必填，该系统的技术负责人 |
| 开发团队 | 必填，该系统的开发人员列表 |
| 测试负责人 | 必填，该系统的测试负责人 |

#### 团队容量配置

| 字段 | 说明 |
|------|------|
| 系统名称 | 自动填充 |
| 本期可用点数 | 必填，该系统在本版本火车中的总可用工作量点数 |
| 已分配点数 | 系统自动计算，已纳版需求的工作量点数总和 |
| 剩余可用点数 | 系统自动计算 = 可用点数 - 已分配点数 |
| 容量使用率 | 系统自动计算 = 已分配点数 / 可用点数 × 100% |

**容量规则**：
- 火车管理员在创建版本火车时，为每个搭载系统配置本期可用点数
- 已纳版需求的工作量点数累计计入已分配点数
- 当已分配点数 ≥ 可用点数时，系统提示容量已满
- 容量已满时仍可手动纳版，但需二次确认
- 容量使用率以进度条形式展示，≥90% 显示红色预警

### 3.3 关键节点管控

**操作权限**: 火车管理员

版本火车包含以下3个关键节点，时间根据版本开始日期和周期自动推算：

| 节点名称 | 节点说明 | 默认时间 |
|----------|----------|----------|
| **统一纳版** | 确定进入本次版本火车的需求清单 | 周期过半前最后一个周五 |
| **统一封板** | UAT测试完成，之后严格控制需求变更 | 投产前一周周五 |
| **统一投产** | 版本火车正式发布上线 | 周期最后一天（周五） |

**不同周期节点时间对照**：

| 周期 | 统一纳版 | 统一封板 | 统一投产 |
|------|----------|----------|----------|
| 2周（14天） | 第1周周五（第5天） | 第2周周五（第12天） | 第2周周五（第14天） |
| 3周（21天） | 第2周周五（第12天） | 第3周周五（第19天） | 第3周周五（第21天） |
| 4周（28天） | 第2周周五（第12天） | 第4周周五（第26天） | 第4周周五（第28天） |

**节点管控规则**：
- 节点时间由火车管理员在创建版本火车时确认，支持自由调整
- 统一纳版节点前：可自由搭载/移除需求
- 统一纳版到统一封板之间：纳版需火车管理员审批
- 统一封板后：不再接受新需求搭载，仅支持紧急变更（需额外审批）
- 到达统一投产节点：系统自动将所有"已纳版"状态的需求批量变更为"已投产"

### 3.4 纳版搭载

**功能描述**：火车管理员确认将已就绪的需求搭载到版本火车

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已就绪"，当前用户为火车管理员 |
| 输入 | 需求ID、版本火车ID |
| 输出 | 需求状态变更为"已纳版"，关联版本火车，扣减系统容量 |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 需求归属系统必须在版本火车的搭载系统列表中<br>3. 需求工作量点数必须已填写<br>4. **依赖校验**：检查依赖需求是否已满足，未满足时按风险等级分级提示（warning/high/critical），允许强制纳版但需填写确认说明<br>5. 纳版后扣减对应系统的可用容量<br>6. 若系统容量不足，弹窗预警但允许强制纳版<br>7. 记录纳版时间和操作人 |

**纳版模式**：
- **AI辅助纳版**: 点击"AI排期建议" → 查看建议 → 勾选确认 → 批量纳版
- **手动纳版**: 直接勾选需求 → 点击"确认纳版" → 逐个/批量纳版

### 3.5 从火车移除

**功能描述**：将已纳版的需求从版本火车移除

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已纳版"，当前用户为火车管理员 |
| 输入 | 需求ID、移除原因（必填） |
| 输出 | 需求状态变更为"已就绪"，释放系统容量 |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 移除原因必填，最多500字<br>3. 移除后释放对应系统的工作量点数容量<br>4. 记录移除时间、操作人、移除原因 |

### 3.6 确认投产

**功能描述**：确认需求已投产上线

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已纳版"，当前用户为火车管理员 |
| 输入 | 需求ID |
| 输出 | 需求状态变更为"已投产" |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 到达统一投产节点时，系统自动将所有"已纳版"需求批量变更为"已投产"<br>3. 也支持手动逐个确认投产<br>4. 记录投产时间和操作人 |

### 3.7 回滚

**功能描述**：将已投产的需求回滚

| 项目 | 说明 |
|------|------|
| 前置条件 | 需求状态为"已投产"，当前用户为火车管理员 |
| 输入 | 需求ID、回滚原因（必填） |
| 输出 | 需求状态变更为"已就绪" |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 回滚原因必填，最多500字<br>3. 记录回滚时间、操作人、回滚原因 |

### 3.8 完成版本火车

**功能描述**：火车管理员手动确认关闭版本火车

| 项目 | 说明 |
|------|------|
| 前置条件 | 当前用户为火车管理员，当前日期不早于统一投产节点，该版本火车不存在仍处于已纳版状态的需求 |
| 输入 | 版本火车ID |
| 输出 | 版本火车状态变更为"已完成" |
| 业务规则 | 1. 校验当前用户是否为火车管理员<br>2. 校验当前日期不早于统一投产节点<br>3. 校验该版本火车下不存在仍处于已纳版状态的需求<br>4. 不满足条件时提示具体原因<br>5. 记录完成时间和操作人 |

### 3.9 日历视图

以日历形式展示版本火车和关键节点：

- **月视图**: 显示当月所有版本火车和节点
- **周视图**: 显示当周节点安排
- **节点标识**: 不同颜色区分节点类型和状态
- **容量概览**: 在日历中展示各系统的容量使用率
- **点击查看**: 点击节点查看详情

## 四、界面设计

### 4.1 版本火车详情页

**功能**：纳版管理、容量管理、操作按钮

**页面元素**：
- 火车基本信息（名称、周期、状态、关键节点）
- 团队容量概览（各系统容量使用率）
- 已纳版需求列表（可移除、确认投产、回滚）
- 待纳版需求列表（已就绪状态，可勾选纳版）
- 操作按钮：AI排期建议、确认纳版、批量确认投产

**操作按钮显示规则**：

| 需求状态 | 火车管理员可见操作 |
|----------|-------------------|
| 已纳版 | 移除、确认投产、取消 |
| 已投产 | 回滚、取消 |

### 4.2 火车日历页面

**功能**：日历视图展示版本火车和关键节点

**页面元素**：
- 月视图/周视图切换
- 版本火车时间轴展示
- 关键节点标识（统一纳版/统一封板/统一投产）
- 容量预警标识
- 节点详情弹窗

**节点颜色标识**：
- 🟢 绿色：统一纳版
- 🟠 橙色：统一封板
- 🔵 蓝色：统一投产
- ⚪ 灰色：已完成
- 🟡 黄色：进行中
- 🔴 红色：容量预警（≥90%）
