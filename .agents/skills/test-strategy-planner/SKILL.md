# 测试策略规划

## 用途

把需求、用户故事、测试案例文档、代码范围和现有测试资产整理成可执行的分层测试策略。

本 skill 只负责策略规划和覆盖分析，不执行测试，不判断单测门禁是否通过，不编写 JUnit5/JMeter/Playwright 脚本，不生成最终管理层测试覆盖总报告。

## 工作边界

| 负责 | 不负责 |
|---|---|
| 识别需求范围、代码影响和测试范围 | 运行单测、接口测试、E2E 测试 |
| 读取已有测试案例文档并做覆盖分析 | 用 API/E2E/人工验证结果抵扣单测覆盖率 |
| 将测试点拆到 L1/L2/L3/人工验证 | 判断增量覆盖率门禁最终是否通过 |
| 输出主报告、测试案例覆盖分析、测试方式对照表 | 直接生成 JUnit5、JMeter、Playwright 代码 |
| 给出后续 skill 路由和输入建议 | 替代 `unit-test-governance` 或 `test-report-generator` |

## 硬性执行门禁：先确认，再分析

正式分析前必须先让用户确认输入范围。未确认前，只允许做文件搜索、候选清单和问题澄清，不要生成覆盖结论。

在任何正式覆盖分析、代码结论、TODO 结论或报告文件生成之前，必须先向用户输出“分析前输入确认清单”，并等待用户明确回复确认。

即使用户已经在请求中提供了分析范围、测试案例来源、需求/设计来源、代码范围、口径差异或类似表格，也只能视为“待确认输入草案”，不能自动视为已确认。

只有当用户明确表达“确认”“按这个范围分析”“可以开始”“纳入以上范围”“继续生成报告”等确认意图后，才能进入正式分析和写报告。

如果用户只是要求“用 test-strategy-planner 分析一下”，下一步必须是确认清单；不得直接读取大量代码后产出结论或报告。


必须确认的输入：

| 输入 | 说明 | 示例 |
|---|---|---|
| `模块名` | Task 编号、需求编号或功能模块 | `T1-需求池`、`T4-变更率`、`紧急变更` |
| `需求/设计来源` | 本地或飞书 MCP 中的 PRD、用户故事、详细设计、接口设计等 | `用户故事 v2.0`、飞书文档/Wiki 链接 |
| `测试案例来源` | 已确认适用于本次范围的本地或飞书测试案例文档 | `RT-T2-US2.3-测试案例_v1.1.md`、飞书文档链接 |
| `文件路径` | 需要分析的前端/后端代码文件或目录 | `service.ts`、`index.tsx`、`routes/` |
| `提交范围` | Git 提交、分支或范围 | `HEAD~5`、`feature/xxx`、`abc123..def456` |
| `函数列表` | 需要重点分析或补单测的具体函数 | `sanitizeDescription`、`generateReqCode` |
| `存量报告` | 已有策略报告、覆盖报告、handoff、历史分析 | `reports/test-strategy/*.md` |

确认规则：

- 如果用户已经明确给出某项输入，可以直接列入确认清单；不要重复追问。
- 如果关键输入缺失，先输出“待确认输入清单”，请用户补充或确认“不适用”。
- 如果找到候选测试案例、候选需求文档、候选代码文件或存量报告，必须先列给用户确认是否纳入本次分析。
- 飞书搜索结果也只是候选来源；用户确认前只能读取标题、链接、类型、更新时间等候选信息，不得据此生成覆盖结论。
- 测试案例是必须输入。用户未确认测试案例来源前，不要进入大范围代码分析；如果确认没有测试案例，停止在输入确认阶段，要求先补测试案例，不生成策略报告。
- 存量报告必须询问用户是否直接复用。用户未确认复用时，旧报告只能当索引；用户明确不复用时，不要在输出中引用旧报告结论，也不要写入旧报告内容，必须重新生成。

确认提示模板：

```markdown
请先确认本次分析输入：

| 输入 | 当前识别结果 | 是否纳入 |
|---|---|---|
| 模块名 |  | 待确认 |
| 需求/设计来源 |  | 待确认 |
| 测试案例来源 |  | 必须确认 |
| 文件路径 |  | 待确认 |
| 提交范围 |  | 待确认 |
| 函数列表 |  | 待确认/不适用 |
| 存量报告 |  | 是否复用待确认 |

确认后我再开始分析；如果某项不适用，请直接标注“不适用”。
```
## 必读输入优先级

1. 测试案例文档：如果用户提供或目录中存在，必须先读。
2. 需求/PRD/用户故事/详细设计：用于确认业务口径。
3. 代码范围：service、route/controller、frontend page/component/service、shared types。
4. 现有测试文件：只能作为已有覆盖证据，不能替代测试案例覆盖分析。
5. 自测检查点快照：需要横向补充风险时，读取 `references/self-test-checkpoints.md`；该文件已复制共享检查点全文。
6. 飞书云文档/Wiki：用户提供飞书链接、关键词、文件夹名称或明确要求查找飞书文件时，优先读取 `references/lark-document-source.md` 并通过飞书 MCP 搜索；先搜索飞书候选并完成输入确认，再读取已确认正文。此类请求在用户允许前不要搜索本地目录或本地同名文件。

如果测试案例文档和用户口头描述不一致，必须在报告中明确写出“口径差异”。例如用户说“版本火车列表/详情”，但测试案例写的是“班次列表/详情”，应先按测试案例编号分析，并提示需要统一命名。

## 小模型文件读取规则

当代码范围较大、模型上下文较小，或用户明确说使用小模型时，必须使用“测试案例索引法”。详细执行规则读取 `references/small-model-reading.md`。

触发条件：

- 小模型：参数规模约 30B 以下、有效上下文约 32k 以下，或实际表现为容易漏读文件、混淆旧报告和源文件、无法稳定处理长代码。
- 代码范围较大：满足任一条件即算较大：跨前后端仓库、超过 10 个相关文件、超过 1500 行相关代码、超过 5 个接口、超过 3 个页面/角色/状态流转、测试案例超过 10 条。

## 找不到测试案例文档时必须询问

测试案例文档是本 skill 的必需输入。如果用户没有直接给出测试案例文档路径，并且在常见目录未找到匹配文件，不要生成策略报告。必须先询问用户测试案例文档在哪里。

询问格式：

```markdown
我还没有找到本次范围对应的测试案例文档。测试案例是本次策略分析的必需输入，请告诉我测试案例文档在哪里。

我已检查：
- {已检查目录1}
- {已检查目录2}

候选文件：
- {候选文件1}
- {候选文件2}

请提供测试案例文档路径，或确认以上候选文件中哪一个适用于本次范围。确认后我再开始分析。
```

常见搜索位置：

- `03-需求与设计/**/测试案例/*.md`
- `03-需求与设计/**/测试用例/*.md`
- `docs/**/测试案例*.md`
- `docs/**/test-case*.md`
- 当前 task/requirement 目录下的 `测试案例`、`测试用例`、`cases`、`test-cases` 子目录。

没有已确认测试案例文档时，只能输出待确认清单或候选文件清单，不允许进入正式分析，也不允许生成主报告。

## 分层模型

| 层级 | 名称 | 常见工具 | 覆盖重点 | 门禁关系 |
|---|---|---|---|---|
| L1 | 后端单测 | JUnit5、Jest、Vitest 等后端单测工具 | 业务规则、状态流转、参数校验、异常分支、金额/时间计算、mapper/converter、service 分支 | 后端硬门禁候选项，最终由 `unit-test-governance` 或 unit-gate 判定 |
| L1 | 前端单测建议 | Jest、Vitest、React/Vue 单测工具 | 表单校验、权限显隐、分页排序筛选、金额/日期计算、hook/composable、工具函数、状态管理 | 仅条件触发建议，不是默认硬门禁 |
| L2 | API/JMeter | JMeter 或团队现有 API 工具 | 接口契约、鉴权、入参校验、关键状态变化、错误码、幂等、跨服务链路 | 不能抵扣后端单测门禁 |
| L3 | UI 自动化 | Playwright、Cypress、React/Vue 测试工具 | P0/P1 用户流程、关键页面跳转、核心按钮、跨页面联动 | 只覆盖高价值稳定流程，不能抵扣后端单测门禁 |
| M | 人工验证 | SIT 环境、截图、日志、数据库查询、HAR、验证记录 | 低频 UI 展示、截图确认、SIT 环境验证、日志证据、需人工判断的视觉或业务状态 | 不能抵扣后端单测门禁 |

读取 `references/layered-testing.md` 获取完整分层判断规则。分层时必须区分 L1 后端单测、L1 前端单测建议、L2 API/JMeter、L3 UI 自动化、M 人工验证；M 和 L2/L3 不能抵扣后端单测门禁。需要人工验证证据时读取 `references/manual-evidence.md`。

## 测试优先级和门禁口径

默认优先级必须是：

1. L1 单元测试优先：满足单测门禁，默认要求增量覆盖率 > 80%，通过率 100%。
2. L2 API 自动化其次：验证接口契约、鉴权、权限、参数、状态流转和数据库副作用。
3. L3 UI 自动化最后：只覆盖少量 P0 主路径和高回归风险交互。
4. 人工验证只在用户确认启用时纳入策略；启用后必须保留可审计证据。

规则：

- L2/API、L3/UI、人工验证不能抵扣 L1 单测覆盖率。
- 如果需求天然偏接口或页面，也必须先识别可单测的增量逻辑，包括分页计算、字段聚合、映射、状态判断、权限按钮矩阵、异常分支和纯函数。
- 对于前端页面，如果核心逻辑都写在组件内，应在策略中建议抽出 `*.logic.ts` 或 composable/helper，以便单测覆盖。
- 如用户确认启用人工验证，必须输出人工验证证据清单。读取 `references/manual-evidence.md` 获取 SIT 人工验证证据采集规则。
- 当策略中存在可由人工验证替代的 UI 场景时，必须先询问用户是否启用人工验证证据包。用户明确回复“要/需要/可以/启用/走人工验证”后，自动创建 `evidence/{scope}/{case-id}/` 目录和 `summary.md` 指引；用户未确认前不要创建证据目录。

## 覆盖判断规则

- 测试案例是主线；先输出案例分层策略，再使用逻辑代码验证策略，最后使用测试代码核对覆盖结果。
- 逻辑代码用于判断测试案例的实现边界和分层策略；测试代码只用于判断已有覆盖，不得混用。
- 推荐测试层级必须在读取测试代码前，根据测试案例语义和逻辑代码确定；已有测试工具、测试文件名和测试数量不得反向决定推荐层级。
- 报告必须分别记录“推荐测试层级”“已有测试资产实际层级”和“覆盖结论”；已有资产属于证据，不属于策略输入。
- commit、branch 或文件列表只用于标识本次变更影响和增量风险，不得限制测试案例覆盖分析的逻辑代码范围，除非用户明确要求“仅分析本次变更”。
- API 被调用过不等于测试案例已覆盖，必须有对应断言。
- UI 展示、按钮可见性、页面跳转、标签页切换、返回列表必须归到 L3 或人工验证，不能用 L2 抵扣。
- 数据库副作用、接口响应字段、鉴权、错误码归到 L2。
- 单测优先覆盖可隔离规则、纯函数、状态判断、参数边界、映射和计算逻辑；策略必须说明如何满足增量覆盖率 > 80% 和通过率 100%。
- 现有测试若覆盖行为但文件名/编号不匹配，应标为“部分覆盖，可复用但需映射或重命名”。
- 如果报告只有提交代码和提交测试文件，没有逐案例分层策略，则不得输出为完整测试案例覆盖分析。
- 不要直接给出单测门禁通过/不通过，只能写“需交给 `unit-test-governance` 判断”。
- UI 自动化可被人工验证替代时，必须说明替代边界：P0 主路径和高回归风险权限显隐优先自动化；低风险展示、文案、截图类验收可人工验证，但必须保留证据。

## 工作流程与核心规则

以下规则贯穿整个工作流程，不因使用大模型或小模型而改变：

- 先确认测试案例。测试案例是必须输入；未确认测试案例来源前，不要进入大范围代码分析；没有测试案例时停止分析并要求补充。
- 先用测试案例建立索引，再定位和读取候选代码；具体索引字段和读取方法遵循 `references/small-model-reading.md`。
- 每轮只处理一个可闭环分析单元。限制的不是文件数，而是能否完成：`测试案例 -> 分层策略 -> 逻辑代码 -> 测试代码覆盖 -> 覆盖结论 -> TODO`。
- 测试文件只在测试案例分层策略和逻辑代码分析完成后，用于核对当前覆盖结果；不能用测试代码决定推荐层级或替代测试案例分析。
- 禁止从代码反推覆盖。代码发现的问题只能标记为“补充风险”，不能当成测试案例覆盖结论。

1. 识别用户请求中的初始范围：commit、branch、requirement、task、用户故事、模块名、文件路径、函数列表或测试案例路径。
2. 只做候选搜索和输入整理：查找候选需求/设计、测试案例、前后端代码、现有测试文件、提交范围和存量报告；用户要求查找飞书文件或提供飞书文件夹名时，先按 `references/lark-document-source.md` 使用飞书 MCP 搜索飞书候选，未经用户允许不搜索本地；此阶段不要生成覆盖结论。
3. 输出“分析前输入确认”清单，让用户确认需求/设计来源、测试案例来源、代码范围、提交范围、函数列表、存量报告是否复用。
4. 用户确认前不得进入正式分析；只能继续澄清缺失输入、补充候选清单或询问是否不适用。
5. 用户确认后，按已确认输入读取本地或飞书测试案例文档；如果用户确认没有测试案例，记录该前提后继续。
6. 读取已确认的本地或飞书需求、PRD、用户故事或详细设计，确认业务口径和口径差异；飞书来源必须记录标题、链接或 token、读取时间和读取状态。
7. 以已确认测试案例为主线，先形成每个案例的测试目标、验证对象、推荐层级和分层理由；此时不读取测试代码作为策略依据。
8. 按测试案例锚点读取相关逻辑代码（route/controller、service、domain、页面、组件、logic/composable、shared types 等），验证分层策略和实现边界；代码范围不局限于本次提交。
9. 在逻辑代码分析完成后，单独读取已有测试代码和其他测试资产，先识别其实际测试层级，再核对与测试案例的对应关系、断言和实际覆盖；测试代码只作为覆盖证据。测试文件必须实际读取。不要只看提交里有没有新增测试；必须查找并读取已有单测、API 测试、JMeter、Playwright/Cypress 等测试资产。
10. 以测试案例编号做覆盖分析，如果用户确认不用存量报告的情况下，不要用存量报告结论替代源文件分析。
11. 读取 `references/self-test-checkpoints.md`，按自测检查点快照筛出本次适用的补充风险；不要用检查点替代测试案例覆盖结论。
12. 先规划 L1 后端单测候选覆盖，再标注 L1 前端单测建议，然后规划 L2 API/JMeter，最后规划 L3 UI 自动化；M 人工验证仅在用户确认启用时纳入。
13. 如果存在人工验证候选场景，询问用户是否启用人工验证证据包；用户确认后立即创建证据目录和 `summary.md` 模板，用户未确认前不要创建。
14. 为每个测试点标注 L1/L2/L3/M、当前覆盖状态、证据来源、缺口和优先级。
15. 扫描项目已有 `.agents/skills/*/SKILL.md`，按“对话中的后续 skill 动态推荐”规则生成建议；后续 skill 只在最终对话中输出，不写入主报告、附件或 TODO 表。
16. 生成中文 Markdown 文件。
17. 生成文档后立即执行输出自检：反读文件、核对章节和表格、确认路径存在、确认内容不是空壳或截断。
18. 对话回复只给路径、范围、关键结论、人工验证目录路径（如已创建）、输出自检结果和下一步建议。

## 没有测试案例文档时的处理规则

本 skill 不提供“无测试案例文档”的正式策略报告模式。

如果用户确认当前确实没有测试案例文档：

- 不生成 `requirement-{scope}-测试策略与覆盖分析-v1.md` 主报告。
- 不生成测试案例方式对照表。
- 只输出待办建议：先补齐测试案例文档，再重新调用本 skill。
- 可以帮助用户整理“测试案例补充清单”，但该清单不是测试策略覆盖分析报告。

## 主报告输出规则

主报告必须按 `references/output-template.md` 生成，不要在 `SKILL.md` 中另起一套报告结构。

使用规则：

- 有测试案例文档时，使用 `references/output-template.md` 中“一、有测试案例文档时”的主报告模板。
- 没有测试案例文档时，不生成主报告；必须先要求补充或确认正确测试案例文档。
- 报告章节、表格字段、分层口径、TODO 和风险项以 `references/output-template.md` 为准。
- 如果本文件中的约束和模板有冲突，以模板为输出结构，以本文件的门禁和流程规则补充判断口径。
- 主报告必须体现：分析前确认结果、测试案例来源、需求/设计来源、代码范围、存量报告是否复用、测试案例分层策略、逻辑代码验证重点、L1 后端单测门禁、L1 前端单测建议、L2 API/JMeter、L3 UI 自动化、M 人工验证、自测检查点映射、覆盖现状、TODO、风险；后续 skill 建议只在最终对话中输出。

## 生成文档后的输出自检

每次生成或覆盖 Markdown 报告、附件、evidence summary 后，必须立即做输出自检。自检失败时，不要声称完成；应修复后重新读取，直到通过或明确说明阻塞原因。

自检步骤：

1. 用文件读取命令反读刚生成的每个文件，至少读取开头和关键章节；不要只依赖写文件命令返回成功。
2. 确认文件路径存在，文件大小大于 0，中文内容可按 UTF-8 正常读取。
3. 对主报告核对 `references/output-template.md` 要求的章节：分析范围、分层测试方案、测试案例分层策略与需求-案例覆盖矩阵、自测检查点映射、覆盖现状、TODO、人工验证方案、风险提醒；不要求报告包含后续 skill 建议。
4. 对测试案例对照表核对统计概览、对照表、测试文件建议、逻辑代码与分层策略映射、测试代码与覆盖结果映射、人工验证替代场景。
5. 如创建人工验证 evidence，核对 `summary.md` 和 `screenshots/`、`logs/`、`db/`、`api/` 目录是否存在，且 `summary.md` 包含基本信息、验证结论、失败/阻塞说明、附件清单和风险说明。
6. 核对报告是否明确写入本轮确认结果：测试案例来源、需求/设计来源、代码范围、存量报告是否复用、自测检查点是否纳入、人工验证是否启用。
7. 核对报告未违反用户确认的口径；例如用户确认不复用存量报告时，不得引用旧报告结论。
8. 在最终回复中简短写明“已完成输出自检”以及检查过的文件。
9. 使用飞书来源时，核对报告已记录来源载体、标题、链接或 token、读取时间和读取状态；不得把仅搜索到但未读取正文的候选写成已分析来源。

推荐步骤：

1. 先扫描项目内已有 skill：读取 `.agents/skills/*/SKILL.md` 的 `name`、`description` 和用途说明。

## 生成文件后的飞书文档与审批任务

生成报告、附件并完成输出自检后，才可以询问用户是否需要发布到飞书；未完成输出自检前，不得创建飞书文档或审批任务。

询问用户：

```markdown
本次文件已生成并完成输出自检。

请确认是否通过飞书 MCP 执行以下操作：
- 创建飞书文档：是 / 否
- 创建“自测策略确认”审批任务：是 / 否
- 审批人：通过飞书 MCP 查询可访问用户后确认

默认发布文件：主报告；如需同时发布附件，请一并确认。
```

执行规则：

1. 用户未明确确认前，不通过飞书 MCP 创建文档，不创建审批任务，不发送给任何飞书用户。
2. 创建飞书文档时，默认使用已完成自检的主报告；附件只有在用户确认后才一并创建或关联。
3. 创建审批任务时，任务名称固定为“自测策略确认”，任务内容必须包含飞书文档链接、分析范围、测试案例来源、主要风险、TODO 和确认要求。
4. 创建审批任务前，必须通过飞书 MCP 的用户目录能力查询当前可访问的飞书用户，并展示候选用户的姓名及可确认身份的信息（邮箱或 open_id）。
5. 如果查询结果只有一个用户，也必须向用户展示该用户并取得最终确认；不能因为只有一个候选人就自动发送。
6. 如果查询结果有多个用户，必须让用户明确选择审批人；如果查不到用户、身份信息不足或用户无法确认，停止发送并说明原因。
7. 执行前必须确认当前飞书 MCP 已授权对应文档、用户查询和审批操作所需权限；权限不足或 MCP 不可用时，说明阻塞原因，不得声称已创建。
8. 创建完成后，必须通过飞书 MCP 验证返回的文档链接、任务标识、审批人和状态，并在最终回复中分别说明创建结果；失败时记录失败原因和可重试事项。
