# 测试策略规划

## 用途

把需求、用户故事、测试案例文档、代码范围和现有测试资产整理成可执行的分层测试策略。

本 skill 只负责策略规划和覆盖分析，不执行测试，不判断单测门禁是否通过，不编写 JUnit5/JMeter/Playwright 脚本，不生成最终管理层测试覆盖总报告。

## 工作边界

| 负责 | 不负责 |
|---|---|
| 识别需求范围、代码影响和测试范围 | 运行单测、接口测试、E2E 测试 |
| 读取已有测试案例文档并做覆盖分析 | 用 API/E2E/人工验证结果抵扣单测覆盖率 |
| 将测试点拆到 L1/L2/L3/人工验证 | 判断增量覆盖率门禁最终是否通过 |
| 输出主报告、测试案例覆盖分析、测试方式对照表 | 直接生成 JUnit5、JMeter、Playwright 代码 |
| 给出后续 skill 路由和输入建议 | 替代 `unit-test-governance` 或 `test-coverage-report` |

## 硬性执行门禁：先确认，再分析

正式分析前必须先让用户确认输入范围。未确认前，只允许做文件搜索、候选清单和问题澄清，不要生成覆盖结论。

在任何正式覆盖分析、代码结论、TODO 结论或报告文件生成之前，必须先向用户输出“分析前输入确认清单”，并等待用户明确回复确认。

即使用户已经在请求中提供了分析范围、测试案例来源、需求/设计来源、代码范围、口径差异或类似表格，也只能视为“待确认输入草案”，不能自动视为已确认。

只有当用户明确表达“确认”“按这个范围分析”“可以开始”“纳入以上范围”“继续生成报告”等确认意图后，才能进入正式分析和写报告。

如果用户只是要求“用 test-strategy-planner 分析一下”，下一步必须是确认清单；不得直接读取大量代码后产出结论或报告。


必须确认的输入：

| 输入 | 说明 | 示例 |
|---|---|---|
| `模块名` | Task 编号、需求编号或功能模块 | `T1-需求池`、`T4-变更率`、`紧急变更` |
| `需求/设计来源` | PRD、用户故事、详细设计、接口设计等 | `用户故事 v2.0`、`US2.3 详细设计` |
| `测试案例来源` | 已确认适用于本次范围的测试案例文档 | `RT-T2-US2.3-测试案例_v1.1.md` |
| `文件路径` | 需要分析的前端/后端代码文件或目录 | `service.ts`、`index.tsx`、`routes/` |
| `提交范围` | Git 提交、分支或范围 | `HEAD~5`、`feature/xxx`、`abc123..def456` |
| `函数列表` | 需要重点分析或补单测的具体函数 | `sanitizeDescription`、`generateReqCode` |
| `存量报告` | 已有策略报告、覆盖报告、handoff、历史分析 | `reports/test-strategy/*.md` |

确认规则：

- 如果用户已经明确给出某项输入，可以直接列入确认清单；不要重复追问。
- 如果关键输入缺失，先输出“待确认输入清单”，请用户补充或确认“不适用”。
- 如果找到候选测试案例、候选需求文档、候选代码文件或存量报告，必须先列给用户确认是否纳入本次分析。
- 测试案例是必须输入。用户未确认测试案例来源前，不要进入大范围代码分析；如果确认没有测试案例，停止在输入确认阶段，要求先补测试案例，不生成策略报告。
- 存量报告必须询问用户是否直接复用。用户未确认复用时，旧报告只能当索引；用户明确不复用时，不要在输出中引用旧报告结论，也不要写入旧报告内容，必须重新生成。

确认提示模板：

```markdown
请先确认本次分析输入：

| 输入 | 当前识别结果 | 是否纳入 |
|---|---|---|
| 模块名 |  | 待确认 |
| 需求/设计来源 |  | 待确认 |
| 测试案例来源 |  | 必须确认 |
| 文件路径 |  | 待确认 |
| 提交范围 |  | 待确认 |
| 函数列表 |  | 待确认/不适用 |
| 存量报告 |  | 是否复用待确认 |

确认后我再开始分析；如果某项不适用，请直接标注“不适用”。
```
## 必读输入优先级

1. 测试案例文档：如果用户提供或目录中存在，必须先读。
2. 需求/PRD/用户故事/详细设计：用于确认业务口径。
3. 代码范围：service、route/controller、frontend page/component/service、shared types。
4. 现有测试文件：只能作为已有覆盖证据，不能替代测试案例覆盖分析。
5. 自测检查点快照：需要横向补充风险时，读取 eferences/self-test-checkpoints.md；该文件已复制共享检查点全文。

如果测试案例文档和用户口头描述不一致，必须在报告中明确写出“口径差异”。例如用户说“版本火车列表/详情”，但测试案例写的是“班次列表/详情”，应先按测试案例编号分析，并提示需要统一命名。

## 小模型文件读取规则

当代码范围较大、模型上下文较小，或用户明确说使用小模型时，必须使用“测试案例索引法”。详细执行规则读取 `references/small-model-reading.md`。

触发条件：

- 小模型：参数规模约 30B 以下、有效上下文约 32k 以下，或实际表现为容易漏读文件、混淆旧报告和源文件、无法稳定处理长代码。
- 代码范围较大：满足任一条件即算较大：跨前后端仓库、超过 10 个相关文件、超过 1500 行相关代码、超过 5 个接口、超过 3 个页面/角色/状态流转、测试案例超过 10 条。

核心规则：

1. 先确认测试案例。测试案例是必须输入；未确认测试案例来源前，不要进入大范围代码分析；没有测试案例时停止分析并要求补充。
2. 先用测试案例建索引。从用例编号、步骤、预期结果中提取页面、API、字段、状态、权限、异常锚点，再定位候选代码。
3. 每轮只处理一个可闭环分析单元。限制的不是文件数，而是能否完成：`测试案例/需求 -> 代码 -> 已有测试 -> 覆盖结论 -> TODO`。
4. 测试文件必须实际读取。不要只看提交里有没有新增测试；必须查找并读取已有单测、API 测试、JMeter、Playwright/Cypress 等测试资产。
5. 禁止从代码反推覆盖。代码发现的问题只能标记为“补充风险”，不能当成测试案例覆盖结论。

每轮读取至少输出三张表：

| 测试案例 | 锚点 | 候选代码 | 相关性 | 是否已读 | 下一步 |
|---|---|---|---|---|---|

| 来源 | 事实 | 对测试策略的影响 |
|---|---|---|

| 测试案例 | 当前覆盖 | 证据 | 缺口 | TODO |
|---|---|---|---|---|
## 找不到测试案例文档时必须询问

测试案例文档是本 skill 的必需输入。如果用户没有直接给出测试案例文档路径，并且在常见目录未找到匹配文件，不要生成策略报告。必须先询问用户测试案例文档在哪里。

询问格式：

```markdown
我还没有找到本次范围对应的测试案例文档。测试案例是本次策略分析的必需输入，请告诉我测试案例文档在哪里。

我已检查：
- {已检查目录1}
- {已检查目录2}

候选文件：
- {候选文件1}
- {候选文件2}

请提供测试案例文档路径，或确认以上候选文件中哪一个适用于本次范围。确认后我再开始分析。
```

常见搜索位置：

- `03-需求与设计/**/测试案例/*.md`
- `03-需求与设计/**/测试用例/*.md`
- `docs/**/测试案例*.md`
- `docs/**/test-case*.md`
- 当前 task/requirement 目录下的 `测试案例`、`测试用例`、`cases`、`test-cases` 子目录。

没有已确认测试案例文档时，只能输出待确认清单或候选文件清单，不允许进入正式分析，也不允许生成主报告。
## 测试优先级和门禁口径

默认优先级必须是：

1. L1 后端单测优先：满足单测门禁，默认要求增量覆盖率 > 80%，通过率 100%。后端 L1 是默认硬门禁，必须单独展示，且不能被 L2/L3/人工验证抵扣。如果没有覆盖率报告或单测执行结果，只能写“阻塞/待补”，不能写通过。
2. L2 API 自动化其次：验证接口契约、鉴权、权限、参数、状态流转和数据库副作用。
3. L3 UI 自动化最后：只覆盖少量 P0 主路径和高回归风险交互。
4. 人工验证只在用户确认启用时纳入策略；启用后必须保留可审计证据。

规则：

- L2/API、L3/UI、人工验证不能抵扣 L1 单测覆盖率。
- 如果需求天然偏接口或页面，也必须先识别可单测的增量逻辑，包括分页计算、字段聚合、映射、状态判断、权限按钮矩阵、异常分支和纯函数。
- 前端不默认要求单测覆盖率门禁；建议只在以下情况补前端单测：表单校验、字段格式化、权限显隐、按钮状态、分页排序筛选、金额日期数量计算、React hook/Vue composable、工具函数、状态管理。纯展示、简单表格、普通弹窗、文案展示，优先走 L3 UI 或人工验证。
- 对于前端页面，如果核心逻辑都写在组件内，应在策略中建议抽出 `*.logic.ts` 或 composable/helper，以便单测覆盖。
- 如用户确认启用人工验证，必须输出人工验证证据清单。读取 `references/manual-evidence.md` 获取 SIT 人工验证证据采集规则。
- 当策略中存在可由人工验证替代的 UI 场景时，必须先询问用户是否启用人工验证证据包。用户明确回复“要/需要/可以/启用/走人工验证”后，自动创建 `evidence/{scope}/{case-id}/` 目录和 `summary.md` 指引；用户未确认前不要创建证据目录。

## 分层模型

| 层级 | 名称 | 常见工具 | 覆盖重点 | 门禁关系 |
|---|---|---|---|---|
| L1 | 后端单测 | JUnit5、Jest、Vitest 等后端单测工具 | 业务规则、状态流转、参数校验、异常分支、金额/时间计算、mapper/converter、service 分支 | 后端硬门禁候选项，最终由 `unit-test-governance` 或 unit-gate 判定 |
| L1 | 前端单测建议 | Jest、Vitest、React/Vue 单测工具 | 表单校验、权限显隐、分页排序筛选、金额/日期计算、hook/composable、工具函数、状态管理 | 仅条件触发建议，不是默认硬门禁 |
| L2 | API/JMeter | JMeter 或团队现有 API 工具 | 接口契约、鉴权、入参校验、关键状态变化、错误码、幂等、跨服务链路 | 不能抵扣后端单测门禁 |
| L3 | UI 自动化 | Playwright、Cypress、React/Vue 测试工具 | P0/P1 用户流程、关键页面跳转、核心按钮、跨页面联动 | 只覆盖高价值稳定流程，不能抵扣后端单测门禁 |
| M | 人工验证 | SIT 环境、截图、日志、数据库查询、HAR、验证记录 | 低频 UI 展示、截图确认、SIT 环境验证、日志证据、需人工判断的视觉或业务状态 | 不能抵扣后端单测门禁 |

读取 `references/layered-testing.md` 获取完整分层判断规则。分层时必须区分 L1 后端单测、L1 前端单测建议、L2 API/JMeter、L3 UI 自动化、M 人工验证；M 和 L2/L3 不能抵扣后端单测门禁。需要人工验证证据时读取 `references/manual-evidence.md`。
## 工作流程

1. 识别用户请求中的初始范围：commit、branch、requirement、task、用户故事、模块名、文件路径、函数列表或测试案例路径。
2. 只做候选搜索和输入整理：查找候选需求/设计、测试案例、前后端代码、现有测试文件、提交范围和存量报告；此阶段不要生成覆盖结论。
3. 输出“分析前输入确认”清单，让用户确认需求/设计来源、测试案例来源、代码范围、提交范围、函数列表、存量报告是否复用。
4. 用户确认前不得进入正式分析；只能继续澄清缺失输入、补充候选清单或询问是否不适用。
5. 用户确认后，按已确认输入读取测试案例文档；如果用户确认没有测试案例，记录该前提后继续。
6. 读取已确认的需求、PRD、用户故事或详细设计，确认业务口径和口径差异。
7. 按测试案例索引读取已确认的相关代码和现有测试文件；代码范围较大或使用小模型时，读取 `references/small-model-reading.md` 并按分批闭环方式执行。
8. 以测试案例编号或需求点为主线做覆盖分析，如果用户确认不用存量报告的情况下，不要用存量报告结论替代源文件分析。
9. 读取 eferences/self-test-checkpoints.md，按自测检查点快照筛出本次适用的补充风险；不要用检查点替代测试案例覆盖结论。
9. 先规划 L1 后端单测门禁候选覆盖，再标注 L1 前端单测建议，然后规划 L2 API/JMeter，最后规划 L3 UI 自动化；M 人工验证仅在用户确认启用时纳入。
10. 如果存在人工验证候选场景，询问用户是否启用人工验证证据包；用户确认后立即创建证据目录和 `summary.md` 模板，用户未确认前不要创建。
11. 为每个测试点标注 L1/L2/L3/M、当前覆盖状态、证据来源、缺口和优先级。
12. 扫描项目已有 `.agents/skills/*/SKILL.md`，按“后续 skill 动态推荐”规则输出后续 skill 路由；不要写死不存在的 skill。
13. 生成中文 Markdown 文件。
14. 对话回复只给路径、范围、关键结论、人工验证目录路径（如已创建）和下一步建议。
## 有测试案例文档时的输出规则

默认生成“一主两附”：

```text
reports/test-strategy/requirement-{scope}-测试策略与覆盖分析.md
reports/test-strategy/appendix-{scope}-测试案例测试方式对照表.md
reports/test-strategy/archive-{scope}-初版策略报告.md   # 仅当已有旧版策略报告需要保留时生成
```

主报告面向项目经理、测试经理和研发负责人，必须聚合策略和结论，不要堆过多执行明细。

附件面向研发和测试执行人员，必须按测试案例编号展开。

如果此前已经生成过未考虑测试案例的旧策略报告，应该：

- 新建主报告整合策略和覆盖分析。
- 将旧报告改名为 `archive-...` 或在主报告中说明旧报告仅作历史参考。
- 避免保留多个平级正式报告造成混淆。

## 没有测试案例文档时的处理规则

本 skill 不提供“无测试案例文档”的正式策略报告模式。

如果用户确认当前确实没有测试案例文档：

- 不生成 `requirement-{scope}-测试策略与覆盖分析.md` 主报告。
- 不生成测试案例方式对照表。
- 只输出待办建议：先补齐测试案例文档，再重新调用本 skill。
- 可以帮助用户整理“测试案例补充清单”，但该清单不是测试策略覆盖分析报告。
## 文件命名规则

- `{scope}` 来自 commit、branch、requirement、task 或用户故事名称。
- 将 `/`、`\`、空格替换为 `-`。
- 中文可以保留。
- 报告内容必须是中文。
- Markdown/Skill 文件使用 UTF-8 无 BOM。

## 主报告输出规则

主报告必须按 `references/output-template.md` 生成，不要在 `SKILL.md` 中另起一套报告结构。

使用规则：

- 有测试案例文档时，使用 `references/output-template.md` 中“一、有测试案例文档时”的主报告模板。
- 没有测试案例文档时，不生成主报告；必须先要求补充或确认正确测试案例文档。
- 报告章节、表格字段、分层口径、TODO 和风险项以 `references/output-template.md` 为准。
- 如果本文件中的约束和模板有冲突，以模板为输出结构，以本文件的门禁和流程规则补充判断口径。
- 主报告必须体现：分析前确认结果、测试案例来源、需求/设计来源、代码范围、存量报告是否复用、L1 后端单测门禁、L1 前端单测建议、L2 API/JMeter、L3 UI 自动化、M 人工验证、TODO、风险和后续 skill 路由。
## 测试案例对照表必须包含

| 案例 | 测试方式 | 工具 | 覆盖逻辑 | 当前状态 | TODO |
|---|---|---|---|---|---|

当前状态使用：`已覆盖`、`部分覆盖`、`缺失`、`不适用`。

## 覆盖判断规则

- 测试案例是主线；不要只按代码函数输出策略。
- API 被调用过不等于测试案例已覆盖，必须有对应断言。
- UI 展示、按钮可见性、页面跳转、标签页切换、返回列表必须归到 L3 或人工验证，不能用 L2 抵扣。
- 数据库副作用、接口响应字段、鉴权、错误码归到 L2。
- 单测优先覆盖可隔离规则、纯函数、状态判断、参数边界、映射和计算逻辑；策略必须说明如何满足增量覆盖率 > 80% 和通过率 100%。
- 现有测试若覆盖行为但文件名/编号不匹配，应标为“部分覆盖，可复用但需映射或重命名”。
- 不要直接给出单测门禁通过/不通过，只能写“需交给 `unit-test-governance` 判断”。
- UI 自动化可被人工验证替代时，必须说明替代边界：P0 主路径和高回归风险权限显隐优先自动化；低风险展示、文案、截图类验收可人工验证，但必须保留证据。

## 后续 skill 动态推荐

后续 skill 建议必须根据当前项目已有的 `.agents/skills/*/SKILL.md` 动态推荐，不要写死固定 skill 名称。

推荐步骤：

1. 先扫描项目内已有 skill：读取 `.agents/skills/*/SKILL.md` 的 `name`、`description` 和用途说明。
2. 按能力匹配后续任务：单测治理、后端单测、API/JMeter、前端测试、UI 自动化、人工验证、最终覆盖报告等。
3. 只推荐项目中真实存在的 skill；如果没有匹配项，写“项目暂未提供对应 skill”，并给出需要补充的能力名称。
4. 如果存在多个候选 skill，按和本次任务的匹配度排序，并说明推荐原因。
5. 不要因为历史报告或旧流程中出现过某个 skill 名称，就默认推荐；必须以当前项目实际存在的 skill 为准。
