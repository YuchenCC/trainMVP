# 报告生成与输出自检

本文件定义模块 E：消费 `analysis-state.md` 中模块 B 的分层策略、模块 C 的自测检查点白盒审查和模块 D 的覆盖结论，生成报告并验证输出完整性。本模块不重新推导推荐层级或覆盖结论。

报告章节和表格字段以 `output-template.md` 为准；本文件定义生成条件、文件命名、职责边界和输出自检。

## 1. 生成前提

只有测试案例来源已经在模块 A 得到确认，且模块 B、模块 C 已完成对应分析时，才能生成正式报告。

没有已确认测试案例文档时：

- 不生成 `requirement-{scope}-测试策略与覆盖分析-v{n}.md` 主报告。
- 不生成测试案例测试方式对照表。
- 只输出测试案例缺失提醒和待办建议：先补齐测试案例文档，再重新调用本 skill。
- 可以帮助整理“测试案例补充清单”，但该清单不是测试策略覆盖分析报告。

## 2. 输出文件与命名

有已确认测试案例时，默认生成“一主一附”：

```text
reports/test-strategy/requirement-{scope}-测试策略与覆盖分析-v{n}.md
reports/test-strategy/appendix-{scope}-测试案例测试方式对照表-v{n}.md
```

- `{scope}` 来自 commit、branch、requirement、task 或用户故事名称。
- 将 `/`、`\`、空格替换为 `-`；中文可保留。
- 已有旧版策略报告时，`v1` 递增为 `v2`、`v3` 等，不覆盖旧版本。
- 报告内容使用中文；Markdown 和 skill/reference 文件使用 UTF-8 无 BOM。

主报告面向项目经理、测试经理和研发负责人，应聚合策略和结论，不堆叠执行明细。附件面向研发和测试执行人员，必须按测试案例编号展开。

## 3. 模板与职责边界

使用 `output-template.md` 中“有测试案例文档时”的主报告和附件模板：

- 模板负责章节、表格字段和审核视图。
- 本文件负责门禁、生成条件、自检与文件规范。
- 模板与本文件冲突时，以模板决定输出结构，以本文件的门禁和流程决定是否允许生成。

主报告必须体现：

- 模块 A 的确认结果、测试案例来源、需求/设计来源、代码范围、存量报告是否复用和口径差异。
- 模块 B 的测试案例分层策略、逻辑代码验证重点、L1 后端单测门禁、L1 前端单测建议、L2 API/JMeter、L3 UI 自动化、M 人工验证。
- 自测检查点映射：模块 C 的原始类别路径、检查点原文、简要代码审查结论、建议层级、案例处理和 TODO；完整原文优先级/测试层次、代码证据和模块 D 的测试资产证据状态放附件。
- 模块 D 的覆盖现状、TODO、人工验证方案和风险。

附件必须体现：

- 逐案例推荐测试层级、分层理由、已有测试资产实际层级、证据、覆盖结论和 TODO。
- 测试资产计划，以及已确认的人工验证替代场景。
- 附录：自测检查点完整审查记录。

后续 skill 建议只在最终对话中输出，不写入主报告、附件或 TODO 表。

## 4. 输出自检

每次生成或覆盖主报告、附件或人工验证 `summary.md` 后，必须立即做输出自检。自检失败时，不得声称完成；应修复后重新读取，直到通过或明确说明阻塞原因。

1. 反读每个生成文件的开头和关键章节；不要只依赖写文件命令返回成功。
2. 确认路径存在、文件大小大于 0、中文内容可按 UTF-8 正常读取。
3. 核对主报告包含：分析范围、分层测试方案、测试案例分层策略与需求-案例覆盖矩阵、自测检查点映射、覆盖现状、TODO、人工验证方案、风险提醒；不要求包含后续 skill 建议。
4. 核对附件主体包含：统计概览、对照表、测试资产计划、人工验证替代场景；附录包含自测检查点完整审查记录。
5. 核对每个案例先有模块 B 的推荐层级和分层理由，再有模块 D 的实际测试层级、证据和覆盖结论。
6. 核对逻辑代码映射和测试代码映射是两个独立部分；不得将测试资产写成逻辑代码，或用测试资产决定策略。
7. 核对报告已写入：测试案例来源、需求/设计来源、代码范围、存量报告复用、自测检查点纳入情况、人工验证启用情况和口径差异。
8. 核对模块 C 已完整遍历共享检查点全部叶子项；主报告的涉及/待确认项保留原始类别路径和检查点文字，使用精简代码结论、建议层级、案例处理和 TODO；附件完整记录保留原文优先级、测试层次、代码证据和测试资产证据，不得概括或改写原文。
9. 核对每个涉及/待确认检查点在附件中同时保留模块 C 的代码证据、案例覆盖判断和补充案例建议，以及模块 D 的测试资产证据字段；不得用“无测试资产”改写为“不涉及”，也不得用已有测试资产决定涉及性或建议层级。
10. 核对未违反用户确认的范围和排除项；例如确认不复用存量报告时，不得引用其结论。
11. 使用飞书来源时，核对已记录来源载体、标题、链接或 token、读取时间和读取状态；仅搜索到而未读取正文的候选不得写成已分析来源。
12. 如创建人工验证 evidence，核对 `summary.md` 和 `screenshots/`、`logs/`、`db/`、`api/` 目录存在；`summary.md` 必须包含基本信息、验证结论、失败/阻塞说明、附件清单和风险说明。

## 5. 输出自检后的状态

自检通过后，最终对话中简短说明已检查的文件及“已完成输出自检”。

报告、附件和 evidence 的飞书发布及“自测策略确认”审批任务，不属于本模块；完成自检后按模块 F 的 `lark-document-source.md` 规则询问用户并执行。
