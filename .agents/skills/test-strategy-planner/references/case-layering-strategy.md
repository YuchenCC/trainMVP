# 测试案例分层策略

本文件定义模块 B：基于已确认的测试案例、需求/设计和逻辑代码，为每个案例或案例组推荐 L1/L2/L3/M 测试层级。模块 B 完成后立即将策略字段写入正式报告目标文件并锁定；工作状态仅用于续跑和记录后续分析结果。

## 1. 固定判断关系

```text
推荐测试层级 = 测试案例语义 + 需求/设计 + 逻辑代码
commit/branch = 变更影响和增量风险信息，不是测试案例覆盖边界
```

测试案例是分析主线。逻辑代码用于验证测试案例的实现边界和分层策略；本 skill 不读取测试代码。

推荐测试层级必须由测试案例、需求/设计和逻辑代码确定；提交范围不得反向决定推荐层级。

## 2. 输入与逻辑代码边界

模块 B 的输入仅包括：

- 已确认的测试案例文档。
- 已确认的需求、PRD、用户故事、详细设计和接口设计。
- 与案例锚点相关的逻辑代码。

逻辑代码是实现产品行为的代码，不是泛指所有源文件：

| 类别 | 典型内容 |
|---|---|
| 后端业务逻辑 | route/controller、service/domain、状态流转、权限判断、异常分支、审计写入 |
| 数据与校验逻辑 | DTO、schema、validator、mapper、常量、权限矩阵 |
| 前端交互逻辑 | 页面/组件的按钮显隐、表单校验、状态、跳转、logic/composable/helper |
| 不属于逻辑代码 | 单测、集成测试、API/JMeter、Playwright/Cypress、fixture、mock、测试工具函数 |

commit、branch 或文件列表只提示本次变更风险。除非用户明确要求“仅分析本次变更”，必须按测试案例锚点扩展到全部相关逻辑代码，不能只看提交中的文件。

如果只获得前端或后端其中一端代码，报告必须明确另一端未纳入；不得补充推测。

## 3. 分层模型与优先级

详细的层级定义、叠加关系和 UI/M 取舍统一读取 `layered-testing.md`。本模块只保留案例分层所需的最小判断：

- 后端纳入范围的可执行逻辑，默认推荐 `L1 后端单测`。
- 案例还验证 HTTP 契约、鉴权链路、错误码、数据库副作用或跨服务流程时，叠加 `L2 API`；L2 不能替代后端 L1。
- 前端根据案例目标和逻辑复杂度选择 `L1 前端单测`、`L3 UI` 或用户确认后的 `M 人工验证`。
- 不直接判定单测门禁通过或不通过，门禁结果交给 `unit-test-governance`。

## 4. 分层选择原则

### 4.1 案例拆解（必须先执行）

每个测试案例必须按以下两步拆解，不得直接对整个案例推一个层级：

**Step 1 — 提取可独立断言的业务规则**
- 从案例的"测试目标"和"预期结果"中提取：状态判断、权限判断、字段校验、异常分支、纯函数调用。
- 按 `layered-testing.md` 的分层规则判断这些逻辑的推荐层级。
- **若案例关联本次范围的后端可执行代码，必须包含 L1 推荐；前端逻辑只在满足前端 L1 条件时建议 L1。**

**Step 2 — 提取接口/页面契约**
- 判断案例中是否还包含：HTTP 响应格式、错误码、数据库副作用、跨页面跳转、按钮显隐。
- 若包含，为对应部分叠加 L2 或 L3。

**输出格式**：同一案例可同时标注 `L1 + L2` 或 `L1 + L3`，不得只保留高层级。

### 4.2 分层选择

模块 B 根据测试案例语义、需求/设计和关联逻辑代码选择推荐层级：

- 案例关联本次纳入范围的后端可执行代码时，**必须推荐 L1 后端单测**（可叠加 L2/L3）。
- 案例主要验证 HTTP 契约、鉴权链路、响应格式、错误码或跨服务链路时，推荐 L2 API。
- 案例主要验证页面交互、跳转、展示或跨页面流程时，推荐 L3 或经用户确认的人工验证。
- 同一案例可叠加多个层级。

### 4.3 批量校验类案例的 L1 拆分

若单个测试案例覆盖多个独立校验项（例如"必填字段不完整时提示"涉及 6 个字段的逐一校验），策略必须：
1. 识别每个校验项为独立的 L1 候选；
2. 在"补充风险"或"TODO"中标注：`建议拆分为 N 个 L1 单测（字段 A/B/C...）`；
3. 案例整体仍可推 L2（验证接口返回），但不得因"已推 L2"而忽略 L1 拆分建议。

## 5. 案例到策略的执行步骤

对每个测试案例或案例组按以下顺序完成闭环：

1. 提取案例编号、测试目标、前置条件、操作、预期结果，以及页面、API、字段、状态、权限和异常锚点。
2. 结合测试案例与需求/设计，先形成验证对象、推荐层级和分层理由。
3. 按案例锚点读取关联逻辑代码：route/controller、service/domain、DTO/schema、状态/权限、页面、组件、logic/composable、shared types 等。
4. 用逻辑代码验证实现边界和策略；逻辑代码发现的测试案例外问题只能标记为“代码发现的补充风险”，不能作为测试案例覆盖结论。
5. 记录案例与逻辑代码映射、建议断言、推荐层级、分层理由、风险和优先级，作为模块 C 自测检查点审查和后续执行的输入。

每轮只处理一个可闭环分析单元。限制的不是文件数，而是是否能完成：

```text
测试案例/需求 -> 分层策略 -> 逻辑代码 -> 自测检查点审查 -> 待补执行项 -> TODO
```

大范围代码或小模型场景下，使用 `small-model-reading.md` 的测试案例索引法；其中索引表仅用于内部读取，不输出到正式报告。

## 6. 人工验证确认门禁

模块 B 完成案例分层后，必须检查是否识别到以下任一候选：

- 页面展示、字段显示、按钮显隐、弹窗、跳转、标签页或跨页面联动；
- 需要真实 SIT 环境、截图、日志、数据库或人工判断的验证；
- 低风险 UI 场景可能由人工验证替代 L3 自动化的情况。

只要存在候选，就必须在进入报告收尾前询问用户：

```markdown
本轮识别到以下 UI/M 人工验证候选：

| 场景 | 当前推荐层级 | 是否可由人工验证承担 |
|---|---|---|
| {场景} | L3 / M | 待确认 |

是否将上述场景纳入人工验证？
```

处理规则：

- 用户明确回复“要、需要、可以、启用、走人工验证”后，才读取 `manual-evidence.md`，并创建 `evidence/{scope}/summary.md` 或用户确认的证据目录。
- 用户明确回复“不需要、不启用”后，记录“人工验证未启用”，保留 L3 或其他已确认层级，不创建 evidence。
- 用户尚未回复时，记录“人工验证待确认”，不默认视为不启用；不读取人工证据规则、不创建 evidence，也不把 M 写成已纳入策略。
- 如果确认本轮完全没有 UI/M 候选，记录“本轮无人工验证候选”，可以不询问。

替代边界：P0 主路径和高回归风险的权限显隐优先自动化；低风险展示、文案、截图类验收可以人工验证，但必须保留证据。

## 7. 模块 B 输出与策略锁定

在进入模块 C 前，每个测试案例必须至少形成：

| 案例 | 测试目标 | 推荐测试层级 | 分层理由 | 内部逻辑代码锚点 | 补充风险 | 优先级 |
|---|---|---|---|---|---|---|

模块 B 必须在本轮正式报告目标路径创建或更新策略部分：分析范围、分层测试方案和测试案例分层策略矩阵。正式报告矩阵只展示可审核的业务信息；`内部逻辑代码锚点` 不输出。矩阵中的 `推荐测试层级`、`分层理由` 在写入后锁定。

模块 C、D 只能补充自测检查点、TODO、风险和人工验证字段；不得静默修改已锁定的策略字段。若出现必须修改策略的事实，标记“策略修订候选”并请求用户确认，随后生成新版本报告。

按 `analysis-state.md` 记录锁定策略和后续补充内容；默认不创建中间文件，只有满足持久化触发条件时才写入文件。模块 B 写入的是策略部分，不得在此阶段声称整份报告已经完成。
