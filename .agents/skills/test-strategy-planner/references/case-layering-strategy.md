# 测试案例分层策略

本文件定义模块 B：基于已确认的测试案例、需求/设计和逻辑代码，为每个案例或案例组推荐 L1/L2/L3/M 测试层级。模块 B 将策略中间结果交给 `analysis-state.md` 定义的工作状态；该状态默认保留在当前对话，只有满足持久化触发条件时才写入文件。完成本模块后，才能读取测试代码并判断已有覆盖。

## 1. 固定判断关系

```text
推荐测试层级 = 测试案例语义 + 需求/设计 + 逻辑代码
覆盖结论 = 推荐测试层级 + 测试代码证据
commit/branch = 变更影响和增量风险信息，不是测试案例覆盖边界
```

测试案例是分析主线。逻辑代码用于验证测试案例的实现边界和分层策略；测试代码只用于后续模块的已有覆盖判断，二者不得混用。

推荐测试层级必须在读取测试代码前确定。已有测试工具、测试文件名、测试数量和提交范围不得反向决定推荐层级。

## 2. 输入与逻辑代码边界

模块 B 的输入仅包括：

- 已确认的测试案例文档。
- 已确认的需求、PRD、用户故事、详细设计和接口设计。
- 与案例锚点相关的逻辑代码。

逻辑代码是实现产品行为的代码，不是泛指所有源文件：

| 类别 | 典型内容 |
|---|---|
| 后端业务逻辑 | route/controller、service/domain、状态流转、权限判断、异常分支、审计写入 |
| 数据与校验逻辑 | DTO、schema、validator、mapper、常量、权限矩阵 |
| 前端交互逻辑 | 页面/组件的按钮显隐、表单校验、状态、跳转、logic/composable/helper |
| 不属于逻辑代码 | 单测、集成测试、API/JMeter、Playwright/Cypress、fixture、mock、测试工具函数 |

commit、branch 或文件列表只提示本次变更风险。除非用户明确要求“仅分析本次变更”，必须按测试案例锚点扩展到全部相关逻辑代码，不能只看提交中的文件。

如果只获得前端或后端其中一端代码，报告必须明确另一端未纳入；不得补充推测。

## 3. 分层模型与优先级

完整的层级适用场景、叠加覆盖和 UI 自动化取舍，读取 `layered-testing.md`。本模块必须区分以下层级：

| 层级 | 名称 | 常见工具 | 覆盖重点 | 门禁关系 |
|---|---|---|---|---|
| L1 | 后端单测 | JUnit5、Jest、Vitest 等 | 业务规则、状态流转、参数校验、异常分支、金额/时间计算、mapper/converter、service 分支 | 后端硬门禁候选项，最终由 `unit-test-governance` 或 unit-gate 判定 |
| L1 | 前端单测建议 | Jest、Vitest、React/Vue 单测工具 | 表单校验、权限显隐、分页排序筛选、金额/日期计算、hook/composable、工具函数、状态管理 | 条件触发建议，不是默认硬门禁 |
| L2 | API/JMeter | JMeter 或团队现有 API 工具 | 接口契约、鉴权、入参校验、关键状态变化、错误码、幂等、跨服务链路 | 不能抵扣后端单测门禁 |
| L3 | UI 自动化 | Playwright、Cypress、React/Vue 测试工具 | P0/P1 用户流程、关键页面跳转、核心按钮、跨页面联动 | 高价值稳定流程，不能抵扣后端单测门禁 |
| M | 人工验证 | SIT、截图、日志、数据库查询、HAR、验证记录 | 低频展示、截图确认、需人工判断的视觉或业务状态 | 不能抵扣后端单测门禁 |

默认优先级：

1. L1 单元测试优先，目标是增量覆盖率大于 80%、通过率 100%。
2. L2 API 自动化验证接口契约、鉴权、权限、参数、状态流转和数据库副作用。
3. L3 UI 自动化只覆盖少量 P0 主路径和高回归风险交互。
4. M 人工验证仅在用户确认启用后纳入，并保留可审计证据。

L2、L3 和 M 都不能抵扣 L1 单测门禁。不要直接判定单测门禁通过或不通过，只能标记“需交给 `unit-test-governance` 判断”。

即使需求天然偏接口或页面，也必须先识别可单测的增量逻辑，例如分页计算、字段聚合、映射、状态判断、权限按钮矩阵、异常分支和纯函数。前端核心逻辑写在组件内时，策略应建议抽出 `*.logic.ts`、composable 或 helper 以便单测。

## 4. 案例到策略的执行步骤

对每个测试案例或案例组按以下顺序完成闭环：

1. 提取案例编号、测试目标、前置条件、操作、预期结果，以及页面、API、字段、状态、权限和异常锚点。
2. 结合测试案例与需求/设计，先形成验证对象、推荐层级和分层理由；此时不得读取测试代码作为依据。
3. 按案例锚点读取关联逻辑代码：route/controller、service/domain、DTO/schema、状态/权限、页面、组件、logic/composable、shared types 等。
4. 用逻辑代码验证实现边界和策略；逻辑代码发现的测试案例外问题只能标记为“代码发现的补充风险”，不能作为测试案例覆盖结论。
5. 记录案例与逻辑代码映射、建议断言、推荐层级、分层理由、风险和优先级，作为模块 C 自测检查点审查和模块 D 覆盖判断的输入。

每轮只处理一个可闭环分析单元。限制的不是文件数，而是是否能完成：

```text
测试案例/需求 -> 分层策略 -> 逻辑代码 -> 自测检查点审查 -> 测试代码覆盖 -> 覆盖结论 -> TODO
```

大范围代码或小模型场景下，使用 `small-model-reading.md` 的测试案例索引法；其中索引表仅用于内部读取，不输出到正式报告。

## 5. 人工验证候选

当策略中存在可由人工验证替代的 UI 场景时，先向用户询问是否启用人工验证证据包。用户明确回复“要”“需要”“可以”“启用”或“走人工验证”后，才读取 `manual-evidence.md` 并创建 `evidence/{scope}/{case-id}/` 与 `summary.md` 指引。

未确认前不得创建证据目录。

替代边界：P0 主路径和高回归风险的权限显隐优先自动化；低风险展示、文案、截图类验收可以人工验证，但必须保留证据。

## 6. 模块 B 输出要求

在进入模块 C 前，每个测试案例必须至少形成：

| 案例 | 测试目标 | 推荐测试层级 | 分层理由 | 逻辑代码验证重点 | 补充风险 | 优先级 |
|---|---|---|---|---|---|---|

这是一份策略中间结论，不得填入“已有测试资产实际层级”或“覆盖结论”。这些字段只能由模块 D 在实际读取测试资产后填写。

按 `analysis-state.md` 将以上中间结论交给工作状态，供模块 C 补充和模块 D 最终渲染；默认不创建中间文件，只有满足持久化触发条件时才写入文件。不得直接生成正式主报告或附件。
