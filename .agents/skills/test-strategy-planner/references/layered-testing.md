# 分层测试判断规则

## 分层判断规则

推荐测试层级必须根据测试案例的验证目标、业务语义和逻辑代码确定，不能根据已有测试文件、测试工具、测试数量或提交范围反推。已有测试资产只用于记录实际测试层级和判断覆盖结果；例如 `Vitest + Supertest` 只能说明已有测试包含接口/集成证据，不能自动把所有相关测试案例推荐为 L2。

| 层级 | 适用内容 | 说明 |
|---|---|---|
| L1 后端单测 | 业务规则、状态流转、参数校验、异常分支、金额/时间计算、mapper/converter、service 分支 | 后端硬门禁候选项，最终由 unit-gate 判定 |
| L1 前端单测建议 | 表单校验、权限显隐、分页排序筛选、金额/日期计算、hook/composable、工具函数、状态管理 | 仅条件触发建议，不是默认硬门禁 |
| L2 API/JMeter | 接口契约、鉴权、入参校验、关键状态变化、错误码、幂等、跨服务链路 | 适合接口自动化验证 |
| L3 UI 自动化 | P0/P1 用户流程、关键页面跳转、核心按钮、跨页面联动 | 只覆盖高价值稳定流程，避免大量慢速 UI 用例 |
| M 人工验证 | 低频 UI 展示、截图确认、SIT 环境验证、日志证据、需人工判断的视觉或业务状态 | 不能抵扣后端单测门禁 |

## 优先级总则

测试策略默认按 L1 → L2 → L3/M 排序：先满足单元测试增量覆盖率 > 80% 且通过率 100%，再补 API 自动化，最后选择少量 UI 自动化或人工验证。API、UI 自动化和人工验证不能抵扣单测覆盖率。

## 后端 L1 门禁

后端 L1 单测是默认硬门禁，必须在策略报告中单独展示。L2、L3 和 M 均不能抵扣后端 L1 的增量覆盖率或通过率要求。

没有覆盖率报告或单测执行结果时，只能写“阻塞/待补”，不得写“门禁通过”。最终门禁结果仍由 `unit-test-governance` 判定。

## 前端单测建议

前端不默认要求单测覆盖率门禁。仅在存在可隔离、稳定的逻辑时建议补充前端单测：

- 表单校验、字段格式化。
- 权限显隐、按钮状态、分页排序筛选。
- 金额、日期、数量计算。
- React hook、Vue composable、工具函数和状态管理。

纯展示、简单表格、普通弹窗和文案展示，优先使用 L3 UI 自动化或 M 人工验证，而不是为了覆盖率机械补前端单测。

## L1 单元测试

优先放入 L1 的场景：

| 场景 | 原因 |
|---|---|
| 业务规则、状态判断、权限判断 | 可用输入输出和 mock 隔离验证 |
| 算法、编号生成、排序、去重、分页计算 | 结果确定，适合白盒单测 |
| 参数校验、边界值、异常分支 | 分支明确，单测反馈最快 |
| DTO 转换、字段派生、枚举映射 | 数据转换逻辑明确 |
| 前端按钮权限矩阵、状态动作映射 | 可抽 pure function，稳定且能拉升增量覆盖率 |
| retry、fallback、idempotency | 风险高，需覆盖成功和耗尽分支 |

如果代码都写在组件或 controller 中，策略中应建议抽出 helper、logic、mapper、validator、policy 等文件，先让 L1 可测。

## L2 API 自动化

优先放入 L2 的场景：

| 场景 | 原因 |
|---|---|
| HTTP 请求/响应契约 | 需要验证接口输入输出 |
| 鉴权、token、角色访问 | 需要经过 API 层验证 |
| 状态流转业务流程 | 需要验证数据库副作用和流程串联 |
| 多表写入、事务、副作用 | 单测不足以证明集成正确 |
| 接口参数组合和错误码 | 适合 JMeter 场景化覆盖 |

## L3 前端 / E2E

优先放入 L3 的场景：

| 场景 | 原因 |
|---|---|
| P0 主路径 | 必须证明用户可走通 |
| 页面 loading/empty/error/success | 用户可见状态 |
| 表单交互、按钮禁用、权限展示 | 需要渲染后验证 |
| 路由跳转、弹窗确认、列表筛选 | 前端交互行为 |
| 核心用户旅程 | 前后端联动体验 |

## UI 自动化和人工验证取舍

| 场景 | 建议 |
|---|---|
| P0 主路径、权限显隐、容易回归的按钮矩阵 | 优先 UI 自动化或组件测试 |
| 字段展示、文案、低风险弹窗、视觉类确认 | 可人工验证替代 |
| 复杂端到端流程、环境准备很重的场景 | 优先 API 自动化 + 人工验证 |
| 人工验证替代 UI 自动化 | 必须提供 SIT 环境、操作人、时间、截图、服务器日志、数据库变化或接口响应证据 |

## 叠加覆盖

同一需求同时影响规则、接口和页面时，允许 L1/L2/L3/M 叠加覆盖。策略报告中必须说明每一层覆盖的不同风险，避免重复测试同一个断言。
