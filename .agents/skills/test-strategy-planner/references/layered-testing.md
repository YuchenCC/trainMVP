# 分层测试判断规则

## 优先级总则

测试策略默认按 L1 → L2 → L3/M 排序：先满足单元测试增量覆盖率 > 80% 且通过率 100%，再补 API 自动化，最后选择少量 UI 自动化或人工验证。API、UI 自动化和人工验证不能抵扣单测覆盖率。

## L1 单元测试

优先放入 L1 的场景：

| 场景 | 原因 |
|---|---|
| 业务规则、状态判断、权限判断 | 可用输入输出和 mock 隔离验证 |
| 算法、编号生成、排序、去重、分页计算 | 结果确定，适合白盒单测 |
| 参数校验、边界值、异常分支 | 分支明确，单测反馈最快 |
| DTO 转换、字段派生、枚举映射 | 数据转换逻辑明确 |
| 前端按钮权限矩阵、状态动作映射 | 可抽 pure function，稳定且能拉升增量覆盖率 |
| retry、fallback、idempotency | 风险高，需覆盖成功和耗尽分支 |

如果代码都写在组件或 controller 中，策略中应建议抽出 helper、logic、mapper、validator、policy 等文件，先让 L1 可测。

## L2 API 自动化

优先放入 L2 的场景：

| 场景 | 原因 |
|---|---|
| HTTP 请求/响应契约 | 需要验证接口输入输出 |
| 鉴权、token、角色访问 | 需要经过 API 层验证 |
| 状态流转业务流程 | 需要验证数据库副作用和流程串联 |
| 多表写入、事务、副作用 | 单测不足以证明集成正确 |
| 接口参数组合和错误码 | 适合 JMeter 场景化覆盖 |

## L3 前端 / E2E

优先放入 L3 的场景：

| 场景 | 原因 |
|---|---|
| P0 主路径 | 必须证明用户可走通 |
| 页面 loading/empty/error/success | 用户可见状态 |
| 表单交互、按钮禁用、权限展示 | 需要渲染后验证 |
| 路由跳转、弹窗确认、列表筛选 | 前端交互行为 |
| 核心用户旅程 | 前后端联动体验 |

## UI 自动化和人工验证取舍

| 场景 | 建议 |
|---|---|
| P0 主路径、权限显隐、容易回归的按钮矩阵 | 优先 UI 自动化或组件测试 |
| 字段展示、文案、低风险弹窗、视觉类确认 | 可人工验证替代 |
| 复杂端到端流程、环境准备很重的场景 | 优先 API 自动化 + 人工验证 |
| 人工验证替代 UI 自动化 | 必须提供 SIT 环境、操作人、时间、截图、服务器日志、数据库变化或接口响应证据 |

## 叠加覆盖

同一需求同时影响规则、接口和页面时，允许 L1/L2/L3/M 叠加覆盖。策略报告中必须说明每一层覆盖的不同风险，避免重复测试同一个断言。
