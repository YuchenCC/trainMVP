# 小模型文件读取规则

当使用小模型、上下文较小，或代码范围较大时，必须使用“测试案例索引法”读取文件。目标是先围绕已确认测试案例建立证据链，再读取代码；不要从大量代码中直接归纳结论。

## 0. 触发定义

### 小模型定义

满足任一条件时，按小模型规则执行：

- 参数规模约 30B 以下，例如 25B 级别模型。
- 有效上下文约 32k 以下，或当前对话/仓库内容已经接近上下文上限。
- 实际表现为容易漏读文件、混淆旧报告和源文件、只看文件名下结论、读完整文件后丢失关键细节。

这不是严格硬件指标，而是风险口径。只要出现以上风险，就用本规则降复杂度。

### 代码范围较大定义

满足任一条件时，视为代码范围较大：

- 跨前端和后端两个仓库或两个应用目录。
- 相关文件超过 10 个。
- 相关代码超过 1500 行。
- 涉及超过 5 个接口。
- 涉及超过 3 个页面、角色、状态流转或权限矩阵。
- 测试案例超过 10 条。
- 需求涉及资金、权限、状态流转、数据一致性或跨系统副作用。

## 1. 第一优先级：读取已确认测试案例

分析开始前，优先确认测试案例来源：

1. 用户明确指定的测试案例文档。
2. 用户确认适用于本次范围的候选测试案例文档。
3. 从需求目录中找到，并经用户确认适用的测试案例文档。

如果没有确认测试案例来源，不要直接进入大范围代码分析。必须先询问用户测试案例在哪里；没有已确认测试案例时停止分析，不生成策略报告。

## 2. 用测试案例索引代码

代码多时，必须从测试案例出发定位代码，而不是先扫代码。

读取顺序：

```text
测试案例 -> 测试步骤/预期结果 -> 页面/API/字段/状态/权限锚点 -> 候选代码 -> 代码片段验证 -> 分层策略与执行准备
```

每条测试案例至少提取：

- 用例编号
- 测试标题
- 前置条件
- 操作步骤
- 预期结果
- 页面路由
- API 路径
- 字段名
- 状态值
- 权限角色
- 异常/边界条件

再用这些锚点定位代码：

- 页面路由 -> 前端 page/router/component
- API 路径 -> 后端 route/controller
- 字段名 -> DTO/schema/entity/service/frontend columns
- 状态值 -> 状态流转逻辑和按钮展示逻辑
- 权限角色 -> RBAC/权限中间件/前端按钮可见性
- 异常/边界 -> 参数校验、错误码、异常分支

### 多 commit 增量

当输入是多个 commit 时，先把它们合并为一个 `base..head` 变更范围，再按最终净差异读取：

1. 用 `git log --oneline base..head` 了解提交顺序和意图。
2. 用 `git diff --stat base..head` 和 `git diff --unified=20 base..head` 定位累计变化。
3. 读取最终代码中变更所在的完整函数/方法、直接调用链、相关类型/配置和入口。
4. 同一函数被多个 commit 重复修改时，只保留最终代码和累计差异的一个分析结论。
5. 只有回退、重命名、删除后新增、临时实现影响最终行为，或用户要求审查提交过程时，才读取具体 commit。

commit 历史用于解释变更意图和风险，不用于扩大测试案例范围，也不能替代测试案例到代码的索引。

## 3. 先做测试案例到代码索引表

进入正式分析前，必须先在内部建立索引表。该表是 AI 的读取工作表，不属于最终报告，也不要求原样输出给用户：

| 测试案例 | 测试意图 | 锚点 | 候选代码 | 相关性 | 是否已读取 | 下一步 |
|---|---|---|---|---|---|---|

相关性取值：

- 强相关：命中页面路由、API 路径、核心方法、状态流转、权限判断或测试断言。
- 弱相关：只命中字段、类型、工具函数或间接依赖。
- 待确认：文件名或目录像相关，但尚未读取片段验证。
- 排除：读取片段后确认无关。

只有“强相关”文件，或用户明确指定的文件，才能进入精读。弱相关文件只在强相关文件读完后发现依赖时再读。

## 4. 每轮只处理一个可闭环分析单元

不要机械按“一个用户故事”或“固定文件数”读取。每轮只处理一个可闭环分析单元。

可闭环分析单元可以是：

- 一个用户故事
- 一个页面主流程
- 一个 API + service 调用链
- 一个状态流转
- 一个权限矩阵
- 一组强耦合的前后端流程

判断标准：

- 能从测试案例追到需求/设计
- 能从测试案例追到代码
- 能形成可执行的分层策略与待办

限制的不是文件数，而是每轮能否闭环：

```text
测试案例/需求 -> 代码 -> 分层策略 -> 自测检查点审查 -> 执行准备 -> TODO
```

## 5. 按复杂度决定读取方式

| 复杂度 | 适用场景 | 读取方式 |
|---|---|---|
| S | 单接口、单函数、单页面小交互 | 可一轮读完 |
| M | 一个页面 + 少量接口 + service | 分前端、后端、测试三批读 |
| L | 多页面、多状态、多角色、多接口 | 先拆流程，再逐流程读 |
| XL | 跨仓库、跨模块、资金/权限/状态复杂 | 先在内部建立索引和分批计划，不直接分析 |

如果读完一轮后事实超过 15 条，必须先总结再继续。
如果出现超过 3 个未确认点，必须先输出待确认清单。

## 6. 测试代码边界

本 skill 不检索、读取或判断既有测试文件，也不根据测试工具、文件名、数量或执行结果推导测试策略。测试实现与真实执行证据分别由 `unit-test-governance` 和 `test-report-generator` 处理。

## 7. 前后端分离项目要分端读取

后端至少读取：

- route/controller
- service
- DTO/schema/entity
- 权限/鉴权
- 状态流转
- 异常分支
- 已有单测/API 测试

前端至少读取：

- 页面入口
- 路由
- 组件交互
- 接口调用
- 表格/字段展示
- 状态按钮
- 权限可见性
- 已有组件测试/UI 自动化

如果只拿到一端代码，报告必须写明另一端未纳入，不允许补脑。

## 8. 禁止从代码反推覆盖

代码很多时，不允许先扫代码再随意归纳测试点。必须先用测试案例建立索引，再读取对应代码。

只有以下情况可以从代码补充测试点：

- 代码中发现测试案例未覆盖的异常分支
- 代码中发现测试案例未覆盖的权限/状态流转
- 代码中发现详细设计未提到但实际存在的行为
- 用户明确要求做代码影响分析

补充出来的测试点必须标记为“代码发现的补充风险”，不能伪装成测试案例已覆盖内容。

## 10. 输出前自检

生成报告前必须确认：

- 是否读取了已确认测试案例。
- 是否读取了需求/设计源文件。
- 是否用测试案例索引了代码。
- 是否读取了前后端强相关代码。
- 是否明确未读取既有测试文件，且未将其作为策略或覆盖结论依据。
- 是否明确了未读范围和待确认项。
- 是否避免用旧报告结论替代源文件分析。
