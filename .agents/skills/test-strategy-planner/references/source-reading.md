# 来源读取与范围确认

本文件定义测试策略分析的模块 A：在任何正式分析前，确认输入范围、搜索候选来源并处理测试案例缺失。模块 A 只读取候选元数据和用户明确提供的范围信息，不读取测试案例、需求、逻辑代码或测试资产正文进行分析。完成本模块并取得用户明确确认后，才能进入分层策略。

## 1. 硬性门禁：先确认，再分析

正式分析前必须先让用户确认输入范围。未确认前，只允许做文件搜索、候选清单和问题澄清；不得生成覆盖结论、代码结论、TODO 结论或正式报告。

即使用户已经在请求中提供了分析范围、测试案例来源、需求/设计来源、代码范围、口径差异或类似表格，也只能视为“待确认输入草案”，不能自动视为已确认。

只有当用户明确表达“确认”“按这个范围分析”“可以开始”“纳入以上范围”“继续生成报告”等确认意图后，才能进入正式分析和写报告。

如果用户只说“用 test-strategy-planner 分析一下”，下一步必须输出确认清单；不得先读取大量代码并产出结论或报告。

## 2. 必须确认的输入

| 输入 | 说明 | 示例 |
|---|---|---|
| `模块名` | Task 编号、需求编号或功能模块 | `T1-需求池`、`T4-变更率`、`紧急变更` |
| `需求/设计来源` | 本地或飞书 MCP 中的 PRD、用户故事、详细设计、接口设计等 | `用户故事 v2.0`、飞书文档/Wiki 链接 |
| `测试案例来源` | 已确认适用于本次范围的本地或飞书测试案例文档 | `RT-T2-US2.3-测试案例_v1.1.md`、飞书文档链接 |
| `文件路径` | 需要分析的前端/后端代码文件或目录 | `service.ts`、`index.tsx`、`routes/` |
| `提交范围` | Git 提交、分支或范围 | `HEAD~5`、`feature/xxx`、`abc123..def456` |
| `函数列表` | 需要重点分析或补单测的具体函数 | `sanitizeDescription`、`generateReqCode` |
| `存量报告` | 已有策略报告、覆盖报告、handoff、历史分析 | `reports/test-strategy/*.md` |

确认规则：

- 用户已经明确提供的输入直接列入确认清单，不重复追问。
- 关键输入缺失时，输出待确认输入清单，请用户补充或确认“不适用”。
- 找到候选测试案例、需求文档、代码文件或存量报告时，必须先列给用户确认是否纳入本次分析。
- 测试案例是必需输入。用户未确认测试案例来源前，不进入大范围代码分析；如果确认没有测试案例，停止分析并要求先补测试案例，不生成策略报告。
- 存量报告必须询问是否直接复用。用户未确认复用时，旧报告只能作索引；用户明确不复用时，不引用其结论或内容，必须重新分析。

## 4. 候选来源搜索与后续读取路由

模块 A 只确认下列来源是否纳入、版本和用途。实际正文读取由后续模块按职责完成：模块 B 读取测试案例、需求/设计和逻辑代码；模块 C 读取共享检查点；模块 D 读取测试资产。

后续模块的读取路由如下：

1. 测试案例文档：模块 B 优先读取。
2. 需求、PRD、用户故事、详细设计和接口设计：由模块 B 读取以确认业务口径。
3. 逻辑代码：由模块 B 按案例锚点读取 service、route/controller、frontend page/component/service、shared types 等产品实现代码。
4. 现有测试文件：由模块 D 读取，只作为已有覆盖证据，不能替代测试案例覆盖分析。
5. 自测检查点：由模块 C 完整遍历 `self-test-checkpoints.md`，以逻辑代码白盒分析补充案例建议；不能替代测试案例覆盖结论。

如果测试案例文档与用户口头描述不一致，报告必须记录“口径差异”。例如用户说“版本火车列表/详情”，但测试案例写的是“班次列表/详情”，先按测试案例编号分析，并提示需要统一命名。

## 5. 飞书来源路由

用户提供飞书链接、关键词、文件夹名称，或明确要求查找飞书文件时，必须先读取 `lark-document-source.md` 并使用飞书 MCP 搜索飞书候选。

- 用户未允许前，不搜索本地目录、本地同名文件或本地内容。
- 飞书搜索结果只是候选来源；确认前只能读取标题、链接、类型、更新时间等元数据，不得据此生成覆盖结论。
- 用户确认后，模块 B、模块 C 或模块 D 才按职责读取已确认飞书文档正文；报告中必须记录来源载体、标题、链接或 token、读取时间和读取状态。
- 用户仅提供飞书文件夹名称时，按 `lark-document-source.md` 的文件夹搜索与能力降级规则处理；没有文件夹检索能力时，不得以本地文件夹代替。

## 6. 找不到测试案例时

如果用户没有直接给出测试案例文档路径，且在常见目录没有找到匹配文件，不生成策略报告。必须询问用户测试案例文档位置。

```markdown
我还没有找到本次范围对应的测试案例文档。测试案例是本次策略分析的必需输入，请告诉我测试案例文档在哪里。

我已检查：
- {已检查目录1}
- {已检查目录2}

候选文件：
- {候选文件1}
- {候选文件2}

请提供测试案例文档路径，或确认以上候选文件中哪一个适用于本次范围。确认后我再开始分析。
```

常见本地搜索位置：

- `03-需求与设计/**/测试案例/*.md`
- `03-需求与设计/**/测试用例/*.md`
- `docs/**/测试案例*.md`
- `docs/**/test-case*.md`
- 当前 task/requirement 目录下的 `测试案例`、`测试用例`、`cases`、`test-cases` 子目录。

没有已确认测试案例文档时，只能输出待确认清单或候选文件清单；不得进入正式分析，也不得生成主报告或测试方式对照表。

## 7. 大范围与小模型读取

当代码范围较大、模型上下文较小，或用户明确要求使用小模型时，读取 `small-model-reading.md` 并执行测试案例索引法。

触发条件：

- 小模型：参数规模约 30B 以下、有效上下文约 32k 以下，或实际表现为容易漏读文件、混淆旧报告和源文件、无法稳定处理长代码。
- 代码范围较大：跨前后端仓库、超过 10 个相关文件、超过 1500 行相关代码、超过 5 个接口、超过 3 个页面/角色/状态流转，或测试案例超过 10 条。

小模型索引表是内部读取工作表，不输出到报告，除非用户主动要求查看分析过程。
