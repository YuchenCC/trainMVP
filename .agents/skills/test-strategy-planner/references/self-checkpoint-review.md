# 自测检查点白盒审查与案例补充

本文件定义模块 C：在模块 B 已完成测试案例分层策略后，以逻辑代码为白盒证据，完整审查 `self-test-checkpoints.md` 的叶子检查点，识别代码涉及项、核对已有测试案例覆盖，并生成补充案例建议。

模块 C 不读取测试代码；检查点是否涉及只由逻辑代码白盒证据决定。

## 1. 输入与边界

模块 C 输入：

- 模块 B 的已确认测试案例分层策略和逻辑代码映射。
- 本次相关逻辑代码、需求/设计和代码事实。
- `self-test-checkpoints.md` 的完整检查点目录及其原文测试层次。

本模块是自测检查点的白盒补充审查，不替代测试案例主线：

- 已确认测试案例仍由模块 B 制定分层策略。
- 模块 C 只产生“涉及检查点映射”和“补充案例建议”，不得伪装为用户已提供的测试案例。
- 模块 C 的“是否涉及”由代码证据决定。

## 2. 完整遍历与原文保留

必须完整遍历 `self-test-checkpoints.md` 中每个表格的叶子检查点，而不是只从测试案例或代码摘要中挑选主题。该文件在模块 C 只读取一次，完成遍历后形成索引和判定记录，后续模块 D/E 直接复用记录，不重复读取原文。

“完整遍历”是完整性要求，不是要求对每个叶子项都执行一次独立的深度代码阅读、案例核对和补充案例设计。按以下两阶段执行。

### C0：结构化预筛与范围证据

1. 从模块 B 已确认的逻辑代码锚点整理**行为清单**，例如：HTTP 调用、页面交互、权限、批量、状态流转、事务、乐观锁、审计、文件、配置、定时任务、外部调用。
2. 整理**范围排除清单**：本轮已确认未出现的行为域及其代码范围证据。每个排除项分配稳定证据编号，例如 `C-SCOPE-01`。
3. 一次性解析全部叶子项，保留原始类别路径、原文、优先级和测试层次，形成完整索引。
4. 按类别和关键词将叶子项路由为：
   - `候选深审`：与行为清单相交，或代码配置/实现边界不足以判定。
   - `批量不涉及`：与行为清单没有交集，且命中已确认范围排除项。
5. `批量不涉及`的每个叶子仍要写入完整遍历记录；代码证据和判定理由可以引用同一个范围证据编号，例如“`C-SCOPE-01`：本轮逻辑范围未发现文件上传、下载、存储或 IO 调用”。

不得仅凭检查点优先级或原文测试层次将叶子项排除。无法由范围证据排除的项必须进入候选深审。

### C1：候选深审与风险簇

对候选深审项按风险簇分批处理，而非逐条重复读取同一段代码。推荐风险簇：校验、权限、状态/版本、事务/审计、接口契约、UI、批量、外部调用/异步。

每个候选叶子仍必须有独立的代码证据、涉及判定和判定理由；同一风险簇可复用已读取的代码事实。先为风险簇核对一次既有测试案例，再映射到簇内叶子项。只有不同的测试目标、前置条件、步骤或预期结果时，才新增补充案例建议；否则多个叶子引用同一建议编号。

范围较大、候选项较多或上下文压力较高时，以风险簇为最小可闭环单元。每完成一个风险簇，立即更新工作状态，记录已审查候选项、复用的范围证据、案例映射和补充建议；未完成的候选簇不得标记完成。

每个叶子检查点至少记录：

| 字段 | 要求 |
|---|---|
| 检查点类别路径 | 保留原始标题层级，例如 `一、交互类 / 1. 接口` |
| 检查点原文 | 原样保留检查点列文字，不得概括、缩写或改写 |
| 原文优先级 | 保留原文 P0/P1/P2 |
| 原文测试层次 | 保留原文 L1/L2/L3/M/Perf 或组合 |
| 代码证据 | 文件、函数、调用链、配置或未发现对应实现的事实 |
| 涉及判定 | 涉及 / 不涉及 / 待确认 |
| 判定理由 | 基于逻辑代码的白盒理由 |

“完整遍历”要求所有叶子检查点都有判定记录。主报告不需要罗列不涉及项，但完整遍历记录必须保存在 `analysis-state.md` 的持久化状态或附件中。批量不涉及记录必须能追溯到范围证据编号，候选深审记录必须能追溯到具体代码事实。

## 3. 白盒涉及判定

先从模块 B 的逻辑代码事实识别行为，再逐项匹配检查点：

| 代码行为 | 典型检查点类别 |
|---|---|
| HTTP 请求、接口封装、重试、超时、第三方调用 | 一、交互类 / 接口 |
| 页面按钮、表单、跳转、错误提示 | 一、交互类 / 交互、九、界面类 |
| 角色判断、资源归属校验 | 二、管理类功能 / 权限控制，十、安全 / 访问控制 |
| 批量接口、循环处理 | 二、管理类功能 / 批量操作，五、复杂逻辑 |
| 状态更新、事务、乐观锁、异步副作用 | 二、管理类功能 / 数据操作，三、交易类功能 / 状态流转或数据一致性 |
| 审计日志 | 二、管理类功能 / 审计 |
| 文件、数据库、配置、定时任务 | 对应原文分类 |

必须以实际代码为准。例如前端 `services/*.ts` 发生 HTTP 调用时，需要逐项审查接口类检查点中的超时、报文格式、防重、限流、正常/边界/错误/重试机制；代码未配置或配置在全局客户端时，应记录为“待确认”并定位全局配置，而不是直接判为不涉及。

## 4. 测试案例覆盖与补充建议

对每个“涉及”或“待确认”的检查点，核对模块 B 的既有测试案例是否已覆盖检查点的验证目标、前置条件、操作和预期结果。同一风险簇先完成一次案例索引和覆盖核对，再映射到簇内检查点；不得为相同缺口反复生成语义重复的补充案例建议。

| 案例覆盖判断 | 含义 | 后续动作 |
|---|---|---|
| 已覆盖 | 既有案例的验证目标和预期结果足以验证检查点 | 建立案例映射 |
| 部分覆盖 | 案例相关但缺少边界、异常或关键断言 | 补充或拆分既有案例 |
| 未覆盖 | 没有可映射案例 | 生成补充案例建议 |
| 待确认 | 案例正文、代码实现或需求口径不足 | 记录待确认项 |

检查点原文中的测试层次是默认验证建议。模块 C 必须结合代码和模块 B 的策略，记录是否沿用、拆分或叠加层级及理由；不得机械照抄。

补充案例建议至少包含：

| 检查点类别路径 | 检查点原文 | 原文测试层次 | 代码证据 | 案例处理 | 补充案例建议 | 建议层级/理由 | 优先级 |
|---|---|---|---|---|---|---|---|

补充案例建议集中转入报告的“待补充案例列表”，不计入用户原始测试案例数量；未经用户确认，不得伪造新的正式案例编号。

## 5. 待补充案例回写确认

生成“待补充案例列表”后，必须向用户询问：

```markdown
本轮发现以下待补充测试案例，是否将其加入正式测试案例文档？

| 建议编号 | 关联检查点 | 测试目标 | 建议层级 |
|---|---|---|---|
| 建议-CP-001 | {检查点} | {测试目标} | {L1/L2/L3/M} |

请选择：回写正式测试案例文档 / 仅保留在本次分析报告 / 暂不决定。
```

处理规则：

- 用户确认回写后，先确认目标文档、写入位置和正式编号规则，再修改本地或飞书测试案例文档。
- 用户只确认“保留在报告”时，建议编号保持临时编号，不计入原始案例数量，不修改源文档。
- 用户暂不决定或未回复时，标记“待确认”，不分配正式编号，不修改源文档。
- 飞书测试案例文档的回写必须遵守 `lark-document-source.md` 的写入权限和最终确认规则；没有写入能力时，只输出可回写内容，不得声称已更新。

## 6. 输出与交接

模块 C 输出：

- 全量检查点遍历记录：涉及、不涉及、待确认及代码判定依据；批量不涉及项保留范围证据编号，候选深审项保留具体代码证据。
- 涉及/待确认检查点的原文映射、原文测试层次、案例覆盖判断和补充案例建议。
- 自测检查点补充风险和 TODO，以及检查点到风险簇、风险簇到补充案例建议的映射。

将结果交给 `analysis-state.md`：默认保留在当前对话，满足持久化条件时写入工作状态文件。模块 C 完成后立即追加主报告第 4 节和附件附录；模块 D 做最终结构校验。
