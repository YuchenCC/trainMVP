# 测试策略与覆盖分析输出模板

## 零、分析前确认清单

正式分析和写报告前，先输出以下确认清单。默认由研发负责人、提测负责人或技术 owner 确认；测试人员不负责猜测前后端分支、commit 或代码影响范围。用户确认后，才进入报告生成。

```markdown
## 待确认信息清单

### 0. 执行人与输入清单
| 项目 | 内容 |
|---|---|
| planner 执行人 | 研发负责人 / 提测负责人 / 技术 owner / 其他 |

| 后端影响清单 | 已提供 / 待补充 / 用户确认不涉及后端 |
| 前端影响清单 | 已提供 / 待补充 / 用户确认不涉及前端 |
| 测试执行清单 | 已提供 / 待补充 |
| 既有报告清单 | 已提供 / 待补充 / 用户确认暂无 |
```

## 一、有测试案例文档时

### 主报告：`requirement-{scope}-测试策略与覆盖分析.md`

```markdown
# {scope} 测试策略与覆盖分析报告

## 1. 分析范围

| 项目 | 内容 |
|---|---|
| 分析范围 |  |
| 测试案例来源 |  |
| 需求/设计来源 |  |
| 代码范围 |  |
| planner 执行人 | 研发负责人 / 提测负责人 / 技术 owner |
| 输入清单完整性 | 完整 / 待补充，说明缺项 |
| 口径差异 | 无 / 有，说明差异 |

## 2. 分层测试方案

| 层级 | 结论 | 覆盖重点 | 后续执行 |
|---|---|---|---|
| L1 单元测试 | 本次增量代码需满足增量覆盖率 > 80%、通过率 100%；前后端分别执行和判定；L2/API、L3/UI、人工验证不能抵扣该门禁，最终由 `unit-test-governance` 判定 | 纯函数、字段聚合、分页、状态判断、权限矩阵、异常分支、可 mock 逻辑 | `unit-test-governance` |
| L2 API 自动化 |  | 接口契约、鉴权、参数校验、状态流转、数据库副作用 |  |
| L3 UI 自动化 |  | P0 主路径、高回归风险交互、权限显隐 |  |
| M 人工验证 |  | 低风险 UI 展示、文案、截图类验收、复杂 SIT 环境链路 | `evidence/{scope}/summary.md` |

## 3. 覆盖现状

| 范围 | 案例数 | 已覆盖 | 部分覆盖 | 缺失 | 可复用测试资产 | 主要缺口 |
|---|---:|---:|---:|---:|---|---|

## 4. 需求-案例覆盖矩阵

| 需求点/案例组 | 测试案例 | 推荐层级 | 当前覆盖判断 | 待补内容 | 优先级 |
|---|---|---|---|---|---|

## 5. 代码影响测试矩阵

| 文件/模块 | 影响逻辑 | 影响案例 | 推荐层级 | 必须覆盖原因 | 后续 skill |
|---|---|---|---|---|---|

## 6. 现有测试资产判断

| 现有测试文件 | 可复用覆盖 | 不足 | 处理建议 |
|---|---|---|---|

## 7. TODO 清单与后续执行

| 优先级 | TODO | 后续 Skill | 输入 | 预期输出 |
|---|---|---|---|---|
| P0 | 对本次增量代码执行单测门禁检查，确认增量覆盖率 > 80%、通过率 100% | `unit-test-governance` | 本报告 + commit/文件范围 | 单测缺口、执行结果、增量覆盖率判断 |

## 8. 人工验证方案

| UI 场景 | 替代方式 | 是否询问用户 | evidence/summary.md 路径 | 必须证据 | 风险 |
|---|---|---|---|---|---|

## 9. 风险提醒

| 风险 | 影响 | 建议 |
|---|---|---|
```
