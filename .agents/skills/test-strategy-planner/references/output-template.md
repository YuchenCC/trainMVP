## 一、有测试案例文档时

### 主报告：`ST-{scope}-测试策略_v{version}_{date}.md`

```markdown
# {scope} 测试策略与覆盖分析报告

## 1. 分析范围

| 项目 | 内容 |
|---|---|
| 分析范围 |  |
| 测试案例来源 |  |
| 需求/设计来源 |  |
| 来源载体与读取状态 | 本地文件 / 飞书 Docx / 飞书 Wiki；已读取正文 / 仅候选 / 读取失败 |
| 代码范围 |  |
| planner 执行人 | 研发负责人 / 提测负责人 / 技术 owner |
| 输入清单完整性 | 完整 / 待补充，说明缺项 |
| 口径差异 | 无 / 有，说明差异 |
| 存量报告处理 | 无 / 仅作索引 / 用户确认复用，说明路径 |

## 2. 分层测试方案

| 层级 | 分层原则 |
|---|---|
| L1 后端单测 | 本次纳入范围的后端可执行代码默认纳入；增量覆盖率和通过率由 `unit-test-governance` 计算和判定。 |
| L1 前端单测建议 | 仅在案例包含需要隔离验证的前端计算、校验、权限或状态逻辑时建议。 |
| L2 API/JMeter | 在后端 L1 基础上，案例还验证接口契约、鉴权链路、参数、错误码、数据库副作用或跨系统/跨服务链路时叠加。 |
| L3 UI 自动化 | 案例主要验证页面交互、关键跳转、按钮、跨页面流程或高风险用户旅程时选用。 |
| M 人工验证 | 仅在用户确认后纳入，用于低风险展示、截图确认、SIT 环境或需要人工判断的场景。 |
## 3. 测试案例分层策略与需求-案例覆盖矩阵

| 需求点/案例组 | 测试案例 | 测试目标 | 推荐层级 | 分层理由 | 资产层级 | 覆盖结论与缺口 | 优先级 |
|---|---|---|---|---|---|---|---|

`推荐层级` 是模块 B 的策略结论，不得由测试资产反推；`资产层级` 是模块 D 识别的已有测试资产实际层级，必须按测试边界、断言和工具识别。`覆盖结论与缺口` 合并填写“已覆盖”“部分覆盖：{缺口}”“缺失：{缺口}”或“不适用”。


## 4. 自测检查点映射

| 共享检查点叶子项总数 | 代码涉及 | 不涉及 | 待确认 | 已有案例已覆盖 | 部分覆盖 | 未覆盖/需补案例 |
|---:|---:|---:|---:|---:|---:|---:|

仅展示“涉及”与“待确认”项；类别路径和检查点文字必须保留 `self-test-checkpoints.md` 原文，不得概括或改写。完整遍历记录见附件。

| 检查点类别路径 | 检查点原文 | 代码审查结论 | 建议层级 | 案例处理 |
|---|---|---|---|---|

`代码审查结论` 简要说明“涉及/待确认”及关键代码事实；`案例处理` 只填写现有案例的覆盖判断，例如“已覆盖：TC1.5.1”“部分覆盖：TC1.5.4”“未覆盖”。原文优先级、原文测试层次、完整代码证据、测试资产实际层级和证据状态只在附件完整审查记录中保留。

### 待补充案例列表

仅列出自测检查点审查发现的部分覆盖、未覆盖或待确认案例建议；不伪造正式案例编号，用户确认后再回写测试案例文档。

| 建议编号 | 关联检查点 | 测试目标 | 前置条件 | 步骤 | 预期结果 | 建议层级 | 优先级 | 回写状态 |
|---|---|---|---|---|---|---|---|---|

`回写状态` 填写“待确认”“仅保留报告”“已回写”或“回写阻塞”；未经用户确认不得填写“已回写”。

## 5. 覆盖现状

| 范围 | 案例数 | 已覆盖 | 部分覆盖 | 缺失 | 可复用测试资产 | 主要缺口 |
|---|---:|---:|---:|---:|---|---|

## 6. TODO 清单与后续执行

只列尚未完成的行动、确认项、门禁或风险处置；不要重复“测试资产计划”的覆盖范围。涉及测试资产时，在“关联测试资产”中引用附件第 3 节即可。

| 优先级 | 待办行动 | 关联测试资产 | 完成标准 |
|---|---|---|---|
| P0 | 对本次后端增量代码执行单测门禁检查 | 附件第 3 节的 L1 测试文件 | 增量覆盖率 > 80%、通过率 100%，由 `unit-test-governance` 判定 |

## 7. 人工验证方案

| UI 场景 | 替代方式 | 是否询问用户 | evidence/summary.md 路径 | 必须证据 | 风险 |
|---|---|---|---|---|---|

## 8. 风险提醒

| 风险 | 影响 | 建议 |
|---|---|---|
```

## 二、没有测试案例文档时

不生成正式主报告。测试案例文档是本 skill 的必需输入。

如果没有已确认测试案例文档，只输出以下待确认提示，不进入覆盖分析：

```markdown
# {scope} 测试案例缺失提醒

本次测试策略分析未开始，因为缺少已确认测试案例文档。

| 项目 | 内容 |
|---|---|
| 分析范围 |  |
| 已检查位置 |  |
| 候选测试案例 |  |
| 需要用户提供 | 测试案例文档路径，或确认候选文件适用于本次范围 |

请补充测试案例文档后重新执行 test-strategy-planner。
```
