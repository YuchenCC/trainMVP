# 测试资产覆盖分析

本文件定义模块 D：在模块 B 已完成逐案例分层策略、模块 C 已完成自测检查点白盒审查后，实际读取已有测试代码和测试资产，识别其实际层级并形成覆盖结论。模块 D 按 `analysis-state.md` 将证据补充到工作状态；默认保持在当前对话，只有满足持久化触发条件时才写入文件，供模块 E 生成正式报告。

本模块不重新决定推荐测试层级。测试资产是覆盖证据，不是测试案例来源、策略输入或覆盖范围边界。

模块 D 完成后立即把逐案例资产字段、覆盖现状和 TODO 增量写入既定报告目标文件，并补充附件的资产字段；不等模块 E 重新渲染整份报告。

## 1. 前置条件与输入边界

进入模块 D 前，模块 B 必须已经为每个案例或案例组形成：测试目标、推荐测试层级、分层理由、内部逻辑代码锚点、补充风险和优先级；模块 C 必须已形成涉及检查点、原文测试层次、代码证据、案例覆盖判断和补充案例建议。

模块 D 输入：

- 模块 B 的逐案例策略中间结论。
- 模块 C 的自测检查点完整遍历记录和补充案例建议。
- 已确认范围内的已有单测、集成测试、API/JMeter、Playwright/Cypress 和测试辅助资产。
- 需要时的 commit、branch 或文件列表，用于识别增量风险和新增/修改资产；它们不是测试案例覆盖边界。

读取边界：测试资产检索分为“目录索引”和“正文读取”两步。目录索引只使用 `rg --files`、测试配置和脚本入口获取文件名、目录、类型及命令，不读取全部测试正文；正文读取前必须用测试案例编号、页面/API/方法锚点、逻辑代码调用链和模块 C 的涉及检查点建立候选索引。

不得用以下内容替代逐案例覆盖分析：

- 本次提交中是否新增了测试文件。
- 测试工具名称、测试文件名或测试数量。
- 旧报告中的覆盖结论。
- 逻辑代码本身存在的行为。

## 2. 必须实际读取测试资产

必须查找并读取相关已有测试资产；不得只看提交列表或文件名。先索引工程约定的下列位置和类型：

- `__tests__`、`test`、`spec` 等单测目录。
- 后端单测、集成测试、API 测试。
- JMeter 脚本或团队约定的接口自动化资产。
- Playwright、Cypress 或其他 UI 自动化脚本。
- fixture、mock、测试 helper、测试数据和报告配置等辅助资产。

索引完成后，按以下顺序筛选候选：

1. 强相关：命中案例编号、API 路径、页面路由、核心方法、状态/权限/异常锚点或明确测试断言；必须读取正文。
2. 用户明确指定：即使没有命中锚点，也必须读取正文，并记录纳入理由。
3. 弱相关：只命中字段、类型、通用 helper 或间接依赖；先不读取，只有强相关文件显示依赖时再补读。
4. 排除：读取片段或索引信息后确认与本次案例无关；不读取正文，只记录排除原因。

候选索引至少记录：

| 测试资产 | 命中锚点 | 关联案例/检查点 | 相关性 | 处理 |
|---|---|---|---|---|

“检查目录”只表示完成文件索引，不表示已读取目录内全部测试文件。未命中锚点且不属于用户明确范围的资产，不纳入本轮正文读取；需要说明“已查找但未纳入”时，只记录目录、搜索范围和数量，不展开无关文件内容。

判断“已覆盖”必须能定位到测试资产中的至少一类实际证据：

- 用例名称。
- 断言。
- 脚本步骤。
- API 请求和预期响应。
- 页面操作与预期结果。
- 人工验证证据。

代码存在、接口被调用或测试文件存在，都不等于测试案例已覆盖。

前后端分离项目应分端核对：后端至少覆盖 route/controller、service、DTO/schema/entity、权限/鉴权、状态流转、异常分支及其已有测试；前端至少覆盖页面入口、路由、组件交互、接口调用、表格字段展示、状态按钮、权限可见性及已有组件测试/UI 自动化。只获得一端时必须明确未纳入的另一端，不得补充推测。

## 3. 识别测试资产的实际层级

先识别每项测试资产的**实际测试层级**，再映射到测试案例：

| 判断项 | 判断依据 |
|---|---|
| 测试边界 | 调用纯逻辑/service、HTTP API、浏览器页面，还是依赖人工环境 |
| 断言对象 | 业务规则、接口响应、数据库副作用、页面交互，还是人工证据 |
| 工具 | 只能辅助识别，不能单独决定层级 |

例如 `Vitest + Supertest` 说明已有接口或集成证据，不能自动将全部关联案例的推荐层级改为 L2。若模块 B 已基于案例目标推荐 L1，已有 L2 证据通常只能形成“部分覆盖”；若模块 B 推荐 L2，则按 L2 的断言和边界判断覆盖。

常见实际层级判断：

| 实际层级 | 证据特征 |
|---|---|
| L1 | 直接验证可隔离的纯函数、service、mapper、validator、状态判断或异常分支 |
| L2 | 通过 HTTP/API 验证契约、鉴权、错误码、数据库副作用或跨服务链路 |
| L3 | 通过浏览器验证页面加载、按钮交互、路由跳转、跨页面流程或权限显隐 |
| M | 人工操作记录、截图、日志、数据库或接口响应等可审计证据 |

## 4. 逐案例覆盖判断

每个测试案例必须分别记录：

```text
推荐层级
资产层级
证据（文件、测试名称、断言）
覆盖结论：已覆盖 / 部分覆盖 / 缺失 / 不适用
缺口和 TODO
```

判断规则：

- API 被调用过不等于案例已覆盖，必须存在对应断言。
- 已有资产实际层级与推荐层级不一致时，不能自动判为已覆盖；应判断断言是否满足策略，通常标为“部分覆盖”。
- 已有测试覆盖行为但文件名或案例编号不匹配时，标为“部分覆盖，可复用但需映射或重命名”。
- UI 展示、按钮可见性、页面跳转、标签页切换、返回列表不能用 L2 抵扣；应分别存在 L3 或 M 证据。
- 数据库副作用、接口响应字段、鉴权、错误码的验证属于 L2 证据。
- 单测门禁只可标记“需交给 `unit-test-governance` 判断”；本模块不得给出通过或不通过结论。
- 只有提交代码和提交测试文件、没有逐案例策略和断言映射时，不得输出为完整测试案例覆盖分析。

## 5. 覆盖结论与输出字段

覆盖结论使用以下枚举：

| 结论 | 含义 |
|---|---|
| 已覆盖 | 已有资产的实际层级和断言满足该案例的推荐策略 |
| 部分覆盖 | 存在相关证据，但层级、场景、断言、边界或映射不完整 |
| 缺失 | 没有可验证的测试资产或人工证据 |
| 不适用 | 经需求和策略确认，该层或该验证方式不适用 |

输出到附件的最小字段：

| 案例 | 测试目标 | 推荐层级 | 资产层级 | 证据 | 覆盖结论 | 缺口 | TODO |
|---|---|---|---|---|---|---|---|

主报告只汇总已覆盖、部分覆盖、缺失和可复用资产数量及主要缺口；逐案例证据保留在附件。

## 6. 自测检查点的测试资产证据

模块 D 必须接收模块 C 已形成的涉及/待确认检查点映射，并为每个检查点补充已有测试资产证据。

处理规则：

- 模块 C 决定检查点是否涉及、原文测试层次、代码证据、案例覆盖判断和补充案例建议；模块 D 不得重新判定或改写这些字段。
- 模块 D 仅查找对应的已有测试资产、识别实际层级并记录断言或人工证据。
- 没有测试资产证据时，保留模块 C 的补充案例建议或风险，并将其作为测试缺口或 TODO；不能因缺少测试资产而把检查点改为“不涉及”。
- 检查点的测试资产证据是对策略的补充视图，不能替代逐案例的覆盖结论。

输出字段：

| 检查点类别路径 | 检查点原文 | 原文测试层次 | 代码证据 | 涉及判定 | 案例覆盖判断 | 补充案例建议 | 资产层级 | 证据 | 当前证据状态 | TODO |
|---|---|---|---|---|---|---|---|---|---|---|

其中“当前证据状态”使用：`有充分证据`、`证据不完整`、`无证据`、`不适用`。它不等同于测试案例的“已覆盖/部分覆盖/缺失/不适用”结论。

## 7. 模块 D 自检

结束模块 D 前确认：

- 已实际读取相关测试文件，而不是只查看提交记录。
- 每个案例先有来自模块 B 的推荐层级和分层理由，再记录实际资产层级与覆盖结论。
- 测试资产没有被写成逻辑代码，也没有被用来决定推荐策略。
- 已为模块 C 中每个涉及/待确认的自测检查点补充测试资产证据，或明确记录“无证据”。
- 自测检查点的涉及判定、原文测试层次、代码证据、案例覆盖判断和补充案例建议未被模块 D 改写。
- 存量报告未在用户未确认复用时替代源文件分析。
- 未读范围、缺失测试资产和无法确认的断言均已标为风险或 TODO。
