# TestGov Analyzer

## 用途

在 `testgov-planner` 之后使用。读取 `reports/testgov/specs-{scope}.md`，输出证据化覆盖分析。

默认执行人：测试人员或研发负责人。审核人：测试负责人 + 研发负责人。

本 skill 不运行测试、不判定后端单测门禁、不创建人工证据、不生成最终管理层报告。

## 必须输入

- TestGov specs 文件。
- specs 中列出的需求、用户故事、详细设计。
- specs 中列出的测试案例文档。
- specs 中列出的后端/前端代码影响范围。
- specs 中列出的已有测试资产和报告路径。

如果必要资料缺失，输出阻塞说明并要求用户确认。

## 分析粒度

默认分析单元是一个用户故事。不得把包含多个用户故事的 Task 当作一个正式分析报告。

面向较小模型或代码量较大时，优先按业务拆分，而不是一次读取全部源码：

| 输入类型 | 处理建议 |
|---|---|
| 单个分析单元相关代码文件 > 20 个 | 暂停并按用户故事/API/页面/模块拆分 |
| 单个代码文件 <= 800 行 | 如果直接相关，可以整文件读取 |
| 单个代码文件 800-1200 行 | 先读结构和 diff，再决定是否整读 |
| 单个代码文件 > 1200 行 | 只读变更方法/组件和附近上下文 |
| 复杂前端组件 > 500 行 | 按组件区域、hook/composable/helper 或交互流程拆分 |
| 需求/测试案例文档 > 2000 行 | 按用户故事、验收点或测试案例组拆分 |

如果超出阈值，先输出拆分方案；拆分会影响业务范围时，让用户确认。

## 分层判断规则

| 层级 | 适用内容 | 说明 |
|---|---|---|
| L1 后端单测 | 业务规则、状态流转、参数校验、异常分支、金额/时间计算、mapper/converter、service 分支 | 后端硬门禁候选项，最终由 unit-gate 判定 |
| L1 前端单测建议 | 表单校验、权限显隐、分页排序筛选、金额/日期计算、hook/composable、工具函数、状态管理 | 仅条件触发建议，不是默认硬门禁 |
| L2 API/JMeter | 接口契约、鉴权、入参校验、关键状态变化、错误码、幂等、跨服务链路 | 适合接口自动化验证 |
| L3 UI 自动化 | P0/P1 用户流程、关键页面跳转、核心按钮、跨页面联动 | 只覆盖高价值稳定流程，避免大量慢速 UI 用例 |
| M 人工验证 | 低频 UI 展示、截图确认、SIT 环境验证、日志证据、需人工判断的视觉或业务状态 | 不能抵扣后端单测门禁 |

## 分析规则

- 以测试案例为主线，映射：需求 -> 测试案例 -> 代码影响 -> 推荐层级 -> 当前证据 -> 缺口。
- 覆盖状态只使用：`已覆盖`、`部分覆盖`、`缺失`、`不适用`、`待确认`。
- 后端 L1 硬门禁只列“需交给 unit-gate 判定”，不得直接写通过。
- 前端单测只作为可隔离复杂逻辑的条件触发建议。
- API 执行证据交给 api-jmeter。
- UI 自动化证据交给 ui-flow。
- 人工证据交给 manual-evidence。

## 输出

写入：

```text
reports/testgov/analysis-{scope}.md
reports/testgov/appendix-{scope}-case-layer-map.md
```

主报告必须包含：

1. 分析范围和证据来源。
2. 需求覆盖摘要。
3. 分层测试方案。
4. 需求 / 测试案例 / 代码影响矩阵。
5. 现有测试资产判断。
6. 后端单测候选项和需传给 unit-gate 的任务。
7. 前端单测触发建议。
8. API/JMeter 覆盖任务。
9. UI 自动化任务。
10. 人工验证候选任务。
11. 不建议放进单测的内容。
12. 未覆盖风险、TODO 清单和下游执行顺序。

附件表头：

| 案例 | 需求 | 后端影响 | 前端影响 | 推荐层级 | 当前证据 | 缺口 | 下游 skill |
|---|---|---|---|---|---|---|---|
## 输出模板规则

生成输出文件前，必须先读取本 skill 自带模板：

```text
references/output-template.md
```

按模板结构输出。没有证据的字段写 `缺失`、`待确认` 或 `不适用`，不得编造结果。
