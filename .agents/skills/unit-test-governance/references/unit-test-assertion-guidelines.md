# 单元测试有效断言规范

## 用途

本规范是独立的单元测试断言参考，不依赖项目编码规范、测试框架或领域文档。

适用于：

- 后端 service、domain、validator、mapper、policy 和工具函数单测。
- 前端表单逻辑、权限逻辑、状态逻辑、hook/composable 和工具函数单测。
- `unit-test-governance` 自动生成或追加的测试。

## 核心规则

有效断言必须验证被测逻辑产生的**可观察业务结果**，不能只证明代码被调用或没有抛异常。

每个测试至少应形成：

```text
输入/前置条件 -> 调用被测逻辑 -> 业务结果断言
```

断言优先级：

1. 返回值、错误类型、错误码和错误消息。
2. 状态变化、字段值、集合内容、排序、分页和计算结果。
3. 外部依赖的关键业务副作用，例如持久化参数、审计记录对象、事件发布内容。
4. 依赖调用次数和参数，仅当调用本身是被测业务行为时验证。

## 必须遵守

### 1. 断言业务结果

```typescript
const result = await service.submitReview(input);

expect(result.status).toBe('PENDING_REVIEW');
expect(result.reviewedBy).toBe(operatorId);
```

不要只写：

```typescript
await service.submitReview(input);
expect(mockRepository.update).toHaveBeenCalled();
```

后者只能证明调用发生，不能证明状态、字段或业务结果正确。

### 2. 异常场景断言异常语义

异常测试至少断言错误类型、错误码或业务消息中的一个稳定语义；高风险权限、状态和参数校验应断言错误码或错误类型。

```typescript
await expect(service.submitReview(input)).rejects.toMatchObject({
  code: 'INVALID_STATUS',
});
```

禁止只写：

```typescript
await expect(service.submitReview(input)).rejects.toThrow();
```

### 3. 分支必须有区分度

不同输入应能产生不同且可验证的结果。测试通过时，删除或破坏被测分支中的关键逻辑，测试应能够失败；否则标记为“断言较弱”。

### 4. Mock 只隔离依赖

Mock 数据库、HTTP client、消息队列、第三方 SDK、时间和随机数等外部依赖，但不要 Mock 正在验证的业务逻辑。

只有当外部调用本身属于业务要求时，才断言调用参数、次数或调用顺序，例如：

- 状态变化必须写入 `StatusLog`。
- 失败后必须按规则重试指定次数。
- 必须发布包含指定字段的领域事件。

如果测试的大部分断言都在验证 Mock 自身，标记为弱覆盖。

### 5. 不用快照替代核心断言

快照可以辅助发现结构变化，但不能替代状态、权限、错误码、金额、日期、分页等核心业务断言。

### 6. 不过度绑定实现细节

优先断言公开行为和业务结果，不断言无业务意义的私有变量、内部调用顺序或具体实现结构。实现细节只有在它本身是稳定业务约束时才断言。

## 自动补齐要求

自动生成单测时，每个候选测试必须先列出：

| 测试目标 | 输入/前置条件 | 预期业务结果 | 关键断言 | 关联分支 |
|---|---|---|---|---|

生成后的测试必须满足：

- 至少有一个业务结果断言。
- 异常测试断言稳定错误语义。
- Mock 断言不能作为唯一断言，除非调用副作用就是测试目标。
- 测试名称描述“场景 -> 预期结果”。
- 删除关键业务逻辑后，测试应有机会失败。

## 审查结论

单个测试的断言状态只能填写：

- `有效断言`：验证了可观察业务结果或明确的业务副作用。
- `部分有效`：有业务断言，但缺少关键字段、分支或异常语义。
- `弱断言`：只断言调用、存在、无异常、快照或 Mock 行为。
- `无断言`：没有有效断言。
