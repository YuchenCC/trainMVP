# 单测治理报告模板

本文件是单测治理报告的唯一结构契约。报告必须使用以下八节标题和表头；没有内容时保留章节并填写“无”或“不适用”。

## 1. 单测门禁摘要

| 项目 | 结果 |
|---|---|
| 报告阶段 | 分析报告 / 已验证报告 |
| 变更范围 |  |
| 已检查变更文件数 |  |
| 必须覆盖项 |  |
| 排除项 |  |
| 本地执行单测数 |  |
| 本地通过数 |  |
| 本地失败数 |  |
| 本地单测通过率 |  |
| 本地增量覆盖率 |  |
| 覆盖率来源摘要 | 本地命令 / 流水线数据 / 待采集 |
| 本地门禁预判 | 本地通过 / 本地不通过 / 本地阻塞 |
| 流水线门禁状态 | 待流水线复核 / 流水线通过 / 流水线不通过 |

本地结果不等于最终门禁；流水线执行后，以流水线通过率和增量覆盖率为最终依据。

## 2. 变更逻辑与单测证据

必须穷举本轮范围内的可执行变更逻辑。只有按 `test-evidence-mapping.md` 实际读取相关测试正文后，才能填写“已有单测证据”和“断言质量”；不得仅凭文件名或提交记录判断。

| 文件 | 变更逻辑 | 分类 | 关联 L1 案例 | 已有单测证据 | 断言质量 | 推荐测试层级 |
|---|---|---|---|---|---|---|

`分类`只能填写：`单测已覆盖`、`缺少单测`、`断言较弱`、`排除：无逻辑`、`需要 API/集成/E2E 覆盖`。

`断言质量`按 `unit-test-assertion-guidelines.md` 填写：`有效断言`、`部分有效`、`弱断言`、`无断言`或`不适用`。

## 3. L1 策略案例覆盖检查

从测试策略报告第 3 节中只提取“推荐层级”包含 `L1 后端单测` 的案例。逐项核对是否能映射到第 2 节的变更逻辑与已有单测证据；既有测试证据不能改变策略案例范围。

| 覆盖重点 | 策略建议 | 第 2 节关联逻辑/证据 | 缺口 | 检查结论 |
|---|---|---|---|---|

`检查结论`只能填写：`已覆盖`、`部分覆盖`、`未覆盖`。

## 4. 自测检查点 L1 覆盖现状

只列适合 L1 单测治理的检查点。逐项核对是否能映射到第 2 节的变更逻辑与已有单测证据。

| 检查点 | 优先级 | 第 2 节关联逻辑/证据 | 检查结论 |
|---|---|---|---|

`检查结论`只能填写：`已覆盖`、`部分覆盖`、`未覆盖`、`不适用`。

## 5. 需补充单测清单

以第 2 节的“缺少单测”“断言较弱”为主，补入第 3、4 节发现但未纳入第 2 节的遗漏项，并去重。

| 优先级 | 文件 | 检查点/变更逻辑 | 建议测试用例名 | 推荐测试层级 |
|---|---|---|---|---|

### 自动补齐记录（仅启用自动补齐时填写）

| 补齐状态 | 目标测试文件 | 新增/追加测试 | 本地执行结果 | 本地覆盖变化 | 阻塞原因 |
|---|---|---|---|---|---|

`补齐状态`只能填写：`待确认`、`已生成`、`本地验证通过`、`本地验证失败`、`阻塞`。未执行单测时不得填写“本地验证通过”。

## 6. 未覆盖或排除风险

生成本节前必须读取 `risk-classification.md`。高风险未覆盖不得给出“本地通过”预判。

| 风险等级 | 文件 | 变更逻辑 | 分类 | 原因 | 风险 | 处理要求 |
|---|---|---|---|---|---|---|

如果没有风险，写一行 `无`。

## 7. 单测执行证据

只记录本轮实际取得的本地执行证据；流水线结果未取得时明确标记待复核。

| 执行命令 | 执行范围 | 本地通过/失败 | 本地增量覆盖率 | 本地报告路径 | 执行时间/流水线链接 |
|---|---|---|---|---|---|
