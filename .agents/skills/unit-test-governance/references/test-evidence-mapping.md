# 单测证据判定规则

本文件用于将用户明确提供的单测文件映射到本轮变更逻辑，形成报告第 2 节的真实覆盖证据。它不用于搜索或猜测测试文件；用户填写“无”时，不执行测试资产映射。

## 判定前提

- 只读取用户在输入确认中列出的相关单测文件。
- 必须阅读测试正文，不能仅凭文件名、目录、提交记录、测试数量或测试框架判断覆盖。
- 测试策略案例用于核对策略，不得由既有测试资产反向决定推荐层级。

## 映射方法

逐个测试用例与变更逻辑比对，可使用以下信号：

| 信号 | 示例 |
|---|---|
| 相同模块路径 | `requirements/service.ts` -> `requirements*.test.ts` |
| 相同功能名 | `requirement`、`reqCode`、需求编号 |
| 相同函数或类名 | `generateReqCode`、`RequirementService` |
| 相同接口或用户故事 | `POST /api/requirements`、`US1.1` |
| 相同单元级缺陷症状 | 数字排序、重试分支、冲突处理 |
| 相同行为变化 | 格式化、校验、权限、状态流转 |

每条映射至少记录：测试文件、测试名称、关联变更逻辑、关键输入或前置条件、关键断言和断言质量。

## 证据结论

| 情况 | 第 2 节分类/证据处理 |
|---|---|
| 测试验证了该逻辑的业务结果或关键业务副作用 | 作为 L1 单测证据；断言质量按《单测有效断言规范》判定。 |
| 测试覆盖同一逻辑但缺少关键分支、字段或异常语义 | 标记为“部分覆盖”或“断言较弱”，列入补充单测清单。 |
| 测试只断言调用、无异常、快照或 Mock 自身行为 | 不作为充分覆盖证据，标记为“弱断言”。 |
| API、集成、E2E、压测、契约测试 | 标记为“非 L1 证据”，不计入 L1 单测门禁，也不替代缺失单测。 |
| 指定测试文件与本轮逻辑无可验证关联 | 记录为“未映射”，不据此声称覆盖。 |

## 输出约束

报告第 2 节的“已有单测证据”必须写出测试名称和关键断言，不得只写“存在测试文件”。没有映射到的变更逻辑，按“缺少单测”或“需要 API/集成/E2E 覆盖”处理。
