# 单测治理

## 配置读取（必须步骤）

1. 检查 `config/project-config.yaml` 是否存在
2. 如果存在，读取配置并使用其中的路径、命令、模式等
3. 如果不存在，使用默认值，并提示用户：
   > "未找到项目配置文件 config/project-config.yaml，请先创建或确认配置文件位置。当前使用默认配置。"

### 使用的配置项

| 配置项                                     | 用途           |
| --------------------------------------- | ------------ |
| `paths.backend.unit_tests`              | 校验用户指定单测路径与自动补齐落点的参考 |
| `paths.backend.coverage_report`         | 覆盖率报告目录      |
| `commands.backend_unit_test`            | 后端单测执行命令     |
| `paths.governance.unit_test_governance` | 单测治理报告输出目录   |
| `report_naming.unit_test_governance`    | 单测治理报告命名模板   |
| `tech_stack.unit_test`                  | 单元测试框架类型     |

## 用途

使用这个 skill 作为前端和后端代码变更的统一单测门禁。它适用于 Java、JavaScript、TypeScript、React、Vue3 等技术栈。

这个 skill 不教授具体测试框架语法，例如 JUnit5、Vitest、Jest、Vue Test Utils。具体实现细节应交给框架专项 skill 或项目已有规范。

## 硬性门禁

### 后端单测门禁

后端 L1 是默认硬门禁：

```text
后端增量单测覆盖率 > 80%
后端单测通过率 = 100%
```

必须单独展示，且不能被 L2/L3/人工验证抵扣。

如果没有本地覆盖率报告或单测执行结果，只能写“本地阻塞/待补”，不能写“本地通过”。

### 前端单测口径

前端不默认要求单测覆盖率门禁。建议只在以下情况补前端单测：

- 表单校验、字段格式化。
- 权限显隐、按钮状态。
- 分页、排序、筛选。
- 金额、日期、数量计算。
- React hook、Vue composable。
- 工具函数、状态管理。

纯展示、简单表格、普通弹窗、文案展示，优先走 L3 UI 或人工验证。

### 通用规则

单测治理只有同时满足下面两个条件，才能给出“本地通过”预判：

1. 本次变更范围内的可执行代码，增量单测覆盖率必须大于 80%。
2. 单测通过率必须等于 100%。

这两个门禁只适用于单元测试。不要使用 API、集成、E2E、JMeter、Postman、Playwright、Cypress 或其他非单测结果来满足单测增量覆盖率或通过率。

如果任一本地门禁不满足，必须标记为“本地不通过”，并列出需要补救的动作。

除非用户明确要求，否则不要使用全项目覆盖率作为主要门禁。优先使用变更代码覆盖率或需求范围覆盖率。

## 适用范围

只评估用户明确确认的可执行变更集；需求、任务和分支只能作为定位入口，不能直接作为门禁范围。

用户必须提供以下输入：

1. **代码范围**：明确的文件/函数列表，或明确基线的 commit range：`base..head`。
2. **相关单测文件**：明确的已有单测文件路径列表；没有已有单测时明确填写“无”。

可选输入：

- 测试策略报告路径：仅用于提取推荐层级包含 `L1 后端单测` 的策略案例。
- 需求或任务名称：仅用于命名和辅助定位，不扩大代码范围。

用户未提供代码范围或相关单测文件时，只输出待补输入清单；不得自行将整个长期分支、大任务或全仓测试目录纳入门禁范围。
## 输入确认流程

开始分析前，必须让用户确认以下输入：

1. **代码范围**：明确文件/函数列表，或明确基线的 `base..head` commit range。
2. **相关单测文件**：明确的已有单测文件路径列表；没有时填写“无”。
3. **L1 策略案例（可选）**：如用户提供测试策略报告，从第 3 节矩阵中只提取“推荐层级”包含 `L1 后端单测` 的案例。既有测试证据不得改变策略案例范围。
4. **自测检查点**：读取 `references/self-test-checkpoints.md`，筛选适合 L1 单测治理的内容。
5. **自动补齐开关**：确认本轮是否允许生成或追加单测；未确认前只输出缺口，不修改测试文件。

确认清单必须原样展示用户提供的代码范围和测试文件；只有用户明确确认后才能开始分析。用户未提供第 1 或第 2 项时，停止在输入确认阶段并要求补充，不做全仓搜索或范围推断。
## 工作流程

报告结构以 `references/output-template.md` 为唯一契约；不得在本文件或输出中自定义、增删或改名章节。流程分两步：先分析并输出报告，再由用户确认后补齐或验证；按模板章节顺序逐段回填。

### 增量输出规则

每个模板章节的证据链闭环后，立即回填并写入同一份报告，不等待后续章节完成。对话中简要说明已完成章节和当前结论；不要重复整份报告。后续步骤只能补充未完成章节，或因新增执行证据修订受影响的既有章节并说明修订原因。
### 第一步：分析并输出报告
1. 确认用户提供的代码范围、相关单测文件、可选 L1 策略案例和自动补齐开关；用户确认后读取 `output-template.md`，锁定报告路径，并初始化第 1 节“单测门禁摘要”为“分析报告”。
2. 读取 `references/change-logic-classification.md`，逐项识别本轮范围内的逻辑。用户提供单测文件时，实际读取并按 `references/test-evidence-mapping.md` 映射测试、逻辑和断言；用户填写“无”时，记录无已有单测证据，不做测试资产搜索。完成第 2 节“变更逻辑与单测证据”并立即回填报告。
3. 仅以用户提供的 L1 策略案例为范围，核对其能否映射到第 2 节，完成第 3 节“L1 策略案例覆盖检查”并立即回填报告。
4. 读取 `self-test-checkpoints.md` 筛选适用 L1 检查点，并按 `unit-test-assertion-guidelines.md` 审查断言质量，完成第 4 节“自测检查点 L1 覆盖现状”并立即回填报告。
5. 从第 2 节的“缺少单测”“断言较弱”出发，补入第 3、4 节交叉发现的遗漏项并去重，完成第 5 节“需补充单测清单”；读取 `risk-classification.md` 后完成第 6 节“未覆盖或排除风险”，并立即回填第 5、6 节。
6. 尚未取得本地执行证据时，进入“未执行验证时的代码分析”：第 7 节填写“未执行/待用户确认验证”，第 8 节填写“本地阻塞或待补齐”的原因和复核条件，并立即回填完整分析报告；先让用户确认是否自动补齐，或仅执行验证。

### 第二步：用户确认后补齐或验证

7. 用户确认自动补齐时，先输出候选测试和写入范围；确认后生成或追加测试。用户不补齐但确认执行验证时，保持测试文件不变。
8. 读取 `references/test-execution-guidelines.md` 和 `references/coverage-rules.md`，再运行最小相关单测和覆盖率命令，完成第 7 节“单测执行证据”；流水线结果不可取得时，标记“待流水线复核”，并立即回填报告。
9. 根据验证结果回填第 2、5、6 节，以及第 1 节摘要和第 8 节“最终决策”，输出已验证报告。高风险未覆盖、本地通过率非 100% 或本地增量覆盖率不达标时，不得给出“本地通过”，并立即写入已验证报告。
## 单元测试自动补齐

自动补齐是可选能力，只有用户明确要求或确认启用后才执行。默认先完成缺口分析并输出“分析报告”，不直接修改测试文件。

1. 输出候选清单：目标测试文件、测试名称、输入、预期结果、覆盖分支、关联 L1 案例和优先级。
2. 询问用户是否写入，以及允许新建文件、追加已有文件还是两者都可以。
3. 用户确认后，读取 `references/test-generation-guidelines.md` 和 `references/unit-test-assertion-guidelines.md`，再适配项目已有测试框架、目录、fixture 和 mock 风格；自动生成的测试必须包含有效业务断言。
4. 优先新增或追加测试，不覆盖已有断言；不得把生成的测试直接标记为已通过。
5. 写入后按 `references/test-execution-guidelines.md` 执行最小相关单测；失败时修正生成测试或标记阻塞，不扩大业务代码修改范围。
6. 重新读取测试结果和覆盖率，更新缺口、TODO 和门禁结论。
## 未执行验证时的代码分析

### 触发时机

在第一步“分析并输出报告”完成第 5 节后，出现任一情况即进入本节：

- 用户尚未确认“自动补齐”或“仅执行验证”。
- 用户明确要求本轮只做分析。
- 指定单测无法运行，或本地覆盖率命令不可用。

它不是全仓扫描，也不是自动发现测试文件；仍只分析用户确认的代码范围和单测文件。

### 处理与退出

1. 按 `references/change-logic-classification.md` 完成变更逻辑识别和分类。
2. 读取用户指定的相关单测文件；用户填写“无”时不做测试资产搜索。
3. 区分单测已覆盖、缺失覆盖和弱覆盖，输出按优先级排序的 TODO，并区分单测 TODO 和 API/集成/E2E TODO。
4. 第 7 节标记为“未执行”，第 8 节标记为“本地阻塞/待验证”，同时说明仍缺哪些执行证据。
5. 用户确认执行验证后，退出本节并进入工作流程第二步；第 7 节写入实际执行证据，再更新相关结论。
TODO 表：

| 优先级 | 文件 | 变更逻辑 | 为什么需要单测 | 建议测试用例 | 推荐测试层级 |
| --- | -- | ---- | ------- | ------ | ------ |

优先级规则：

- `P0`：高风险业务逻辑、状态流转、权限、数据损坏、唯一性、冲突、重试、幂等、事务行为。
- `P1`：校验、转换、边界值、排序、分页、格式化、用户可见分支。
- `P2`：低风险分支强化、防御性检查、可选清晰度测试。

如果某个行为不适合单测，仍然写入 TODO，但推荐测试层级标记为 API、Integration 或 E2E。

## 报告结构契约

生成报告前必须读取 `references/output-template.md`，并按其八节标题、表头和字段输出。SKILL 只定义分析、判定和 reference 触发规则；模板是唯一报告结构来源。
## Markdown 报告文件

只要有文件系统写入权限，就必须把单测治理结果写入 Markdown 文件。

推荐路径（从配置 `paths.governance.unit_test_governance` 读取，默认 `reports/unit-test-governance`）：

统一命名格式 `ST-{scope}-单测治理_v{version}_{date}.md`（从配置 `report_naming.unit_test_governance` 读取）：

```text
{paths.governance.unit_test_governance}/ST-{scope}-单测治理_v{version}_{date}.md
```

命名规则：

- commit 范围：`ST-{short-sha}-单测治理_v{version}_{date}.md`，其中 `{short-sha}` 是 commit 前 7 位。
- branch 范围：`ST-{branch-name}-单测治理_v{version}_{date}.md`，将 `/` 和空格替换为 `-`。
- requirement 或 task 范围：`ST-{scope}-单测治理_v{version}_{date}.md`，将 `/` 和空格替换为 `-`。

命名优先级：
1. 优先从 `config/project-config.yaml` 的 `report_naming.unit_test_governance` 读取
2. 配置不存在时，使用默认命名：`ST-{scope}-单测治理_v1.0_{date}.md`

如果仓库没有对应的目录，创建它。

Markdown 文件必须使用中文，并包含 `output-template.md` 定义的八个章节和下游报告提示。

最终对话回复只需要链接报告文件，并摘要说明本地门禁预判、本地单测通过率、本地增量覆盖率和流水线复核状态。

如果无法写入文件，要明确说明原因，并在对话中提供完整中文 Markdown 内容。

## 本地预判与流水线门禁

| 本地门禁预判 | 含义 |
|---|---|
| 本地通过 | 本地增量覆盖率大于 80%、本地单测通过率 100%，且不存在高风险未覆盖逻辑。 |
| 本地不通过 | 本地覆盖率低于或等于 80%、通过率不是 100%，或存在高风险未覆盖逻辑。 |
| 本地阻塞 | 缺少本地覆盖率或测试结果，无法完成本地预判。 |

`流水线门禁状态`只能填写：`待流水线复核`、`流水线通过`、`流水线不通过`。本 skill 无法取得流水线数据时，最终状态必须为“待流水线复核”；流水线结果是最终门禁依据。

本地不通过或阻塞时，列出最小补救动作和复核条件。
## 报告发布与确认任务

先读取 `references/lark-document-source.md`。

只有报告完成“已验证报告”更新和输出自检后，才能询问用户是否通过飞书 MCP 发布。输出自检未完成时，不得创建飞书文档或任务。

询问模板：

```markdown
本次单测治理报告已生成并完成输出自检。

请确认是否通过飞书 MCP 执行以下操作：
- 创建飞书文档：是 / 否
- 创建"单测门禁确认"审批任务：是 / 否
- 审批人：请提供姓名、邮箱或 open_id；随后通过飞书 MCP 查询并确认

默认发布文件：单测治理报告；如需同时发布其他相关文档，请一并确认。
```

执行规则：

1. 用户未明确确认前，不创建飞书文档、不创建审批任务、不向任何飞书用户发送内容。
2. 创建文档时，默认发布已完成自检的单测治理报告；附件只有在用户明确确认后才一并创建或关联。
3. 创建任务时，名称固定为"单测门禁确认"，内容包含飞书文档链接、分析范围、单测门禁结论、需补充单测清单、主要风险和确认要求。
4. 创建任务前，先让用户提供审批人的姓名、邮箱或 open_id；不得先枚举全部可访问用户作为审批人候选。
5. 通过飞书 MCP 的用户目录能力按用户提供的条件查询，展示匹配用户的姓名及可确认身份信息（邮箱或 open_id）。
6. 即使查询仅匹配一个用户，也必须取得最终确认；多个匹配结果时必须让用户明确选择。
7. 查不到用户、身份信息不足、用户无法确认，或 MCP 未暴露文档/用户查询/任务或审批能力时，停止发送并说明阻塞原因。
8. 执行前确认当前飞书 MCP 已获得文档、用户查询和任务/审批所需权限；权限不足时不得声称已创建。
9. 创建完成后，通过飞书 MCP 验证文档链接、任务或审批实例标识、审批人和状态；最终回复分别说明文档和任务的创建结果，失败时记录原因和可重试事项。

普通飞书任务与原生审批实例均可承载"单测门禁确认"；优先使用当前会话实际暴露且已授权的能力，不得假设 MCP 已具备某个工具。

## 最终对话中的后续 skill 建议

后续 skill 建议只出现在最终对话，不写入报告或 TODO 表。建议必须依据当前项目实际存在的 `.agents/skills/*/SKILL.md` 动态生成。

执行步骤：

1. 扫描项目内 skill 的 `name`、`description` 和用途说明。
2. 按后续任务匹配能力：测试策略规划、API/JMeter、前端测试、UI 自动化、人工验证、最终覆盖报告等。
3. 只推荐真实存在的 skill；没有匹配项时说明"项目暂未提供对应 skill"，并给出缺失能力名称。
4. 多个候选时按与本次任务的匹配度排序，并说明原因和输入建议。
5. 不得因历史报告或旧流程出现过某个 skill 名称就默认推荐。

建议输出格式：

| 后续任务 | 推荐 skill | 是否存在 | 推荐原因 | 输入建议 |
|---|---|---|---|---|

## 最终回复约束

报告生成后，最终对话回复保持简短，只包含：

- 报告路径。
- 分析范围。
- 单测门禁结论（通过率、增量覆盖率）。
- 本地门禁预判和流水线复核状态。
- 需补充单测清单摘要。
- 输出自检结果。
- 必要的下一步建议及动态 skill 路由。
