# TestGov UI Flow

## 用途

用于 L3 UI 自动化测试治理。优先覆盖 P0/P1 高价值、稳定、跨页面的核心用户流程。

默认执行人：测试人员或前端开发。审核人：测试负责人。

UI 自动化不能抵扣后端单测门禁。UI 自动化成本高时，可以由人工验证替代，但必须在报告中说明替代原因和证据位置。

## 适用范围

- 核心入口和页面路由。
- 关键按钮和主要操作链路。
- 查询、查看、创建、编辑、提交等高频流程。
- 跨页面联动和跳转。
- 需要浏览器真实行为验证的交互。

不建议把低价值纯展示、文案、一次性低频路径全部做成 UI 自动化。

## 工作流程

1. 读取 specs 和 analyzer 中的 L3/UI 任务。
2. 查找已有 Playwright、Cypress 或其他 UI 自动化脚本和报告。
3. 能安全执行时，运行最小相关 UI 自动化；成本过高时输出人工替代建议。
4. 映射 UI 脚本到需求、测试案例和页面路径。
5. 输出已覆盖、部分覆盖、缺失、阻塞和建议人工验证项。

## 输出

写入：

```text
reports/testgov/ui-flow-{scope}.md
```

必须包含：

1. UI 自动化覆盖摘要。
2. 执行环境、浏览器、分支、报告来源。
3. 页面路径 / 测试案例 / 需求映射表。
4. 通过、失败、阻塞、未执行清单。
5. 建议新增或复用的 UI 脚本。
6. 可由人工验证替代的场景和原因。
7. 对 final-report 的结论摘要。
## 输出模板规则

生成输出文件前，必须先读取本 skill 自带模板：

```text
references/output-template.md
```

按模板结构输出。没有证据的字段写 `缺失`、`待确认` 或 `不适用`，不得编造结果。
