# TestGov Planner

## 用途

这是 TestGov 系列的第一步，用于在提测前把测试治理范围说清楚。

默认执行人：提测负责人、研发负责人或后端/前端技术负责人。测试人员可以协助，但不应独自承担分支、提交、代码影响范围确认。

本 skill 不运行测试、不判定门禁、不生成最终管理报告，只生成后续 skill 可直接使用的范围说明。

## 分析前必须确认

如果用户没有明确确认范围，先询问并等待确认。必须确认以下信息，缺失项要写入 specs 的“待确认项”：

| 项目 | 说明 |
|---|---|
| 需求范围 | Task、用户故事、验收点，默认按用户故事拆分 |
| 代码仓库 | 后端仓库、前端仓库，允许两个仓库并行存在 |
| 分支/提交 | 本次变更分支、commit、PR、diff 范围 |
| 设计文档 | 需求、详细设计、接口说明、页面说明 |
| 测试案例 | 已有测试案例文档位置；如果没有，必须询问用户在哪里或是否尚未编写 |
| 单测要求 | 后端默认增量覆盖率 > 80%，通过率 100%；前端默认不作为硬门禁 |
| 自动化脚本 | JMeter、Playwright/Cypress 等已有脚本位置 |
| 人工验证 | 是否需要 SIT 人工验证；如果需要，后续由 manual-evidence 创建轻量证据包 |
| 审核人 | 研发负责人、测试负责人、项目经理是否需要审核 |

## 拆分规则

默认按用户故事拆分。一个 Task 如果包含多个用户故事，不要直接作为一个正式分析单元。

推荐顺序：

1. Task / 大需求 -> 用户故事。
2. 用户故事仍然过大 -> 验收点或测试案例组。
3. 后端范围 -> API 组、service/module、变更方法组。
4. 前端范围 -> 页面路由、用户流程、组件组。
5. 超大文件 -> diff、变更方法/组件、附近上下文。

如果单个分析单元相关代码文件超过 20 个，或需求/测试案例超过 2000 行，必须先输出拆分建议，让用户确认后再进入正式分析。

## 单测门禁口径提醒

后端 L1 单元测试是默认硬门禁：

```text
后端增量单测覆盖率 > 80%
后端单测通过率 = 100%
```

L2 API、L3 UI 自动化、人工验证、集成测试、前端测试，都不能抵扣后端单测门禁。

前端默认没有单测硬门禁。只有遇到可隔离的复杂前端逻辑时，输出前端单测建议，例如表单校验、权限显隐、分页排序筛选、金额/日期计算、hook/composable、工具函数、状态管理。

## 输出

写入：

```text
reports/testgov/specs-{scope}.md
```

必须包含：

1. 本次范围和用户故事清单。
2. 前后端仓库、分支、commit、diff 范围。
3. 需求、设计、接口、测试案例文档索引。
4. 已有单测、JMeter、UI 自动化、人工证据索引。
5. 按用户故事拆分的 analysis_units。
6. 后端单测门禁提醒。
7. 前端单测条件触发说明。
8. 人工验证意向和待确认项。
9. 后续执行建议：analyzer -> unit-gate -> api-jmeter -> ui-flow/manual-evidence -> final-report。
## 输出模板规则

生成输出文件前，必须先读取本 skill 自带模板：

```text
references/output-template.md
```

按模板结构输出。没有证据的字段写 `缺失`、`待确认` 或 `不适用`，不得编造结果。
