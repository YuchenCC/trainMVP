# TestGov Planner

## 用途

这是 TestGov 系列的第一步，用于在提测前把测试治理范围说清楚。

默认执行人：提测负责人、研发负责人或后端/前端技术负责人。测试人员可以协助，但不应独自承担分支、提交、代码影响范围确认。

本 skill 不运行测试、不判定门禁、不生成最终管理报告，只生成后续 skill 可直接使用的范围说明。

## 分析前必须确认

**第一步：主动确认范围信息**

如果用户没有明确提供，必须先询问并等待确认，**不允许**直接写入 specs。必须确认以下信息：

| 项目 | 说明 | 如何确认 |
|---|---|---|
| 需求范围 | Task、用户故事、验收点，默认按用户故事拆分 | 读取用户输入的需求文档，或询问具体范围 |
| 代码仓库 | 后端仓库、前端仓库，允许两个仓库并行存在 | 从项目结构判断，或询问仓库路径 |
| 分支/提交 | 本次变更分支、commit、PR、diff 范围 | 通过 git 命令获取，或询问 |
| **代码范围** | **分析范围限定，必须明确指定** | **仅支持三种方式：<br>1. feature分支与主分支的差异（如 `feature/xxx...main`）<br>2. 指定 commit hash（如 `6833184`）<br>3. 指定代码文件路径（如 `trains/routes/schedule.ts`）<br>禁止使用"全量代码分析"等模糊表述** |
| 设计文档 | 需求、详细设计、接口说明、页面说明 | 检测存量文档位置，或询问 |
| 测试案例 | 已有测试案例文档位置；如果没有，必须询问用户在哪里或是否尚未编写 | 搜索存量文档，或询问 |
| 自动化脚本 | JMeter、Playwright/Cypress 等已有脚本位置 | 搜索存量脚本，或询问是否需要补充 |
| 差距项状态 | 测试案例中提到的差距项是否已修复 | 必须询问研发负责人 |
| 存量报告处理 | 检测到存量测试策略报告时，确认是否重新生成 | 使用存量报告 vs 重新生成 |
| 人工验证意向 | 是否需要 SIT 人工验证；如果需要，后续由 manual-evidence 创建轻量证据包 | 询问用户 |
| 审核人 | 研发负责人、测试负责人、项目经理是否需要审核 | 询问用户 |

**确认流程：**
1. 先读取用户提供的文档（需求、测试案例）
2. 搜索存量资产（测试策略报告、测试脚本、代码）
3. 检测到关键信息缺失或存量报告冲突时，**必须**通过 AskUserQuestion 等待用户确认
4. **禁止**在用户确认前生成 specs 文件
5. 只有获得用户明确确认后，才生成 specs 文件

## 拆分规则

默认按用户故事拆分。一个 Task 如果包含多个用户故事，不要直接作为一个正式分析单元。

推荐顺序：

1. Task / 大需求 -> 用户故事。
2. 用户故事仍然过大 -> 验收点或测试案例组。
3. 后端范围 -> API 组、service/module、变更方法组。
4. 前端范围 -> 页面路由、用户流程、组件组。
5. 超大文件 -> diff、变更方法/组件、附近上下文。

如果单个分析单元相关代码文件超过 20 个，或需求/测试案例超过 2000 行，必须先输出拆分建议，让用户确认后再进入正式分析。

## 单测门禁口径提醒

后端 L1 单元测试是默认硬门禁：

```text
后端增量单测覆盖率 > 80%
后端单测通过率 = 100%
```

L2 API、L3 UI 自动化、人工验证、集成测试、前端测试，都不能抵扣后端单测门禁。

前端默认没有单测硬门禁。只有遇到可隔离的复杂前端逻辑时，输出前端单测建议，例如表单校验、权限显隐、分页排序筛选、金额/日期计算、hook/composable、工具函数、状态管理。

## 输出

写入：

```text
reports/testgov/specs-{scope}.md
```

必须包含：

1. 本次范围和用户故事清单。
2. 前后端仓库、分支、commit、diff 范围。
3. 需求、设计、接口、测试案例文档索引。
4. 已有单测、JMeter、UI 自动化、人工证据索引。
5. 按用户故事拆分的 analysis_units。
6. 后端单测门禁提醒。
7. 前端单测条件触发说明。
8. 人工验证意向和待确认项。
9. **可执行任务清单**：按执行顺序列出后续步骤，每步包含：
   - 任务名称
   - 依赖的前置条件
   - 输入（从哪个文件读取）
   - 输出（生成哪个文件）
   - 使用哪个 skill

**可执行任务示例：**

| 顺序 | 任务 | 依赖 | 输入 | 输出 | Skill | 状态 |
|-----|------|------|------|------|-------|------|
| 1 | testgov-analyzer | 无 | specs.md | analysis-{scope}.md | testgov-analyzer | 待执行 |
| 2 | testgov-unit-gate | Task 1 完成 | specs.md + coverage report | unit-gate-{scope}.md | testgov-unit-gate | 待执行 |
| 3 | testgov-api-jmeter | Task 2 完成 | analysis.md + JMeter脚本 | api-jmeter-{scope}.md | testgov-api-jmeter | 待执行 |
| 4 | testgov-ui-flow | Task 2 完成 | analysis.md + E2E脚本 | ui-flow-{scope}.md | testgov-ui-flow | 待执行 |
| 5 | testgov-manual-evidence | Task 3/4 完成 | 差距项列表 | manual-evidence-{scope}.md | testgov-manual-evidence | 待执行 |
| 6 | testgov-final-report | Task 5 完成 | 所有前置报告 | final-{scope}.md | testgov-final-report | 待执行 |
## 输出模板规则

生成输出文件前，必须先读取本 skill 自带模板：

```text
references/output-template.md
```

按模板结构输出。**重要规则：**

- 没有证据的字段写 `缺失`、`待确认` 或 `不适用`，不得编造结果
- **如果用户选择忽略/重新生成某个存量报告，该报告不写入 specs 的「文档与测试资产」表格**，避免误导后续流程使用已废弃的文档

## 执行流程

**按以下顺序执行，每步必须完成才能进入下一步：**

1. **信息收集阶段**：
   - 读取用户提供的文档（需求、测试案例）
   - 搜索存量资产（测试策略报告、测试脚本、代码）
   - 通过 git 命令获取分支和 commit 信息

2. **用户确认阶段**：
   - 检测关键信息是否缺失（JMeter脚本、差距项修复状态、存量报告处理方式等）
   - 如有缺失，通过 AskUserQuestion 等待用户确认
   - **禁止**在用户确认前生成任何文件

3. **Specs 生成阶段**：
   - 读取输出模板 `references/output-template.md`
   - 基于收集信息和用户确认结果，填写模板
   - 生成 `reports/testgov/specs-{scope}.md`
   - specs 中包含可执行任务清单

4. **任务清单确认**：
   - 向用户展示生成的任务清单
   - 确认任务顺序、依赖关系是否正确
   - 如需调整，修改 specs 中的任务清单

5. **返回用户**：
   - 告知用户 specs 已生成，可按任务清单执行后续步骤
   - 或用户可直接指定执行某个特定任务
