## 一、系统概述

版本火车需求管理系统用于管理多系统协同发布过程中的需求池、版本火车计划、团队容量、关键节点、智能纳版和投产回滚。

**核心价值**: 将需求从"静态记录"推进到"可规划、可纳版、可追踪、可审计"的发布计划管理。

**系统架构**:
- **前端**: 基于 React 的单页应用
- **后端**: Node.js + Express + Prisma ORM
- **数据库**: PostgreSQL
- **AI能力**: 集成 Coze 实现智能纳版建议

## 四、需求池管理

### 4.1 需求生命周期

```
草稿 → 评审中 → 已就绪 → 已纳版 → 开发中 → SIT测试 → UAT测试 → 封板 → 已投产
       ↓拒绝    ↓变更    ↓移除    ↓取消
      草稿     草稿     已就绪   已取消
```

### 4.2 创建需求

1. 点击「需求池」→「新建需求」
2. 填写以下信息：
   - **标题**: 简明描述需求内容
   - **描述**: 详细说明需求背景和目标
   - **归属系统**: 选择需求所属系统
   - **优先级**: P0(紧急)、P1(高)、P2(中)、P3(低)
   - **工作量点数**: 预估开发工作量（1-13点）
   - **BA**: 业务负责人
   - **PM**: 产品负责人
   - **依赖需求**: 关联依赖的其他需求
3. 点击「保存草稿」或「提交评审」

### 4.3 编辑需求

- **草稿状态**: 可自由编辑所有字段
- **非草稿状态**: 仅可编辑部分字段，受权限限制

### 4.4 需求评审

1. BA 发起评审后，项目经理收到评审任务
2. 项目经理进入需求详情页
3. 选择「评审通过」或「评审拒绝」
4. 拒绝的需求自动退回草稿状态

### 4.5 紧急变更

当需求已封板但需要修改时：
1. 点击需求详情页的「申请紧急变更」
2. 填写变更原因和内容
3. 等待项目经理或测试经理审批
4. 审批通过后需求退回草稿状态

### 4.6 需求筛选

需求列表支持以下筛选条件：
- 状态（草稿、评审中、已就绪、已纳版等）
- 归属系统
- 优先级
- BA/PM
- 创建时间范围

## 五、版本火车与班次管理

### 5.1 核心概念

- **版本火车**: 发布计划容器，如 "V2.0 版本火车"
- **班次**: 具体的发布时间点，如 "2026-06-01 班次"
- **容量**: 班次可承载的工作量上限
- **纳版**: 将需求加入班次的过程

### 5.2 创建版本火车

1. 点击「版本火车」→「新建火车」
2. 填写信息：
   - **名称**: 火车名称
   - **描述**: 火车说明
   - **搭载系统**: 选择参与发布的系统
   - **人员容量**: 设置团队总容量
3. 点击「创建」

### 5.3 创建班次

1. 进入火车详情页
2. 点击「新建班次」
3. 填写信息：
   - **名称**: 班次名称（如 "2026-06-01"）
   - **目标日期**: 计划发布日期
   - **关键节点**: 设置纳版截止、SIT截止、UAT截止、封板时间
4. 点击「创建」

### 5.4 需求纳版

1. 进入班次详情页
2. 在「候选需求」列表中选择已就绪的需求
3. 点击「纳入班次」
4. 或使用「AI智能纳版」获取建议

### 5.5 移除需求

1. 在班次详情页找到已纳版的需求
2. 点击「移除」按钮
3. 确认后需求返回已就绪状态

### 5.6 投产与回滚

**投产**:
1. 班次状态为「封板」后可投产
2. 点击「确认投产」
3. 确认后所有需求状态变为「已投产」

**回滚**:
1. 已投产的班次可回滚
2. 点击「回滚」按钮
3. 填写回滚原因
4. 确认后需求状态回退

### 5.7 完成班次

所有需求投产完成后：
1. 点击「完成班次」
2. 班次进入归档状态

## 六、AI智能纳版

### 6.1 功能说明

AI智能纳版基于以下规则提供纳版建议：
- **优先级排序**: 高优先级优先纳入
- **容量检查**: 不超过班次容量限制
- **可装则装**: 符合条件的需求尽量纳入
- **依赖风险提示**: 提示未纳入的依赖需求

### 6.2 使用步骤

1. 进入班次详情页
2. 点击「AI智能纳版」按钮
3. 等待 AI 分析（约5-10秒）
4. 查看建议：
   - **建议纳入**: AI推荐纳入的需求
   - **建议延期**: 因容量或优先级原因建议延期的需求
   - **风险警告**: 依赖缺失等风险提示
5. 点击「确认纳版」应用建议，或手动调整

> **注意**: AI 建议仅作为参考，需火车管理员人工确认后才真正纳版。

## 八、常见问题

### Q1: 如何修改已纳版的需求？

A: 已纳版的需求需要先从班次中移除，或申请紧急变更（封板后）。

### Q2: 班次容量是如何计算的？

A: 班次容量为创建时设置的人员容量，系统会检查所有已纳版需求的工作量总和是否超过容量。

### Q3: AI 纳版建议是否一定会被采纳？

A: 不会。AI 仅提供建议，最终决策由火车管理员人工确认。

### Q4: 已投产的需求如何回滚？

A: 在班次详情页点击「回滚」按钮，填写回滚原因后确认即可。

### Q5: 如何查看操作历史？

A: 在需求详情页的「操作历史」标签页可查看所有变更记录。

## 九、附录

### A. 需求状态说明

| 状态 | 说明 | 可执行操作 |
|------|------|----------|
| 草稿 | 需求创建后未提交评审 | 编辑、提交评审、取消 |
| 评审中 | 等待项目经理评审 | 评审通过、评审拒绝 |
| 已就绪 | 评审通过，等待纳版 | 纳版、变更 |
| 已纳版 | 已纳入班次 | 移除、开发中 |
| 开发中 | 技术经理标记开发开始 | SIT测试 |
| SIT测试 | 系统集成测试中 | SIT通过、UAT测试 |
| UAT测试 | 用户验收测试中 | UAT通过、封板 |
| 封板 | 冻结，等待投产 | 紧急变更、投产 |
| 已投产 | 已发布到生产环境 | 回滚 |
| 已取消 | 投产前取消 | 删除 |
