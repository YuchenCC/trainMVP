#### RBAC权限矩阵落地实现

将PRD中的权限矩阵从文档落地为代码，实现前后端统一的权限控制：

| 组件 | 说明 |
|------|------|
| `Operation` 枚举 | 15个操作权限点，与PRD矩阵一一对应 |
| `PERMISSION_MATRIX` 常量 | `Record<Operation, Role[]>`，集中定义每个操作允许的角色 |
| `hasPermission()` 函数 | 工具函数，判断角色是否拥有操作权限，前后端共享 |
| `rbacMiddleware(Operation)` | 后端中间件，基于操作枚举自动查矩阵校验 |
| `rbacRolesMiddleware(...Role[])` | 兼容旧用法的角色白名单中间件 |
| `checkPermission(Operation)` | 前端auth store方法，控制按钮/菜单显示 |

**设计决策**：采用方案B（权限常量映射）而非方案A（数据库权限表），因为MVP阶段权限固定不变，无需动态管理。

## 四、交流过程复盘与改进建议

### 4.1 做得好的地方

1. **规范意识强**：主动建立安全规范、文件命名规范、AI协作规范，形成可复用的项目标准
2. **同步更新机制**：每次设计变更都同步更新PRD + 设计方案 + 版本记录，保持文档一致性
3. **增量迭代**：复杂任务拆成小步骤逐步推进，每步可验证
4. **安全前置**：编码前先制定安全规范，编码中严格遵守（无明文密码、RBAC二次校验等）

### 4.2 可以改进的地方

#### 1. 需求变更频繁，缺少一次性确认
- **现象**：角色权限、状态流转等核心设计反复修改多轮（火车创建员存废、技术经理权限、节点时间限制等）
- **改进**：每个模块开始前，用清单列出所有待确认点，一次性确认后再动手

#### 2. 编码阶段被文档修改打断
- **现象**：已开始写代码，用户仍在提设计方案修改，导致代码和文档反复同步
- **改进**：严格执行设计审核→冻结→编码的流程，设计阶段所有修改完成后再动代码

#### 3. 环境问题处理不够前置
- **现象**：编码完成后才发现沙箱没有Docker和PostgreSQL，无法做完整联调验证
- **改进**：每个Task开始前先检查运行环境，不可用时提前告知

#### 4. 长时间等待期间缺少进度反馈
- **现象**：pnpm install因网络慢耗时较长，用户等待期间失去耐心
- **改进**：长时间操作先在后台跑，同时告知预计等待时间

#### 5. Git操作意图不够明确
- **现象**：用户说"提交吧"时，有时本地已提交过无变更，需要二次确认是否推送；revert操作后又想回退，导致多次来回
- **改进**：执行git操作前，先简述当前状态和即将执行的操作，让用户确认

#### 6. PRD文档组织策略摇摆
- **现象**：PRD先拆分为Task0~3子文档，后又合并回统一文档，来回2次
- **改进**：文档组织策略在项目启动时一次性约定，避免中途反复

### 4.3 改进建议（后续协作规范）

| 编号 | 建议 | 具体做法 |
|------|------|----------|
| 1 | 需求批量确认 | 每个模块开始前，用清单列出所有待确认点，一次性确认后再动手 |
| 2 | 设计冻结机制 | 设计方案输出后标注"待审核/已冻结"，冻结后不再接受设计变更 |
| 3 | 环境前置检查 | 每个Task开始前先检查依赖环境，不可用时提前告知 |
| 4 | 进度可视化 | 长时间操作显示进度或预估时间，减少等待焦虑 |
| 5 | 变更影响评估 | 修改PRD/设计方案时，主动列出受影响的文件和代码 |
