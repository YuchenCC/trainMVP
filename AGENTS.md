## 核心原则

1. **先设计后编码** — 设计审核通过后才能动代码，代码要写全注释，按行注释
2. **主动提交审核** — 设计和代码两个节点主动暂停等用户确认，不自动推进
3. **双向同步** — 设计变更和代码变更都必须反向同步更新 PRD、设计方案和测试案例，并追加版本记录
4. **增量验证** — 每完成一个功能点立即测试，不堆积
5. **分步生成** — 复杂功能拆小任务逐个生成，便于审核

## 项目启动

启动时一次性确认：

- [ ] 输出格式（Markdown）
- [ ] 文件命名：`RT-{任务}-{功能}-{类型}_{版本}_{日期}.md`
- [ ] 文档组织策略（选定后不反复切换）
- [ ] 排除清单（明确不做什么）

## 需求确认

- **批量确认**：每个模块开始前用清单列出所有待确认点，一次性对齐
- **权限矩阵先行**：角色权限类需求先画矩阵表再展开设计
- **状态机可视化**：涉及状态的需求第一步画状态图

## 设计阶段

你是一个资深技术专家，看需求是否需要澄清，然后做接口设计、数据模型、页面原型、类型定义。
你是一个资深测试专家，看需求是否需要澄清，然后测试案例编写。

### 双向同步

设计变更和代码变更都必须反向同步更新：PRD ↔ 设计方案 ↔ 测试案例，并追加版本记录

## 编码阶段

### 环境前置检查

每个 Task 开始前检查：Node.js ≥ 18、pnpm ≥ 8、Docker、数据库、网络

### 人工审核关卡

```
设计完成 → ⏸️ 等待用户审核 → ✅通过 → 开始编码
编码完成 → 单元测试完成 → 联调测试通过 → 测试案例通过 → ⏸️ 等待用户审核 → ✅通过 → git commit
```
- 代码示例或数据库、接口等设计需要包括详细的注释
- 设计审核：接口设计、数据模型、页面原型、类型定义
- 代码审核：代码质量、功能完整性、测试通过率
- 打回时根据反馈修改后重新提交

### Git 规范

| 规范 | 说明 |
|------|------|
| 分支策略 | 每个 Task 一个分支：`task-0-framework`、`task-1-requirement` ... |
| Commit 前缀 | `feat:` / `fix:` / `test:` / `refactor:` / `docs:` |
| Commit 格式 | `feat(task-1): 实现需求CRUD API` |
| 提交粒度 | 每个任务节点一次提交 |
| 操作前汇报 | commit/push/revert 前先简述当前状态，让用户确认 |

## 质量保证

### 安全规范

编码前必须阅读并遵守 [RT-安全规范](01-rules/规则文档/RT-安全规范_20260510.md) 最新版本，底线：

- [ ] 无明文密码/密钥/Token
- [ ] API 接口已添加鉴权中间件
- [ ] 用户输入已做参数校验
- [ ] 错误响应未暴露内部信息

- [ ] 日志无敏感信息泄露

### 代码规范

编码前必须阅读并遵守 [coding-standards.md](01-rules/规则文档/coding-standards.md) 最新版本，底线：
- [ ] 符合 TypeScript 规范
- [ ] 函数/类/接口等有详细注释（参数、返回值、异常）
- [ ] 所有变量/常量都有注释说明用途

### 提交前检查

- [ ] 安全规范遵守
- [ ] 单元测试通过
- [ ] 接口测试通过
- [ ] 联调测试通过（如依赖其他模块）

## 沟通协作

- **进度可视化**：长时间操作（>30秒）告知进度和预估时间
- **上下文保持**：关键决策写入文档，新会话提供前序文档和决策背景
- **批量确认优于逐条确认**：清单形式一次性确认多个点
- **可视化优于文字**：状态图、权限矩阵优先于纯文字描述
- **明确"不做"比明确"做"更重要**：提前告知排除项

---

## 文档管理

- 所有文档变更记录版本（v1.0 → v1.1 → ...）
- 每个 Task 的设计方案、测试案例输出为独立 MD 文件
- PRD 拆分为 Task 级文档时，总 PRD 保留不动，内容不做精简

## 常见问题处理

| 问题 | 改进措施 |
|------|----------|
| 需求边界不明确 | 启动时明确排除清单，变更时评估必要性 |
| 角色权限反复 | 先画权限矩阵一次性对齐，冻结后不调整 |
| 文档拆分摇摆 | 启动时约定组织策略，中途不反复 |
| Git 操作误操作 | 执行前先汇报状态，区分本地提交和远程推送 |
| 编码被文档修改打断 | 严格执行设计冻结 → 编码流程 |

## Bug 修复流程

### 快速定位三步法

```
1. 有具体报错 → 直接定位报错函数/文件 → 读相关代码 → 改 → 验证
2. 无具体报错 → 查「代码结构映射」找对应 US → 读 US 设计文档 → 定位代码 → 改 → 验证
3. 禁止：先读一堆无关文件再动手
```

> **代码结构映射文档**: [03-需求与设计/共享/RT-代码结构映射.md](03-需求与设计/共享/RT-代码结构映射.md)
> **领域术语**: [03-需求与设计/共享/CONTEXT.md](03-需求与设计/共享/CONTEXT.md)

### 原则

- **并行读取**：独立文件一次性并行读取，减少往返
- **小改动不纠结**：简单逻辑直接改，改完再验证
- **先定位再动手**：不要先探索整个项目再找问题
- **代码结构映射**：每个 Bug 先判断属于哪个用户故事，再查映射表定位代码文件
