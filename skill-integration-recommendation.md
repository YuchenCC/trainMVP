# 测试治理 Skills 整合建议

## 目标

这套 TestGov skills 用于研发提测前和测试准入阶段，把本次需求、代码变更、测试案例、单测门禁、接口自动化、UI 自动化、人工验证证据和最终管理报告串起来。

目标不是让一个 skill 做完所有事情，而是让每一步职责清晰、输入清晰、输出固定，方便研发人员、测试人员、测试经理和项目经理协作。

## 真实项目约束

- 前后端通常是两个独立源码仓库。
- 后端以 Java 为主，单测使用 JUnit5，覆盖率使用 JaCoCo。
- 接口自动化以 JMeter 为主，可以在独立测试资产仓库。
- 前端使用 React 或 Vue3。
- 后端单测硬门禁：增量覆盖率 > 80%，通过率 100%。
- L2 API、L3 UI、人工验证、集成测试、前端测试都不能抵扣后端单测门禁。
- 前端默认没有单测硬门禁；只有可隔离复杂逻辑才触发前端单测建议。
- UI 自动化不放弃，但只覆盖 P0/P1 主路径、高风险权限显隐、核心跳转和关键表单提交。
- 低风险 UI 展示、文案、截图类验收可以用 SIT 人工验证替代，但必须留 evidence。

## TestGov 系列

| Skill | 定位 | 默认执行人 | 审核人 | 输出 |
|---|---|---|---|---|
| `testgov-planner` | 提测前输入清单检查，生成统一 specs | 研发负责人 / 提测负责人 / 技术 owner | 测试负责人 / 测试经理 | `reports/testgov/specs-{scope}.md` |
| `testgov-analyzer` | 基于 specs 做需求、案例、代码、测试资产覆盖分析 | 测试人员或研发负责人 | 测试负责人 + 研发负责人 | `reports/testgov/analysis-{scope}.md`、appendix |
| `testgov-unit-gate` | 后端 L1 单测硬门禁；前端单测条件触发建议 | 后端开发，前端开发按需协同 | 研发负责人 + 测试负责人 | `reports/testgov/unit-gate-{scope}.md` |
| `testgov-api-jmeter` | L2 API/JMeter 自动化覆盖治理 | 后端开发 / API 自动化测试人员 | 后端负责人 + 测试负责人 | `reports/testgov/api-jmeter-{scope}.md` |
| `testgov-ui-flow` | L3 UI 自动化路径覆盖治理 | 前端开发 / UI 自动化测试人员 | 前端负责人 + 测试负责人 | `reports/testgov/ui-flow-{scope}.md` |
| `testgov-manual-evidence` | SIT 人工验证证据包和 summary | 测试人员 | 测试负责人，必要时项目经理 | `evidence/{scope}/summary.md`、`reports/testgov/manual-evidence-{scope}.md` |
| `testgov-final-report` | 聚合分层证据，生成管理层最终报告 | 测试负责人 / 测试经理 | 项目经理 / 测试经理 / 研发负责人 | `reports/testgov/final-{scope}.md` |

推荐流水线：

```text
testgov-planner
  -> testgov-analyzer
  -> testgov-unit-gate
  -> testgov-api-jmeter
  -> testgov-ui-flow
  -> testgov-manual-evidence
  -> testgov-final-report
```

## 谁来执行 planner

`testgov-planner` 应由研发负责人、提测负责人或本次需求技术 owner 执行，不建议由测试人员独自执行。

原因是分支、commit、代码影响范围、单测报告、接口变更、前端页面影响主要掌握在研发手里。测试人员和测试经理是准入审核方和后续验证方，不应该替研发猜分支和代码范围。

推荐先让各角色填写输入清单，再由研发负责人汇总给 planner：

| 输入清单 | 填写人 |
|---|---|
| 需求范围、用户故事、纳入/不纳入内容 | 产品 / 项目经理 |
| 后端仓库、分支、commit、影响模块/API、单测和覆盖率报告 | 后端开发 |
| 前端仓库、分支、commit、影响页面/路由、UI 自动化或前端测试结果 | 前端开发 |
| 测试案例、测试环境、测试账号、人工验证口径 | 测试人员 |
| 既有报告清单和缺项责任人 | 研发负责人 / 提测负责人 |

## 多仓库目录建议

治理工作区可以和源码仓库并行，不要把所有测试治理产物塞进前端或后端仓库。

```text
test-governance-workspace/
  repos/
    backend/                    # 后端源码仓库，只引用和扫描
    frontend/                   # 前端源码仓库，只引用和扫描
    test-assets/                # JMeter、跨系统 Playwright、测试数据模板，可选
    docs/                       # 需求、设计、测试案例，可选
  .agents/
    skills/                     # TestGov skills
  reports/                      # specs、analysis、L1/L2/L3/M、最终报告
  evidence/                     # SIT 人工验证证据
  config/                       # 多仓库路径、分支、命令配置，可选
```

源码仓库内部建议：

| 类型 | 建议位置 | 说明 |
|---|---|---|
| 后端 JUnit5 单测 | 后端源码仓库 | 和业务代码强耦合，参与后端 CI 和 JaCoCo |
| 前端组件/逻辑单测 | 前端源码仓库 | 仅在复杂可隔离逻辑触发时建议 |
| 单前端应用 Playwright | 前端源码仓库优先 | 主要验证当前前端应用核心旅程 |
| 跨系统 Playwright | 独立测试资产仓库优先 | 跨多个前端、后端或外部系统 |
| JMeter 接口自动化 | 独立测试资产仓库优先 | 便于维护环境变量、测试数据和跨服务链路 |
| SIT/UAT 人工证据 | 治理工作区或测试资产仓库 | 保存截图、日志、接口证据，不污染源码 |
| 测试治理报告 | 治理工作区 | 面向项目和测试管理 |

## 分析粒度

默认按用户故事拆分。一个 Task 经常有几千行代码和多个验收点，不应直接作为一个分析单元。

拆分顺序：

1. Task / 大需求 -> 用户故事。
2. 用户故事仍过大 -> 验收点或测试案例组。
3. 后端范围 -> API 组、service/module、变更方法组。
4. 前端范围 -> 页面路由、用户流程、组件组。
5. 超大文件 -> diff、变更方法/组件、附近上下文。

建议阈值：

| 输入类型 | 处理建议 |
|---|---|
| 单个分析单元相关代码文件 > 20 个 | 暂停并拆分 |
| 单个代码文件 <= 800 行 | 可按需整文件读取 |
| 单个代码文件 800-1200 行 | 先读结构和 diff |
| 单个代码文件 > 1200 行 | 只读变更方法/组件和附近上下文 |
| 需求或测试案例文档 > 2000 行 | 按用户故事、验收点或案例组拆分 |

## 分层策略

| 层级 | 适用内容 | 产物 |
|---|---|---|
| L1 后端单测 | 业务规则、状态流转、参数校验、异常分支、金额/时间计算、mapper/converter、service 分支 | `unit-gate-{scope}.md` |
| L1 前端单测建议 | 表单校验、权限显隐、分页排序筛选、金额/日期计算、hook/composable、工具函数、状态管理 | 写入 unit-gate 建议，不作为默认硬门禁 |
| L2 API/JMeter | 接口契约、鉴权、入参校验、关键状态变化、错误码、幂等、跨服务链路 | `api-jmeter-{scope}.md` |
| L3 UI 自动化 | P0/P1 用户流程、关键页面跳转、核心按钮、跨页面联动 | `ui-flow-{scope}.md` |
| M 人工验证 | 低频 UI 展示、SIT 截图、日志证据、需人工判断的业务状态 | `evidence/{scope}/summary.md` |

## 后端单测门禁

后端 L1 是默认硬门禁：

```text
后端增量单测覆盖率 > 80%
后端单测通过率 = 100%
```

必须单独展示，且不能被 L2/L3/人工验证抵扣。

如果没有覆盖率报告或单测执行结果，只能写“阻塞/待补”，不能写通过。

## 前端单测口径

前端不默认要求单测覆盖率门禁。建议只在这些情况补前端单测：

- 表单校验、字段格式化。
- 权限显隐、按钮状态。
- 分页、排序、筛选。
- 金额、日期、数量计算。
- React hook、Vue composable。
- 工具函数、状态管理。

纯展示、简单表格、普通弹窗、文案展示，优先走 L3 UI 或人工验证。

## 人工验证证据

只有用户确认要人工验证时，才由 `testgov-manual-evidence` 创建轻量证据包：

```text
evidence/{scope}/
  summary.md
  screenshots/
  api/
  logs/
```

默认不创建 `db/`，也不要求数据库地址或账号密码。

只有资金、支付、退款、结算、库存、容量、关键状态、审计合规、异步最终一致性等场景，才要求人工补充脱敏 DB 前后截图。

人工步骤应控制在 3 到 5 步：操作、截图、保存接口/网络证据、保存日志、必要时保存 DB 脱敏截图。

## 人工审核节点

| 节点 | 是否必须审核 | 主责人 | 审核内容 |
|---|---|---|---|
| 输入清单完整性 | 必须 | 研发负责人 / 提测负责人 | 需求范围、前后端代码范围、测试案例、既有报告是否齐全 |
| specs 范围确认 | 必须 | 研发负责人 / 提测负责人 | 分支、commit、仓库路径、用户故事、纳入/不纳入范围 |
| 测试案例缺失确认 | 必须 | 测试负责人 | 是否允许基于需求和代码生成临时策略 |
| 口径差异确认 | 必须 | 产品 / 需求负责人 | 用户故事、设计、测试案例、代码实现不一致时以哪个为准 |
| 后端 L1 单测门禁 | 必须 | 后端开发 / 后端负责人 | 覆盖率是否 > 80%，通过率是否 100%，未覆盖风险是否可接受 |
| 前端单测触发判断 | 涉及前端时建议 | 前端开发 / 前端负责人 | 是否存在可隔离复杂逻辑，是否需要补前端单测 |
| L2 API 覆盖结论 | 建议必须 | 后端开发 / API 自动化测试人员 | 接口契约、鉴权、参数、状态流转、失败项 |
| L3 UI 自动化取舍 | 必须 | 前端开发 / UI 自动化测试人员 | 哪些主路径必须自动化，哪些允许人工验证 |
| 人工验证证据 | 用户启用时必须 | 测试人员 | summary、截图、日志、接口证据是否完整 |
| 最终报告 | 必须 | 测试负责人 / 测试经理 | 是否可提测/上线，风险接受人是否明确 |

## 最终报告口径

`testgov-final-report` 只聚合上游证据，不重新计算、不编造结果。

最终报告必须让项目经理或测试经理看出：

- 本次需求和用户故事覆盖到了哪里。
- 代码变更对应的 L1/L2/L3/M 证据在哪里。
- 后端单测门禁是否通过。
- API/UI/人工验证是否覆盖关键路径。
- 未覆盖风险是什么，谁接受。
- 最终建议是可提测、有条件提测、不建议提测、可上线或不建议上线。
