## 2. 项目背景与目标

### 2.1 项目背景

随着企业系统数量增加，软件发布逐渐从单系统发布转向多系统协同发布。多个系统之间存在需求依赖、投产窗口、测试节奏、团队容量和版本范围等协同问题。传统需求管理工具通常以单系统、单需求、单任务为中心，缺少跨系统统一发布计划能力，导致发布管理中容易出现需求范围失控、依赖关系不透明、团队容量不可见、关键节点不统一等问题。

版本火车是一种面向多系统协同发布的管理模式，通过固定发布周期、统一关键节点、明确纳版机制和容量约束，实现多个系统在统一节奏下协同交付。

本项目建设“版本火车需求管理系统”，用于支撑需求从提出、评审、就绪、纳版、变更、投产到回滚/取消的全生命周期管理，并在纳版阶段引入AI辅助排期建议能力，提升发布计划编排效率和风险识别能力。

### 2.2 项目目标

MVP阶段目标是完成一个可演示、可落地、可继续扩展的版本火车需求管理系统，重点实现以下目标：

1. 建立统一需求池，支持需求录入、分类、评审、状态流转、依赖关系和已纳版需求变更管理。
2. 建立版本火车计划机制，支持版本火车创建、搭载系统管理、团队容量管理和关键节点管控。
3. 支持火车管理员基于需求优先级、工作量点数、团队容量和跨系统依赖进行纳版决策。
4. 引入AI排期建议能力，输出建议纳入、建议延期、依赖风险和容量预估结果。
5. 形成适合AI Coding实施的清晰需求基线，为后续概要设计、详细设计和开发任务拆解提供依据。

### 2.3 MVP范围

| 模块 | MVP范围 |
|---|---|
| 需求池管理 | 需求录入、分类、评审、状态流转、已纳版需求管理、变更回退、依赖关系管理、需求列表与详情 |
| 版本火车计划 | 版本火车创建、搭载系统配置、团队容量配置、关键节点管理、日历视图；系统主数据采用内置静态字典 |
| 纳版管理 | 手动纳版、AI辅助纳版、容量校验、依赖校验、移除、投产、回滚 |
| AI排期建议 | 基于容量、优先级、工作量和依赖关系生成纳版建议、延期建议和风险提示 |
| 权限控制 | 基于角色的功能权限和关键数据操作权限；用户和角色采用内置演示数据 |
| 操作审计 | 记录评审、纳版、移除、变更、取消、投产、回滚等关键操作的结构化摘要 |

### 2.4 非MVP范围

以下内容不纳入MVP阶段：

1. 完整研发任务管理。
2. 测试用例管理和缺陷管理。
3. CI/CD流水线集成。
4. 企业级流程审批引擎。
5. 复杂甘特图和资源排班。
6. 多项目组合管理。
7. 复杂数据分析报表和BI看板。
8. 企业级消息通知中心。
9. 需求拆解为研发任务的自动化Agent。

## 3. 项目价值分析

### 3.1 功能价值

本系统面向多系统协同发布场景，通过版本火车机制将需求管理、发布计划、团队容量、关键节点和跨系统依赖统一纳入一个管理闭环，解决传统需求管理工具偏单系统、偏静态记录、缺少发布节奏管理的问题。

#### 3.1.1 解决多系统协同发布痛点

| 痛点 | 传统方式问题 | 系统解决方式 |
|---|---|---|
| 发布节奏不统一 | 各系统独立规划，缺少统一版本节奏，版本对齐困难 | 通过固定周期版本火车统一发布节奏 |
| 需求优先级混乱 | 排期依赖会议讨论和主观判断，缺少容量依据 | 结合优先级、工作量点数和团队容量进行纳版决策 |
| 跨系统依赖失控 | 依赖关系依赖人工沟通，容易遗漏且不可追溯 | 支持跨系统前置依赖配置，并在纳版时自动检查 |
| 团队容量不可见 | 无法量化每个系统团队的承载能力 | 通过工作量点数、可用容量、已分配容量和容量使用率实现可视化管理 |
| 关键节点不清晰 | 需求冻结、封板、投产缺少统一时间基准 | 自动推算统一纳版、统一封板、统一投产节点 |
| 纳版后变更失控 | 需求纳版后继续修改，导致范围、容量和测试边界变化 | 已纳版需求变更自动回退未就绪，重新评审和重新纳版 |

#### 3.1.2 管理效率提升

| 维度 | 传统方式 | 版本火车方式 | 预期提升 |
|---|---|---|---|
| 发布规划 | 人工协调，周期不固定 | 固定周期，关键节点自动推算 | 规划效率提升约60% |
| 需求排期 | 会议讨论，主观判断 | AI辅助排期 + 人工确认 | 排期准确率提升约40% |
| 容量管理 | 无量化估算 | 工作量点数 + 实时容量监控 | 资源利用率提升约30% |
| 依赖管理 | 口头沟通，不可追溯 | 系统关联 + AI自动检查 | 依赖风险降低约50% |
| 变更控制 | 纳版后变更依赖人工同步 | 已纳版变更回退 + 操作审计 | 降低版本范围失控风险 |

> 上述指标为MVP阶段目标化收益假设，用于竞赛展示和后续试点评估。正式落地后可结合实际版本数据进行校准。

### 3.2 AI能力价值

本项目的AI能力定位为“辅助决策”，不替代火车管理员的最终判断。AI能力嵌入纳版决策场景，基于结构化输入输出结果，重点提升排期效率、风险识别和决策解释性。

| AI能力 | 业务输入 | AI输出 | 人工确认点 | 价值 |
|---|---|---|---|---|
| AI排期建议 | 已就绪需求、优先级、工作量点数、团队容量 | 建议纳入本车需求清单 | 火车管理员确认纳版 | 降低排期决策成本 |
| 依赖风险识别 | 需求依赖关系、依赖需求状态、搭载系统范围 | 依赖未满足、依赖已取消、依赖系统未参车等风险 | 火车管理员判断是否强制纳版 | 降低跨系统发布风险 |
| 容量预估 | 各系统可用容量、已分配容量、候选需求点数 | 各系统容量使用率预估 | 火车管理员调整纳版范围 | 提升资源利用透明度 |
| 延期建议 | 容量不足、优先级较低、依赖未满足的需求 | 建议延期清单和原因说明 | 火车管理员确认延期 | 提升排期解释性 |

### 3.3 AI原生开发竞赛价值

本项目适合作为AI原生开发竞赛选题，原因包括：

1. **业务复杂度适中**：包含多角色、状态机、权限体系、容量计算和跨系统依赖，能够体现AI对复杂业务规则的理解与实现能力。
2. **前后端链路完整**：覆盖数据库设计、接口设计、后端业务逻辑、前端页面和日历视图，能够展示AI Coding的全链路工程能力。
3. **AI Agent场景明确**：AI排期建议不是泛化问答，而是基于结构化业务输入输出的辅助决策场景，具备可解释、可验证、可落地的特点。
4. **具备量化评估基础**：可从开发效率、代码质量、需求覆盖率、测试覆盖率、AI建议命中率等维度评估AI开发效果。
5. **可沉淀复用方法论**：项目可总结结构化PRD生成、数据建模、状态机实现、RBAC权限设计、AI Agent设计和AI辅助测试等经验。

| 产出项 | 说明 | 价值 |
|---|---|---|
| 版本火车需求管理系统 | 可演示、可试点的需求与版本计划管理工具 | 解决真实发布管理问题 |
| AI纳版模块 | 可复用的AI辅助排期与风险识别能力 | 可迁移到其他计划管理场景 |

## 4. 用户角色与职责

| 角色 | 主要职责 |
|---|---|
| 业务BA | 作为业务归属人录入需求、维护需求信息、发起评审、取消本人负责的需求 |
| 产品经理 | 辅助BA录入和编辑需求，参与需求评审，支持已纳版需求变更发起，但不具备取消需求权限 |
| 项目经理 | 参与需求评审，判断需求是否具备进入版本规划的条件 |
| 火车创建员 | 创建版本火车，定义搭载系统，配置系统成员和团队容量 |
| 火车管理员 | 管理版本火车关键节点、需求纳版、已纳版需求变更、移除、投产、回滚和容量控制 |
| 超级管理员 | 配置全局版本周期等基础参数 |
| 研发团队 | 查看版本火车计划、纳版需求、依赖关系和关键节点安排 |

### 4.1 角色边界

火车创建员负责“谁参加这趟车”，包括版本火车创建、系统搭载、团队成员和容量配置。

火车管理员负责“这趟车怎么运行”，包括关键节点、纳版、移除、投产、回滚、容量控制和已纳版变更治理。

### 4.2 MVP用户与角色来源

MVP使用内置演示用户和角色，不提供用户管理、组织架构、企业通讯录和角色授权页面。系统应提供当前用户切换能力，用于演示不同角色下的功能权限和数据权限。业务归属人、产品经理、项目经理、技术经理、开发团队和测试负责人等人员字段均从演示用户中选择。

## 6. 需求池管理

### 6.1 功能目标

需求池用于集中管理所有需求，覆盖待处理需求、未就绪需求、已就绪需求、已纳版需求、已投产需求和已取消需求。需求池支持需求录入、分类、评审、状态跟踪、已纳版需求变更回退、依赖关系维护和待纳版筛选。

### 6.2 需求录入规则

1. BA和产品经理均可新增需求。
2. 需求标题、需求描述、归属系统、优先级、工作量点数、业务归属人、产品经理为必填项。
3. 工作量点数范围为1-100。
4. 新建需求默认为草稿状态。
5. 需求可配置多个前置依赖，依赖需求可以跨系统。
6. 依赖关系作为纳版风险判断依据，不强制阻断需求录入。
7. 工作量点数由业务BA或产品经理在需求编辑阶段填写；评审通过表示该点数被接受为后续容量计算依据。

### 6.3 依赖需求规则

1. 一个需求可关联多个前置依赖。
2. 依赖需求必须是系统内已存在需求。
3. 不允许自依赖，也不允许直接或间接循环依赖；系统应在保存依赖关系时阻断并提示循环链路。
4. 当前置依赖状态为“已纳版”或“已投产”时，视为依赖已满足。
5. 当前置依赖状态为“已取消”时，系统应给出风险预警。
6. 当前置依赖未纳入当前版本火车时，AI排期建议和手动纳版时应提示风险。
7. 存在依赖风险时允许火车管理员强制纳版，但必须进行风险提示和操作记录；循环依赖属于数据一致性错误，不允许强制通过。

### 6.4 需求状态流转

#### 6.4.1 主流程

```text
草稿 → 待评审 → 已就绪 → 已纳版 → 已投产
```

#### 6.4.2 分支流程

```text
待评审 → 已拒绝 → 草稿
已纳版需求发生实质变更 → 未就绪 → 待评审 → 已就绪
已就绪 → 已纳版 → 已就绪
已纳版 → 已投产 → 已就绪
草稿/待评审/未就绪/已就绪/已拒绝/已纳版 → 已取消
```

“未就绪”用于承接已纳版需求发生变更后的回退状态，表示该需求当前不再满足直接纳版或继续执行条件，需要补充信息、重新评估影响，并重新发起评审。

### 6.5 状态说明

| 状态 | 含义 | 可转换状态 |
|---|---|---|
| 草稿 | 初始录入状态，可编辑 | 待评审、已取消 |
| 待评审 | 已提交评审，等待确认 | 已就绪、已拒绝、已取消 |
| 未就绪 | 已纳版需求发生实质变更后的回退状态，需重新补充信息和评审 | 待评审、已取消 |
| 已就绪 | 评审通过，可进入版本规划 | 已纳版、已取消 |
| 已拒绝 | 评审未通过，需修改后重新提交 | 草稿、已取消 |
| 已纳版 | 已搭载到版本火车 | 未就绪、已就绪、已投产、已取消 |
| 已投产 | 已上线发布 | 已就绪 |
| 已取消 | 需求已取消或终止 | 无 |

### 6.6 状态操作规则

| 操作 | 状态变化 | 权限 | 关键规则 |
|---|---|---|---|
| 发起评审 | 草稿/未就绪 → 待评审 | 业务归属人BA | 校验必填项完整；记录操作人和时间 |
| 评审通过 | 待评审 → 已就绪 | 产品经理、项目经理 | 记录评审人、评审时间、评审意见 |
| 评审拒绝 | 待评审 → 已拒绝 | 产品经理、项目经理 | 拒绝原因必填，最多500字 |
| 重新编辑 | 已拒绝 → 草稿 | 业务归属人BA、产品经理 | 回到草稿后可继续编辑 |
| 修改已就绪需求 | 已就绪 → 草稿 | 业务归属人BA、产品经理 | 已就绪需求如需修改需求描述、优先级、工作量点数、归属系统或依赖关系等影响评审结论的字段，应回到草稿并重新发起评审 |
| 取消需求 | 草稿/待评审/未就绪/已就绪/已拒绝/已纳版 → 已取消 | 业务归属人BA、火车管理员 | 取消原因必填，最多500字；已纳版需求取消时自动从火车移除；已投产需求不可取消，应通过回滚处理 |
| 已纳版需求变更 | 已纳版 → 未就绪 | 业务归属人BA、产品经理、火车管理员 | 变更原因必填，记录变更内容；从当前版本火车移除并释放容量；需重新评审后才能再次纳版 |

### 6.7 已纳版需求变更管理

已纳版需求仍属于需求池管理范围。为避免版本范围失控，已纳版需求发生实质变更时，必须触发状态回退。

#### 6.7.1 触发条件

以下字段发生变更时，视为实质变更：

1. 需求标题或需求描述发生影响范围的修改。
2. 优先级发生变化。
3. 工作量点数发生变化。
4. 归属系统发生变化。
5. 依赖需求新增、删除或变更。
6. 业务归属人或产品经理发生变化。
7. 火车管理员认定会影响本次版本范围、容量或投产风险的其他变更。

#### 6.7.2 流转规则

1. 已纳版需求发生实质变更后，状态由“已纳版”回退为“未就绪”。
2. 系统自动解除该需求与当前版本火车的纳版关系。
3. 系统释放该需求占用的系统容量。
4. 变更人必须填写变更原因，最多500字。
5. 系统记录变更前后关键字段摘要、变更人、变更时间和变更原因，不要求保存完整对象快照。
6. 未就绪需求需补充完整信息后重新发起评审。
7. 重新评审通过后，状态变更为“已就绪”，可再次进入纳版。

#### 6.7.3 权限规则

| 操作 | 权限 |
|---|---|
| 发起已纳版需求变更 | 业务归属人BA、产品经理、火车管理员 |
| 确认回退未就绪 | 系统自动执行 |
| 重新发起评审 | 业务归属人BA |
| 再次纳版 | 火车管理员 |

#### 6.7.4 业务价值

已纳版需求变更回退机制能够保证版本火车范围受控，避免需求在纳版后被随意修改导致容量失真、依赖失控、测试范围扩大和投产风险上升。

### 6.8 需求池页面要求

#### 6.8.1 需求列表页

需求列表页用于查询和管理需求，支持：

1. 按系统切换需求列表。
2. 按状态、创建人、版本火车筛选。
3. 展示需求标题、状态、优先级、创建人、提出时间、所属版本火车等核心信息。
4. 根据当前用户角色和需求状态动态展示操作按钮。
5. 支持在需求池中查看和筛选“已纳版”需求，避免纳版后需求脱离需求池管理。

#### 6.8.2 需求详情页

需求详情页展示需求完整信息，支持：

1. 查看基础信息、需求描述、依赖需求、操作历史。
2. 根据权限执行编辑、发起评审、重新编辑、取消、已纳版变更等操作。
3. 依赖需求以状态标识展示：依赖已满足、依赖未满足、依赖已取消。
4. 点击依赖需求可跳转到对应详情。
5. 对已纳版需求，如发生需求描述、优先级、工作量点数、依赖关系、归属系统等实质变更，系统应提示该需求将回退为“未就绪”，并要求填写变更原因。

#### 6.8.3 需求评审页

需求评审页用于产品经理和项目经理处理待评审需求，支持：

1. 查看待评审需求列表。
2. 查看需求详情。
3. 填写评审意见。
4. 执行评审通过或评审拒绝。
5. 支持批量通过和批量拒绝；批量拒绝时需统一填写拒绝原因。

## 7. 版本火车计划管理

### 7.1 功能目标

版本火车计划模块用于创建和管理固定周期发布计划，支持搭载系统、配置容量、管控关键节点、执行纳版、投产和回滚。

### 7.2 版本火车创建规则

1. 火车创建员可创建版本火车。
2. 创建时需填写版本火车名称、开始时间、版本周期和描述。
3. 版本周期由超级管理员全局配置，支持2周、3周、4周。
4. 开始时间默认最近一个周一。
5. 结束时间根据开始时间和周期自动计算。
6. 同一系统同一时间只能参与一个版本火车；时间冲突按起止日期闭区间重叠判断，已取消火车不参与冲突判断，已完成火车仍保留历史占用。

### 7.3 搭载系统与容量规则

1. 火车创建员在创建版本火车时选择本车搭载系统。
2. 每个搭载系统必须配置业务BA、产品经理、技术经理、开发团队和测试负责人。
3. 每个搭载系统必须配置本期可用点数。
4. 已纳版需求点数计入对应系统的已分配点数。
5. 容量使用率达到90%及以上时应预警。
6. 容量已满或超出时，仍允许火车管理员强制纳版，但必须二次确认。
7. 已纳版需求因变更回退为未就绪时，系统自动释放对应容量。

### 7.4 关键节点规则

版本火车包含三个关键节点：统一纳版、统一封板、统一投产。

| 节点 | 含义 | 默认时间 |
|---|---|---|
| 统一纳版 | 确定进入本车的需求清单 | 周期过半前最后一个周五 |
| 统一封板 | UAT完成后控制变更 | 投产前一周周五 |
| 统一投产 | 版本火车正式发布上线 | 周期最后一天周五 |

不同周期默认节点：

| 周期 | 统一纳版 | 统一封板 | 统一投产 |
|---|---|---|---|
| 2周 | 第1周周五 | 第2周周五 | 第2周周五 |
| 3周 | 第2周周五 | 第3周周五 | 第3周周五 |
| 4周 | 第2周周五 | 第4周周五 | 第4周周五 |

节点管控规则：

1. 节点时间由系统自动推算，火车管理员可在±3天内微调。
2. 统一纳版前，可自由搭载和移除需求。
3. 统一纳版至统一封板之间，新增纳版需火车管理员确认。
4. 统一封板后，原则上不再接受新需求，仅支持紧急变更。
5. 到达统一投产节点时，系统自动将本车所有“已纳版”需求批量变更为“已投产”。

### 7.5 纳版管理

#### 7.5.1 手动纳版

火车管理员可将已就绪需求搭载到版本火车。

规则：

1. 需求状态必须为“已就绪”。“未就绪”需求不可直接纳版，必须重新评审通过后才能纳版。
2. 需求归属系统必须在当前版本火车搭载系统范围内。
3. 需求必须填写工作量点数。
4. 系统应校验依赖是否满足，未满足时提示风险但允许强制纳版。
5. 系统应校验容量是否充足，不足时提示风险但允许强制纳版。
6. 纳版成功后，需求状态变更为“已纳版”，并关联版本火车。
7. 纳版成功后，对应系统已分配点数增加。
8. 系统记录纳版时间和操作人。

#### 7.5.2 从火车移除

火车管理员可将已纳版需求从版本火车移除。

规则：

1. 需求状态必须为“已纳版”。
2. 移除原因必填，最多500字。
3. 移除后需求状态变更为“已就绪”。
4. 移除后释放对应系统容量。
5. 系统记录移除时间、操作人和移除原因。

#### 7.5.3 确认投产

火车管理员可手动确认需求投产，也可由系统在统一投产节点自动批量投产。

规则：

1. 需求状态必须为“已纳版”。
2. 投产后需求状态变更为“已投产”。
3. 系统记录投产时间和操作人。
4. 到达统一投产节点时，系统自动批量处理本车全部已纳版需求。

#### 7.5.4 回滚

火车管理员可将已投产需求回滚。

规则：

1. 需求状态必须为“已投产”。
2. 回滚原因必填，最多500字。
3. 回滚后需求状态变更为“已就绪”。
4. 系统记录回滚时间、操作人和回滚原因。

## 8. AI排期建议

### 8.1 功能定位

AI排期建议用于辅助火车管理员进行纳版决策，不直接替代人工决策。AI输出仅作为建议，最终纳版需由火车管理员确认。

### 8.2 触发方式

火车管理员在版本火车详情页点击“AI排期建议”按钮后触发。

### 8.3 输入数据

| 输入 | 说明 |
|---|---|
| 版本火车信息 | 当前版本火车ID、周期、搭载系统 |
| 团队容量 | 各系统可用点数、已分配点数、剩余点数 |
| 已就绪需求 | 需求标题、归属系统、优先级、工作量点数、依赖关系 |
| 依赖需求状态 | 依赖需求是否已纳版、已投产、已取消或未满足 |

### 8.4 建议逻辑

1. 优先检查需求依赖关系。
2. 依赖已取消的需求标记为风险。
3. 依赖未满足的需求建议延期或提示需联动纳入依赖需求。
4. 按优先级排序：P0 > P1 > P2 > P3。
5. 同优先级需求按工作量点数从小到大排序。
6. 按系统维度结合剩余容量进行分配。
7. 容量充足的需求建议纳入本车。
8. 容量不足的需求建议延期。
9. 同一系统内采用“可装则装”策略：候选需求按规则排序后逐个判断容量，放不下的需求建议延期，但不阻塞后续仍可放入剩余容量的需求。
10. 依赖联动建议只提示火车管理员人工评估，不自动将依赖需求加入建议纳入清单。
11. 若依赖需求所属系统未参与当前版本火车，应提示火车创建员补充搭载系统或调整计划。
12. 未就绪需求不纳入AI排期候选范围，仅在风险说明中提示需先完成评审。

### 8.5 输出结果

AI排期建议页应输出：

1. 建议纳入本车需求清单。
2. 建议延期需求清单及延期原因。
3. 依赖联动建议。
4. 依赖风险提示。
5. 各系统容量使用率预估。
6. AI总结建议。

### 8.6 人工确认

火车管理员可在AI建议结果中勾选部分或全部建议纳入需求，并点击确认纳版完成批量纳版。

### 8.7 AI能力边界

1. AI只提供排期建议，不自动变更需求状态。
2. AI不直接执行纳版、延期、取消、移除或回滚操作。
3. AI建议必须展示原因，便于火车管理员判断。
4. AI建议结果允许人工调整。
5. AI建议结果和人工确认结果均应记录，便于后续复盘。

## 13. 工程设计输入

本章用于将PRD中的业务规则转化为后续技术设计和Coding时可直接引用的工程输入。后续详细设计应优先保证本章约束被实现和测试覆盖。

### 13.1 领域边界与模块拆分

MVP建议拆分为以下深模块。模块之间通过稳定的领域接口协作，避免把状态流转、容量计算、权限判断和AI建议逻辑散落在页面或接口层。

| 模块 | 核心职责 | 主要输入 | 主要输出 |
|---|---|---|---|
| 需求生命周期模块 | 管理需求创建、编辑、评审、取消、已纳版变更回退和状态流转 | 当前需求、操作类型、操作者、原因/意见 | 新状态、校验结果、审计事件 |
| 版本火车模块 | 管理火车创建、周期、关键节点、搭载系统和火车状态 | 火车基础信息、版本周期、搭载系统 | 火车计划、关键节点、状态 |
| 容量模块 | 计算系统维度可用容量、已分配点数、剩余点数和容量预警 | 搭载系统容量、已纳版需求点数 | 容量快照、预警结果、超限风险 |
| 纳版模块 | 执行手动纳版、批量纳版、移除、投产、回滚 | 需求、火车、容量快照、依赖校验结果 | 纳版结果、释放/占用容量、审计事件 |
| 依赖风险模块 | 判断前置依赖是否满足、取消、未参车或存在联动纳入建议 | 候选需求、依赖需求、当前火车搭载系统 | 依赖风险清单、阻断/提示等级 |
| AI排期建议模块 | 基于结构化规则和AI总结生成纳版建议 | 候选需求、容量快照、依赖风险、排序规则 | 建议纳入、建议延期、风险解释、容量预估 |
| 权限模块 | 判断角色是否可执行某项操作 | 用户角色、数据归属、资源状态、操作类型 | 允许/拒绝、拒绝原因 |
| 审计模块 | 记录关键业务操作和关键字段变更摘要 | 操作上下文、变更摘要、业务对象 | 操作日志、复盘数据 |
| 日历查询模块 | 聚合火车、节点和容量状态，供日历展示 | 时间范围、火车计划、容量快照 | 日历事件、节点标识、预警标识 |

### 13.2 核心不变量

以下不变量必须在后端业务层集中保障，前端校验只能作为体验优化。

1. 需求编号由系统生成，创建后不可修改。
2. 需求只有在“已就绪”状态下才能被纳版。
3. “未就绪”需求不能直接纳版，必须重新发起评审并通过。
4. 一个需求同一时间最多关联一个未完成的版本火车。
5. 已纳版需求发生实质变更时，必须自动解除纳版关系、释放容量并回退为“未就绪”。
6. 取消已纳版需求时，必须同步从版本火车中移除并释放容量。
7. 容量不足、依赖未满足和依赖系统未参车都不是硬阻断，但必须产生风险提示和审计记录。
8. 任何强制纳版都必须记录操作者、时间、风险类型和确认说明。
9. 已投产需求回滚后回到“已就绪”，不自动恢复原版本火车关联。
10. 操作日志不可被业务用户修改或删除。

### 13.3 状态机设计要求

需求状态机应作为独立业务能力实现，所有接口和页面操作都通过统一状态机执行。状态机需要输出三类结果：是否允许、状态变化、伴随副作用。

| 事件 | 前置状态 | 目标状态 | 伴随副作用 |
|---|---|---|---|
| submitReview | 草稿、未就绪 | 待评审 | 校验必填项，记录提交日志 |
| approveReview | 待评审 | 已就绪 | 记录评审意见和评审人 |
| rejectReview | 待评审 | 已拒绝 | 校验拒绝原因，记录评审日志 |
| reopenRejected | 已拒绝 | 草稿 | 允许继续编辑 |
| editReadyRequirement | 已就绪 | 草稿 | 修改影响评审结论的字段后需重新评审 |
| includeInTrain | 已就绪 | 已纳版 | 关联火车，占用容量，记录依赖和容量风险 |
| removeFromTrain | 已纳版 | 已就绪 | 解除火车关联，释放容量，记录移除原因 |
| changeIncludedRequirement | 已纳版 | 未就绪 | 解除火车关联，释放容量，记录变更差异和原因 |
| releaseToProduction | 已纳版 | 已投产 | 记录投产时间和操作人 |
| rollbackProduction | 已投产 | 已就绪 | 校验回滚原因，记录回滚日志 |
| cancelRequirement | 草稿、待评审、未就绪、已就绪、已拒绝、已纳版 | 已取消 | 已纳版时释放容量，记录取消原因；已投产需求不可取消 |

版本火车状态MVP可先保留“计划中、进行中、已完成、已取消”。状态触发规则为：创建后为计划中，到达开始时间或首次纳版后可进入进行中；当前日期不早于统一投产节点，且本车不存在仍处于“已纳版”状态的需求时，系统提示可完成；火车管理员手动确认后版本火车变更为“已完成”。MVP不做统一投产后的自动完成。

### 13.4 数据模型设计要求

数据库设计应围绕以下聚合建模，字段名和表名可在详细设计中确定，但关系和约束必须保留。

| 聚合/实体 | 必备关系与约束 |
|---|---|
| Requirement | 关联归属系统、业务归属人、产品经理、当前状态、当前版本火车；保存优先级、点数、类型、来源 |
| RequirementDependency | 维护需求到前置需求的多对多关系；禁止自依赖和任何可检测到的循环依赖；保存依赖关系时应提示循环链路 |
| ReleaseTrain | 保存周期、起止时间、关键节点和状态；同一系统同一时间只能参与一个未完成火车 |
| TrainSystem | 关联版本火车和系统，保存本期角色配置和容量 |
| CapacitySnapshot | 可由纳版关系实时计算，也可保存快照；若保存快照，必须保证与纳版关系一致 |
| TrainRequirement | 维护需求与版本火车的纳版关系、纳版时间、纳版人、强制纳版风险说明 |
| OperationLog | 记录对象类型、对象ID、操作类型、操作人、操作时间、原因/意见、前后状态、关键字段变更摘要、风险确认信息 |
| AiScheduleRun | 记录一次AI建议的输入摘要、输出结果、操作人、生成时间和后续人工确认结果 |
| GlobalConfig | 保存版本周期等全局配置，MVP优先支持2周、3周、4周 |

### 13.5 接口契约范围

后续接口设计至少覆盖以下业务命令和查询。命令接口应返回业务结果、状态变化、风险提示和审计追踪ID；查询接口应返回页面需要的聚合视图，避免前端拼装复杂业务规则。

| 能力 | 接口意图 |
|---|---|
| 需求查询 | 查询需求列表、需求详情、依赖状态、操作历史 |
| 需求命令 | 新增、编辑、发起评审、通过、拒绝、重新编辑、取消、已纳版变更 |
| 火车查询 | 查询火车列表、火车详情、搭载系统、关键节点、容量快照 |
| 火车命令 | 创建火车、配置搭载系统、配置容量、调整关键节点、取消火车、完成版本火车 |
| 纳版命令 | 手动纳版、批量纳版、从火车移除、确认投产、批量投产、回滚 |
| AI建议 | 生成排期建议、查询建议结果、确认部分/全部建议纳版 |
| 日历查询 | 按时间范围查询火车事件、关键节点和容量预警 |
| 配置管理 | 查询和修改版本周期配置 |

### 13.6 前端设计要求

前端页面应围绕“列表筛选、详情决策、批量处理、风险确认”组织，不做营销式首页。MVP优先交付可操作后台管理体验。

1. 需求列表页需要清晰区分状态、系统、优先级和所属火车，操作按钮按权限和状态动态展示。
2. 需求详情页需要把基础信息、依赖、状态历史和可执行操作放在同一决策上下文中。
3. 版本火车详情页需要以系统维度展示容量、纳版需求和关键节点。
4. 纳版管理页需要同时展示候选需求、已纳版需求、容量预警和依赖风险。
5. AI排期建议页需要把“建议纳入”“建议延期”“风险原因”“容量变化”分区展示，支持人工勾选确认。
6. 日历视图以周/月切换展示火车周期、节点和容量预警，点击事件进入对应详情。
7. 所有强制操作、取消、移除、回滚和已纳版变更都必须要求填写原因。

### 13.7 AI排期实现边界

AI排期建议应采用“确定性规则引擎 + AI解释生成”的边界。

1. 确定性规则负责候选范围过滤、优先级排序、容量计算、依赖风险识别和建议分类。
2. 建议纳入和建议延期结果完全由确定性规则引擎生成。
3. AI负责把结构化结果转化为可读总结、风险解释和延期说明，不改变规则引擎输出的建议结果。
4. AI不得直接决定状态变更，不得绕过权限、容量和状态机校验。
5. AI输入必须是脱敏且结构化的数据，避免把无关业务文本直接送入模型。
6. AI输出必须保留结构化字段，不能只返回自然语言。
7. 当AI不可用时，系统应仍能基于规则引擎输出基础纳版建议。

建议输出结构至少包含：

| 字段 | 说明 |
|---|---|
| recommendedRequirementIds | 建议纳入本车的需求ID集合 |
| deferredRequirementIds | 建议延期的需求ID集合 |
| dependencyRisks | 依赖风险清单 |
| capacityForecast | 各系统纳版后的容量预估 |
| reasons | 每个建议项的原因说明 |
| summary | AI总结建议 |

### 13.8 测试决策

MVP测试应优先覆盖外部可观察行为，不测试内部实现细节。核心业务规则建议用单元测试覆盖，跨模块流程用集成测试覆盖，关键页面用端到端测试或组件交互测试覆盖。

| 测试对象 | 重点用例 |
|---|---|
| 需求状态机 | 合法流转、非法流转、取消、拒绝后重编辑、已纳版变更回退 |
| 容量模块 | 纳版占用、移除释放、取消释放、变更回退释放、90%预警、超限风险 |
| 依赖风险模块 | 依赖已满足、未满足、已取消、依赖系统未参车、自依赖和循环依赖阻断 |
| 纳版模块 | 手动纳版、强制纳版、批量纳版、移除、投产、回滚 |
| 权限模块 | 不同角色在不同状态下的允许和拒绝结果 |
| AI建议模块 | 候选过滤、排序、延期原因、容量预估、AI不可用降级、AI不改变规则建议结果 |
| 审计模块 | 关键操作均产生结构化摘要日志，日志包含原因、前后状态、关键字段变更摘要和风险确认信息 |
| 页面流程 | 需求录入到纳版、已纳版变更回退、AI建议确认纳版、日历跳转详情 |

### 13.9 任务拆解建议

后续Coding可按以下垂直切片推进，每个切片都应交付可运行、可验证的端到端能力。

1. 基础数据与权限骨架：用户角色、系统字典、全局版本周期、权限判断。
2. 需求池闭环：需求创建、编辑、列表、详情、评审和基础状态机。
3. 版本火车骨架：创建火车、配置搭载系统、配置容量、自动计算关键节点。
4. 手动纳版闭环：候选需求、容量校验、依赖风险、纳版、移除和审计。
5. 已纳版变更治理：实质变更识别、回退未就绪、释放容量、重新评审。
6. 投产与回滚：单需求投产、批量投产、回滚和对应日志。
7. AI排期建议：规则引擎、AI总结、建议结果页、人工确认批量纳版。
8. 日历视图：火车周期、关键节点、容量预警和详情跳转。
9. 验收硬化：补充异常路径、权限边界、测试覆盖和演示数据。

## 14. MVP验收标准

验收应以可操作场景为主，每条验收标准都需要能通过页面操作、接口调用或自动化测试验证。

1. 能创建版本火车，并配置搭载系统、系统成员和团队容量。
2. 能录入需求，并完成草稿、待评审、未就绪、已就绪、已拒绝、已纳版、已投产、已取消等状态流转。
3. 非法状态流转会被拒绝，并给出明确错误原因。
4. 能配置跨系统依赖需求，并在纳版和AI建议时展示依赖风险。
5. 能基于系统容量进行纳版管理，并在容量使用率达到90%或超出容量时提示预警。
6. 火车管理员能在存在容量或依赖风险时强制纳版，且系统记录风险确认和操作日志。
7. 能通过AI排期建议生成纳版清单、延期清单、依赖风险、容量预估和总结说明。
8. 火车管理员能基于AI建议勾选部分或全部需求，并批量确认纳版。
9. 能在版本火车详情或纳版管理页完成移除、投产、批量投产和回滚操作。
10. 已纳版需求发生实质变更时，能自动回退为未就绪，解除火车关联，并释放对应版本火车容量。
11. 已取消需求不可继续流转；已取消的已纳版需求会自动释放容量；已投产需求不可取消，只能回滚。
12. 能通过日历视图查看版本火车周期、统一纳版、统一封板、统一投产和容量预警。
13. 不同角色只能看到或执行其权限范围内的操作。
14. 关键操作均有操作日志，日志包含操作人、时间、对象、原因和变更摘要。
15. 系统满足基本性能、安全和浏览器兼容要求。

## 16. 待技术设计澄清项

以下问题不阻塞MVP PRD，但应在概要设计或详细设计阶段定稿：

1. 系统字典采用内置静态字典，不提供系统管理页面；每趟版本火车中的角色和容量配置保存在搭载系统中。
2. 工作量点数是否允许小数，当前PRD按1-100正整数处理。
3. 同一系统同一时间只能参加一个版本火车时，按火车起止日期闭区间计算重叠；已取消火车不参与冲突判断，已完成火车仍保留历史占用。
4. 循环依赖在保存需求依赖关系时阻断；MVP不允许自依赖和任何可检测到的直接或间接循环依赖。
5. 自动批量投产由定时任务触发，还是由管理员在投产日手动触发。
6. AI建议使用真实模型接口还是演示环境中的模拟AI服务。
7. 审计日志MVP采用结构化摘要记录，不做完整历史版本回放；是否导出后续扩展，MVP仅查询展示。
8. 容量快照采用实时计算还是保存历史快照，若要支持复盘建议保存AI建议运行时快照。
