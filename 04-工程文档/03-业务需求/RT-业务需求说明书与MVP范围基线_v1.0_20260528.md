# 业务需求说明书与MVP范围基线

**版本号**: v1.0  
**日期**: 2026-05-28  
**对应 Issue**: ISSUE-003  
**需求基线**: 总 PRD v2.11 + Task1/2/3/4 当前文档与代码实现

## 一、文档目标

本文档将版本火车需求管理系统的业务需求整理为单一可信基线，供竞赛评审、后续 AI 接手和工程文档编写使用。本文档以最新总 PRD v2.11、Task 级文档和当前 `release-train` 代码为准。

## 二、产品定位

版本火车需求管理系统用于管理多系统协同发布过程中的需求池、版本火车计划、团队容量、关键节点、智能纳版和投产回滚。系统的核心价值是将需求从“静态记录”推进到“可规划、可纳版、可追踪、可审计”的发布计划管理。

## 三、目标用户与角色

| 角色 | 主要职责 |
|------|----------|
| 业务BA | 录入需求、维护业务归属、发起评审、UAT通过、查看自己的待办 |
| 产品经理 | 辅助录入和编辑需求，参与需求信息完善 |
| 项目经理 | 评审需求、审批紧急变更、查看项目级待办 |
| 技术经理 | 推进开发完成、参与子状态管理 |
| 测试经理 | 推进 SIT 通过、审批紧急变更 |
| 火车管理员 | 创建火车和班次、配置搭载系统、纳版、移除、投产、回滚、完成班次 |
| 超级管理员 | 系统级兜底权限 |

MVP 使用内置演示用户验证权限，不接入企业 SSO 和组织架构。

## 四、MVP模块范围

### 4.1 需求池管理

需求池管理覆盖需求从录入到投产前后的主要生命周期：

- 需求录入：标题、描述、归属系统、优先级、工作量点数、BA、PM、依赖需求。
- 需求编辑：草稿阶段可编辑，非草稿阶段受权限和状态限制。
- 需求评审：发起评审、评审通过、评审拒绝、重新编辑。
- 需求取消：投产前可取消，已投产只能回滚。
- 需求变更：已就绪或已纳版非封板需求显式退回草稿。
- 紧急变更：封板后变更需审批。
- 子状态流转：开发中、SIT测试、UAT测试、封板。
- 需求列表和详情：支持筛选、分页、操作按钮和操作历史。

### 4.2 版本火车与班次管理

Task2 已将早期“需求直接关联火车”的模型演进为“需求关联班次，班次属于火车”：

- 版本火车是发布计划容器。
- 一个版本火车可以有多个班次。
- 火车配置搭载系统和人员容量。
- 班次创建时保存容量快照。
- 需求纳版到具体班次。
- 班次包含统一纳版、SIT、UAT、封板、投产等关键节点。
- 支持从班次移除、确认投产、回滚和完成班次。

### 4.3 AI智能纳版

智能纳版用于辅助火车管理员做纳版决策：

- 输入：班次、候选需求、容量、优先级、依赖关系。
- 规则：优先级排序、容量检查、可装则装、依赖风险提示。
- 输出：建议纳入、建议延期、风险警告和 AI 解释。
- 确认：火车管理员人工确认后才真正纳版。

该能力是项目参加 AI 原生研发赛道的关键业务亮点。

### 4.4 统一仪表盘

Task4 将早期按角色拆分的仪表盘合并为统一 `/dashboard`：

- 所有角色访问同一页面。
- 数据根据当前用户角色动态分发。
- 展示需求状态统计、待办事项、关键时间倒计时和班次进度。
- 统计卡片支持跳转到需求列表筛选结果。

## 五、明确排除清单

| 排除项 | 说明 |
|--------|------|
| 企业 SSO | MVP 仅保留内置演示用户，SSO 后续版本支持 |
| 用户和组织管理 | 不做完整通讯录、组织架构、角色授权后台 |
| 研发任务拆解 | 已纳版后的研发协同任务暂用子状态模拟 |
| 自动依赖闭包排期 | 仅提示依赖联动，不自动纳入依赖链 |
| AI自动纳版 | AI 不直接改变业务状态，必须人工确认 |
| 完整历史版本回放 | 仅保留结构化审计摘要，不做对象快照恢复 |
| 通知中心 | 飞书、邮件、站内红点暂不纳入当前 MVP |
| 自定义仪表盘布局 | Task4 仅实现统一仪表盘和角色数据分发 |

## 六、核心业务流程

### 6.1 需求从录入到纳版

1. BA 或产品经理创建草稿需求。
2. 补齐系统、BA、PM、优先级、点数和依赖。
3. 发起评审。
4. 项目经理评审通过后进入已就绪。
5. 火车管理员在班次中选择已就绪需求。
6. 系统执行容量和依赖风险检查。
7. 火车管理员确认纳版，需求进入已纳版和开发中子状态。

### 6.2 班次从规划到投产

1. 火车管理员创建版本火车并配置搭载系统。
2. 创建班次，系统生成关键节点并保存容量快照。
3. 班次中纳入需求，占用对应系统容量。
4. 按开发、SIT、UAT、封板推进。
5. 封板后进入严格变更治理。
6. 投产节点后确认需求投产。
7. 所有需求处理完成后，火车管理员完成班次。

### 6.3 智能纳版辅助流程

1. 火车管理员在班次详情选择候选需求。
2. 点击智能纳版。
3. 后端组织班次容量、需求和依赖输入。
4. 规则引擎生成建议，AI 生成解释。
5. 前端展示建议、容量分析和风险。
6. 火车管理员选择确认项并执行纳版。
