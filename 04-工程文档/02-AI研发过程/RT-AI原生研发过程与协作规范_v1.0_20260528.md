## 一、文档目标

本文档说明版本火车需求管理系统如何以纯 AI coding 方式完成从需求到交付的工程闭环。它不是单个功能说明，而是本作品参加 AI 原生研发能力攻关赛道的核心过程证据。

本项目的 AI 原生研发不是"让 AI 写几段代码"，而是将 AI 作为持续工程协作者，参与需求澄清、文档设计、实现定位、代码生成、测试验证、问题交接、项目记忆和竞赛材料沉淀。

### 2.3 Bug 修复快速定位

AGENTS.md 内置了三步定位法 + 两个关键参考文档的路径：

> **代码结构映射文档**: `03-需求与设计/共享/RT-代码结构映射.md`

## 三、核心协作原则

| 原则 | 工程含义 | 项目落地 |
|------|----------|----------|
| 先设计后编码 | 未冻结设计前不进入实现 | `AGENTS.md` 要求设计审核通过后才能编码 |
| 文档代码双向同步 | 设计变更和代码变更必须反向更新 | PRD、设计方案、测试案例、版本记录共同维护。T5/T6 所有文档均双向同步 |
| 增量验证 | 每完成一个功能点立即验证 | Task/US 粒度测试和 handoff 记录 |
| 状态机先行 | 涉及流程先画状态流转 | 需求状态机、班次状态机、紧急变更审批、变更单 PENDING→CONFIRMED |
| 权限矩阵先行 | 涉及角色先定义权限边界 | `PERMISSION_MATRIX` 与 PRD 权限表对应 |
| 上下文可接手 | 每次阶段性工作可由新 AI 继续 | handoff、工作小结、代码结构映射、issue 化文档 |

## 四、端到端AI研发流程

### 4.1 需求智能分析

AI 首先读取项目调研、总 PRD、Task PRD、用户故事和详细设计，抽取业务对象、状态、角色、权限和排除清单。项目中最重要的领域基线是：

- `03-需求与设计/共享/版本火车需求管理系统_MVP_PRD_v2.11.md`
- `03-需求与设计/共享/CONTEXT.md`
- `AGENTS.md`（规则索引 + 代码映射引用）
- Task1~Task6 各自的 PRD、设计方案、测试案例

AI 在该阶段的关键产出不是代码，而是统一语言和边界。例如确认"纳版"不是投产，"回滚"不新增长期状态，"AI排期建议"不自动改变规则引擎结果，"AI需求审查"不替代人工评审。

### 4.2 方案设计

设计阶段由 AI 以技术专家和测试专家双重视角推进：

1. 梳理接口设计。
2. 梳理数据模型。
3. 梳理页面原型。
4. 梳理类型定义。
5. 编写测试案例。
6. 追加版本记录。

在项目规则中，设计完成后必须暂停等待用户审核。该机制使 AI 生成内容进入代码前经过人工确认，降低需求漂移和误实现风险。

### 4.3 代码实现

编码阶段遵循 TypeScript monorepo 分层：

| 层级 | 职责 |
|------|------|
| `packages/shared` | 共享枚举、类型、DTO、权限矩阵 |
| `apps/server` | Fastify 路由、业务 service、Prisma 数据访问、鉴权与审计 |
| `apps/web` | React 页面、Ant Design 交互、Zustand 状态、API service |

AI 实现代码时优先从需求映射到用户故事，再从用户故事定位文件，避免大范围无目标探索。`RT-代码结构映射.md` 是该过程的主要导航文档（当前 v1.5，覆盖 T0~T6 全部模块）。

### 4.4 AI 智能体集成（Coze）

本项目通过 Coze 平台集成了三个 AI 能力，构成完整的智能体流水线：

| 智能体 | Coze 类型 | 状态 | 核心能力 |
|--------|----------|------|---------|
| T3 智能纳版 | 工作流 | ✅ | 需求列表+容量→纳版建议+AI解释 |
| T6 AI需求审查 | 工作流 | ✅ | 需求表单→评分+问题+建议+验收条件 |
| T5 需求变更智能体 | 智能体 Bot | ✅ | 飞书群聊→检测变更→插件查询→变更单+卡片 |

**Coze SDK 集成模式**：统一封装在 `coze/index.ts`，使用 `@coze/api` SDK 的流式处理（stream），处理 PING/Message/Done/Error 四种事件。关键踩坑：SDK 已自动将参数包装为 `{ parameters: {...} }`，业务代码直接传参即可，不能双重嵌套。

**Coze 选型理由**：对比自建 LLM（DeepSeek），Coze 在工作流编排、流式响应、调试体验（debug_url）、插件生态、飞书渠道和零运维方面有显著优势。MVP 阶段免费额度充足。

### 4.5 测试验证

验证遵循"功能点完成即验证"的思路：

- 后端使用 Vitest 覆盖认证、安全、需求池、火车和班次逻辑。
- 前端使用 Vitest 覆盖需求列表交互。
- 智能纳版用专门测试案例覆盖容量、优先级、依赖、边界和 E2E 场景。
- 全项目 275 个测试用例 100% 通过。

## 七、文档反向同步机制

AI coding 项目容易出现"代码已变，文档未变"的漂移。本项目规则要求：

1. 设计变更必须同步 PRD、设计方案、测试案例，并追加版本记录。
2. 代码实现若改变业务行为，必须反向更新需求或设计说明。
3. 每个 Task 文档都保留版本记录。
4. handoff 文档必须记录已完成、未完成、下一步和相关文件。

## 八、AI原生研发风险与治理

| 风险 | 表现 | 治理方式 |
|------|------|----------|
| 领域概念漂移 | 把纳版误写成投产 | 使用 `CONTEXT.md` 约束术语，含 T5/T6 新增术语 |
| 权限误判 | 前端隐藏但后端未校验 | RBAC 中间件和 service 二次校验 |
| 文档滞后 | handoff 与代码不一致 | 工程文档以 Git 最新提交和代码为准，反向同步机制 |
| 依赖环境不稳定 | pnpm/docker 不在 PATH | 运行手册记录环境前置检查 |
| AI过度自动化 | AI 直接决定纳版 | 规则引擎决策，AI 只解释 |
| Coze SDK 参数包装 | 双重嵌套导致 Missing parameter | `coze/index.ts` 代码注释 + 架构文档说明 |
| Coze 插件网络依赖 | 飞书回调需 HTTPS | 风险登记 + 部署方案明确 |
| 前端热更新缓存 | Vite 未感知文件变更 | `.vite` 缓存清理 + kill 端口重启 SOP |
