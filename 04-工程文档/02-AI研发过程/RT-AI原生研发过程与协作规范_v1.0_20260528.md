# AI原生研发过程与协作规范

**版本号**: v1.0  
**日期**: 2026-05-28  
**对应 Issue**: ISSUE-002  
**定位**: 竞赛核心方法论文档

---

## 一、文档目标

本文档说明版本火车需求管理系统如何以纯 AI coding 方式完成从需求到交付的工程闭环。它不是单个功能说明，而是本作品参加 AI 原生研发能力攻关赛道的核心过程证据。

本项目的 AI 原生研发不是“让 AI 写几段代码”，而是将 AI 作为持续工程协作者，参与需求澄清、文档设计、实现定位、代码生成、测试验证、问题交接、项目记忆和竞赛材料沉淀。

## 二、核心协作原则

| 原则 | 工程含义 | 项目落地 |
|------|----------|----------|
| 先设计后编码 | 未冻结设计前不进入实现 | `AGENTS.md` 要求设计审核通过后才能编码 |
| 文档代码双向同步 | 设计变更和代码变更必须反向更新 | PRD、设计方案、测试案例、版本记录共同维护 |
| 增量验证 | 每完成一个功能点立即验证 | Task/US 粒度测试和 handoff 记录 |
| 状态机先行 | 涉及流程先画状态流转 | 需求状态机、班次状态机、紧急变更审批 |
| 权限矩阵先行 | 涉及角色先定义权限边界 | `PERMISSION_MATRIX` 与 PRD 权限表对应 |
| 上下文可接手 | 每次阶段性工作可由新 AI 继续 | handoff、工作小结、代码结构映射、issue 化文档 |

## 三、端到端AI研发流程

### 3.1 需求智能分析

AI 首先读取项目调研、总 PRD、Task PRD、用户故事和详细设计，抽取业务对象、状态、角色、权限和排除清单。项目中最重要的领域基线是：

- `版本火车需求管理系统_MVP_PRD_v2.11.md`
- `CONTEXT.md`
- Task1 需求池 PRD
- Task2 版本火车管理 PRD
- Task3 智能纳版测试案例
- Task4 仪表盘重构 PRD

AI 在该阶段的关键产出不是代码，而是统一语言和边界。例如确认“纳版”不是投产，“回滚”不新增长期状态，“AI排期建议”不自动改变规则引擎结果。

### 3.2 方案设计

设计阶段由 AI 以技术专家和测试专家双重视角推进：

1. 梳理接口设计。
2. 梳理数据模型。
3. 梳理页面原型。
4. 梳理类型定义。
5. 编写测试案例。
6. 追加版本记录。

在项目规则中，设计完成后必须暂停等待用户审核。该机制使 AI 生成内容进入代码前经过人工确认，降低需求漂移和误实现风险。

### 3.3 代码实现

编码阶段遵循 TypeScript monorepo 分层：

| 层级 | 职责 |
|------|------|
| `packages/shared` | 共享枚举、类型、DTO、权限矩阵 |
| `apps/server` | Fastify 路由、业务 service、Prisma 数据访问、鉴权与审计 |
| `apps/web` | React 页面、Ant Design 交互、Zustand 状态、API service |

AI 实现代码时优先从需求映射到用户故事，再从用户故事定位文件，避免大范围无目标探索。`RT-代码结构映射` 是该过程的主要导航文档。

### 3.4 测试验证

验证遵循“功能点完成即验证”的思路：

- 后端使用 Vitest 覆盖认证、安全、需求池、火车和班次逻辑。
- 前端使用 Vitest 覆盖需求列表交互。
- 智能纳版用专门测试案例覆盖容量、优先级、依赖、边界和 E2E 场景。
- 人工演示路径覆盖登录、需求创建、评审、纳版、AI建议、仪表盘。

### 3.5 交接与项目记忆

项目通过以下机制保持 AI 间上下文连续：

| 机制 | 作用 |
|------|------|
| `handoff-*.md` | 记录阶段性完成内容、遗留问题、关键决策 |
| `工作小结/` | 记录日期维度的工作过程和决策 |
| `待办事项.md` | 记录延后项和非当前范围事项 |
| `RT-代码结构映射` | 让后续 AI 按 US 快速定位代码 |
| `issues/project-docs/` | 将竞赛工程文档拆成可执行 issue |

## 四、人工审核关卡

本项目保留必要的人在回路机制：

| 关卡 | 审核内容 | 目的 |
|------|----------|------|
| 需求确认 | 模块范围、排除清单、权限矩阵、状态机 | 防止 AI 自行扩展范围 |
| 设计审核 | 接口、数据模型、页面原型、类型定义、测试案例 | 防止错误设计进入代码 |
| 代码审核 | 功能完整性、权限、安全、测试结果 | 防止未验证代码进入主线 |
| Git 操作确认 | commit、push、revert 等关键操作 | 防止误提交和误推送 |
| 竞赛材料审核 | 叙事、证据、演示路径 | 保证材料符合比赛表达 |

本次工程文档 issue 生成和执行由用户明确授权，因此每个 issue 完成后自动提交，全部完成后统一推送。

## 五、文档反向同步机制

AI coding 项目容易出现“代码已变，文档未变”的漂移。本项目规则要求：

1. 设计变更必须同步 PRD、设计方案、测试案例。
2. 代码实现若改变业务行为，必须反向更新需求或设计说明。
3. 每个 Task 文档都保留版本记录。
4. handoff 文档必须记录已完成、未完成、下一步和相关文件。

当前项目中也存在历史文档滞后现象，例如旧 handoff 仍提到 `/dashboard/ba` 和 `/dashboard/pm`，但最新代码已统一到 `/dashboard`。后续工程文档需要显式指出“以当前代码和最新 PRD 为准”。

## 六、AI原生研发风险与治理

| 风险 | 表现 | 治理方式 |
|------|------|----------|
| 领域概念漂移 | 把纳版误写成投产 | 使用 `CONTEXT.md` 约束术语 |
| 权限误判 | 前端隐藏但后端未校验 | RBAC 中间件和 service 二次校验 |
| 文档滞后 | handoff 与代码不一致 | 工程文档以 Git 最新提交和代码为准 |
| 依赖环境不稳定 | pnpm/docker 不在 PATH | 运行手册记录环境前置检查 |
| AI过度自动化 | AI 直接决定纳版 | 规则引擎决策，AI 只解释 |

## 七、竞赛展示口径

建议在竞赛中将本项目表达为：

> 一个以版本火车需求管理为业务载体、以 AI coding 为研发范式、以领域语言和状态机为工程约束、以智能纳版为 AI 能力亮点、以安全和质量体系为加分支撑的 AI 原生研发作品。

展示重点应从“功能多”转向“研发范式完整”：

- AI 如何理解需求。
- AI 如何约束设计。
- AI 如何定位代码。
- AI 如何验证结果。
- AI 如何交接上下文。
- AI 如何形成可评审的工程资产。
