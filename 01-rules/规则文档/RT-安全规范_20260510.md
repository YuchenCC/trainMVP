## 二、敏感信息管理

### 2.1 禁止明文原则

所有敏感配置项（密码、密钥、Token、API Key等）**严禁**以明文形式出现在以下位置：

- ❌ 代码文件（`.ts`、`.tsx`、`.js`、`.json`等）
- ❌ Docker Compose / Dockerfile
- ❌ 配置文件（`vite.config.ts`、`app.ts`等）
- ❌ 设计文档 / README / 注释
- ❌ Git提交记录（commit message、diff中不得包含密钥）

### 2.2 环境变量分层

| 文件 | 用途 | 是否提交Git |
|------|------|:-----------:|
| `.env.example` | 变量名+占位符模板，供新成员参考 | ✅ 提交 |
| `.env` | 本地开发默认值（仅非敏感配置） | ❌ 不提交 |
| `.env.local` | 敏感配置（密码、密钥、Token） | ❌ 不提交 |
| `.env.production` | 生产环境配置 | ❌ 不提交 |

### 2.3 .gitignore 强制规则

```gitignore
# 环境变量（必须包含）
.env
.env.local
.env.*.local

# 密钥文件
*.pem
*.key
*.p12
*.jks

# IDE
.idea/
.vscode/
*.swp

# 系统文件
.DS_Store
Thumbs.db

# 依赖
node_modules/

# 构建产物
dist/
build/
```

## 三、认证安全

### 3.1 密码存储

| 规则 | 说明 |
|------|------|
| 加密算法 | bcrypt，cost factor ≥ 10 |
| 禁止 | 明文存储、MD5、SHA1等弱哈希 |
| 传输 | 登录请求必须走 HTTPS（生产环境） |

### 3.2 JWT安全

| 规则 | 说明 |
|------|------|
| 签名算法 | HS256 或 RS256 |
| 密钥强度 | ≥ 32字符随机字符串 |
| 有效期 | Access Token 24小时，后续可引入 Refresh Token 轮转 |
| 存储 | 前端存于 `localStorage` 或 `httpOnly Cookie`（生产环境推荐后者） |
| 禁止 | Token 不得出现在 URL 参数中 |

### 3.3 SSO预留

- 生产环境走企业SSO，本地开发保留用户名/密码登录
- SSO配置（`ssoId`）仅存储第三方用户标识，不存储SSO Token

## 四、API安全

### 4.1 接口鉴权

| 规则 | 说明 |
|------|------|
| 全局鉴权 | 除 `/api/auth/login` 外，所有接口必须校验JWT |
| 角色校验 | RBAC中间件校验用户角色，无权限返回 `403` |
| 输入校验 | 所有请求参数必须校验（类型、长度、格式） |
| 防注入 | Prisma ORM参数化查询，禁止拼接SQL |

### 4.2 接口限流

| 规则 | 说明 |
|------|------|
| 登录接口 | 同一IP 5次/分钟，防止暴力破解 |
| 普通接口 | 同一用户 100次/分钟 |
| 短信/验证码接口 | 同一手机号 1次/60秒，同一IP 10次/小时，防止短信炸弹 |
| 实现方式 | Redis滑动窗口计数 |

> **短信炸弹防护**：如后续引入短信验证码功能，必须满足以下要求：
> - 同一手机号：60秒内仅可发送1次，24小时内最多5次
> - 同一IP：1小时内最多发送10条
> - 发送前必须增加图形验证码/滑块验证，防止自动化脚本滥用
> - 短信内容不得包含敏感链接，避免被运营商拦截或用于钓鱼
> - 接入运营商级短信防护（如阿里云短信防火墙）

### 4.3 错误信息

| 规则 | 说明 |
|------|------|
| 生产环境 | 不暴露堆栈信息、SQL语句、内部路径 |
| 错误响应 | 返回统一格式 `{ success: false, message: "操作失败" }` |
| 日志 | 详细错误信息仅记录到服务端日志，不返回给前端 |

## 五、数据库安全

### 5.1 连接安全

| 规则 | 说明 |
|------|------|
| 连接字符串 | 通过环境变量注入，禁止硬编码 |
| SSL | 生产环境强制SSL连接 |
| 最小权限 | 应用账号仅授予业务所需权限（SELECT/INSERT/UPDATE/DELETE），禁止DDL权限 |

### 5.2 数据保护

| 规则 | 说明 |
|------|------|
| 密码字段 | `password` 字段在API响应中不得返回 |
| 软删除 | 核心业务数据使用软删除（后续迭代引入 `deletedAt`） |
| 审计日志 | 状态变更通过 `StatusLog` 记录，保留操作人、时间、原因 |

## 六、前端安全

### 6.1 XSS防护

| 规则 | 说明 |
|------|------|
| 用户输入 | React默认转义，禁止使用 `dangerouslySetInnerHTML` 处理用户输入 |
| 富文本 | 如后续引入富文本编辑器，必须做HTML Sanitization |

### 6.2 CSRF防护

- API请求使用 `Content-Type: application/json`，天然防止传统CSRF
- 如后续引入Cookie认证，需额外增加CSRF Token

### 6.3 路由守卫

| 规则 | 说明 |
|------|------|
| 未登录 | 重定向到 `/login` |
| 无权限 | 显示403页面，不暴露路由信息 |
| 菜单渲染 | 根据用户角色动态渲染菜单，不展示无权限入口 |

## 十、常见渗透测试漏洞防护（OWASP Top 10）

### 10.1 注入攻击（Injection）

| 漏洞类型 | 风险 | 防护措施 |
|----------|------|----------|
| SQL注入 | 数据泄露、数据篡改、拖库 | Prisma ORM参数化查询，**禁止拼接SQL字符串** |
| NoSQL注入 | 同上 | Prisma内置防护，禁止使用 `$queryRaw` 拼接用户输入 |
| 命令注入 | 服务器被控 | **禁止使用 `child_process` 执行用户输入**，如必须执行则白名单校验 |
| LDAP注入 | 目录服务信息泄露 | 本项目不使用LDAP，无需额外处理 |

```typescript
// ❌ 危险：拼接用户输入
await prisma.$queryRaw`SELECT * FROM User WHERE name = '${userInput}'`;

// ✅ 安全：Prisma参数化查询
await prisma.user.findMany({ where: { name: userInput } });
```

### 10.2 越权访问（Broken Access Control）

| 漏洞类型 | 风险 | 防护措施 |
|----------|------|----------|
| 水平越权 | 用户A访问/修改用户B的数据 | 所有数据查询必须带 `userId` 过滤条件 |
| 垂直越权 | 普通用户执行管理员操作 | RBAC中间件校验角色，**服务端必须二次校验**（不依赖前端按钮控制） |
| IDOR（不安全的直接对象引用） | 遍历ID获取他人数据 | 资源访问接口校验当前用户是否有权操作该资源 |

```typescript
// ❌ 危险：仅校验登录，不校验数据归属
app.get('/api/requirements/:id', async (req, res) => {
  const req = await prisma.requirement.findUnique({ where: { id: req.params.id } });
  return req; // 任何登录用户都能查看任意需求
});

// ✅ 安全：校验数据归属或权限
app.get('/api/requirements/:id', async (req, res) => {
  const user = req.user; // 从JWT解析
  const req = await prisma.requirement.findFirst({
    where: { id: req.params.id, systemId: { in: user.accessibleSystemIds } }
  });
  if (!req) throw new ForbiddenError('无权访问该需求');
  return req;
});
```

### 10.3 身份认证失效（Broken Authentication）

| 漏洞类型 | 风险 | 防护措施 |
|----------|------|----------|
| 弱密码 | 暴力破解 | 密码复杂度校验：≥8位，含大小写+数字 |
| 暴力破解 | 账号被盗 | 登录接口限流（5次/分钟/IP），连续失败锁定账号 |
| 会话固定 | 劫持用户会话 | 登录成功后重新签发JWT，旧Token立即失效 |
| 凭证泄露 | Token被窃取 | 生产环境使用 `httpOnly + Secure + SameSite=Strict` Cookie存储Token |

### 10.4 敏感数据泄露（Sensitive Data Exposure）

| 漏洞类型 | 风险 | 防护措施 |
|----------|------|----------|
| 响应泄露 | 密码、内部ID暴露 | API响应中剔除 `password`、`ssoId` 等敏感字段 |
| 日志泄露 | 日志中包含敏感信息 | 日志脱敏：密码、Token、手机号、身份证号打码输出 |
| 传输泄露 | 中间人攻击 | 生产环境强制HTTPS（TLS 1.2+） |
| 错误信息泄露 | 暴露堆栈/SQL/路径 | 生产环境统一错误响应，详细错误仅记录服务端日志 |

```typescript
// ✅ API响应中剔除敏感字段
const safeUser = await prisma.user.findUnique({
  where: { id },
  select: { id: true, username: true, displayName: true, role: true, email: true }
  // 不返回 password、ssoId
});
```

### 10.5 XML外部实体注入（XXE）

| 风险 | 防护措施 |
|------|----------|
| 读取服务器文件、SSRF攻击 | 本项目API使用JSON格式，不解析XML。如后续需解析XML，**必须禁用外部实体解析** |

### 10.6 错误配置（Security Misconfiguration）

| 漏洞类型 | 风险 | 防护措施 |
|----------|------|----------|
| 默认凭证 | 使用默认密码被入侵 | 禁止使用默认密码，所有密码通过 `.env.local` 注入 |
| 不必要的端口/服务 | 攻击面增大 | Docker仅暴露必要端口（3000、5432、6379），生产环境关闭Swagger |
| 目录遍历 | 源码/配置文件泄露 | Fastify静态文件服务限定目录，禁止路径穿越（`../`） |
| CORS配置不当 | 恶意站点跨域访问 | CORS白名单限定允许的Origin，**禁止使用 `*` 通配符** |

```typescript
// ❌ 危险：允许所有域名跨域
await app.register(cors, { origin: true });

// ✅ 安全：白名单限定
await app.register(cors, {
  origin: ['http://localhost:5173', 'https://your-domain.com'],
  credentials: true
});
```

### 10.7 跨站脚本攻击（XSS）

| 漏洞类型 | 风险 | 防护措施 |
|----------|------|----------|
| 存储型XSS | 恶意脚本持久化，影响所有用户 | React默认转义；如需渲染HTML必须使用 `DOMPurify` 消毒 |
| 反射型XSS | 恶意链接诱导用户点击 | React默认转义；URL参数不做直接HTML渲染 |
| DOM型XSS | 前端JS动态生成恶意内容 | 避免使用 `innerHTML`、`eval()`、`document.write()` |

```typescript
// ❌ 危险：直接渲染用户输入为HTML
<div dangerouslySetInnerHTML={{ __html: userInput }} />

// ✅ 安全：消毒后再渲染
import DOMPurify from 'dompurify';
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(userInput) }} />
```

### 10.8 不安全的反序列化（Insecure Deserialization）

| 风险 | 防护措施 |
|------|----------|
| 反序列化恶意数据导致RCE | 本项目API使用JSON，不使用 `eval()` 反序列化。如后续引入文件上传，必须校验文件类型和内容 |

### 10.9 使用含已知漏洞的组件（Using Components with Known Vulnerabilities）

| 风险 | 防护措施 |
|------|----------|
| 依赖库漏洞被利用 | 定期 `pnpm audit`，高危漏洞24小时内升级；引入新依赖前检查安全公告 |

### 10.10 不足的日志与监控（Insufficient Logging & Monitoring）

| 漏洞类型 | 风险 | 防护措施 |
|----------|------|----------|
| 无操作日志 | 无法追溯安全事件 | 所有状态变更写入 `StatusLog`，登录/登出记录日志 |
| 无异常监控 | 攻击行为无法及时发现 | 生产环境接入日志监控（ELK/自建），异常登录、频繁失败触发告警 |
| 日志信息不足 | 事件复盘困难 | 日志包含：时间、操作人ID、操作类型、目标资源、IP地址 |

## 十二、检查清单

每次提交代码前，确认以下事项：

- [ ] 无明文密码/密钥/Token
- [ ] `.env.local` 未被 `git add`
- [ ] API接口已添加鉴权中间件
- [ ] 用户输入已做参数校验
- [ ] 错误响应未暴露内部信息
- [ ] 密码字段未返回给前端
- [ ] 数据查询已校验用户权限（防越权）
- [ ] 无SQL拼接（仅使用Prisma参数化查询）
- [ ] 无 `dangerouslySetInnerHTML` 或已消毒
- [ ] CORS已配置白名单（未使用 `*`）
- [ ] 无 `eval()` / `child_process` 执行用户输入
- [ ] 日志无敏感信息泄露
