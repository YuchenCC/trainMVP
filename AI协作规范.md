## 一、项目启动规范

### 1.1 格式偏好前置声明

| 项目 | 规范 |
|------|------|
| 默认文档格式 | Markdown (.md) |
| 代码文档格式 | Markdown + 代码注释 |
| 正式交付格式 | Word (.docx) 按需生成 |
| 文件命名规范 | `RT-{任务}-{功能}-{类型}_{日期}.md` |
| 目录结构 | Monorepo（apps + packages） |

**启动时一次性确认**：
- [ ] 默认输出格式
- [ ] 文件命名规则
- [ ] 文档组织策略（单一主文档 vs 模块化多文件）
- [ ] 排除清单（明确不做什么）

### 1.2 排除清单模板

```markdown
## 本次明确不做
- [ ] 需求层级（Epic/Feature/Story）- 扁平化需求
- [ ] 需求拆解模块 - 从MVP移除
- [ ] 应用/模块/功能维度 - 取消
```

## 二、需求确认规范

### 2.1 批量确认机制

**原则**：每个模块开始前，用清单列出所有待确认点，一次性确认后再动手。

**确认清单模板**：

```markdown
## 需求确认清单 - {模块名称}

### 角色权限
- [ ] 角色列表确认
- [ ] 权限矩阵确认（角色 × 操作）
- [ ] 特殊权限规则确认

### 状态流转
- [ ] 状态列表确认
- [ ] 流转路径确认
- [ ] 触发条件确认

### 功能边界
- [ ] 核心功能确认
- [ ] 排除功能确认
- [ ] 未来扩展预留
```

### 2.2 权限矩阵先行

**角色权限类需求，先画权限矩阵表，一次性对齐后再展开详细设计**。

| 操作 | 角色A | 角色B | 角色C | ... |
|------|:-----:|:-----:|:-----:|:---:|
| 操作1 | ✅ | ❌ | ✅ | ... |
| 操作2 | ❌ | ✅ | ✅ | ... |

### 2.3 状态机可视化

**涉及状态机的需求，第一步就画状态图**，用图形对齐理解，再补充文字描述。

```
[状态A] --事件1--> [状态B] --事件2--> [状态C]
   |                                      |
   └------------事件3---------------------┘
```

## 三、设计阶段规范

### 3.1 设计冻结机制

| 阶段 | 状态 | 说明 |
|------|------|------|
| 待审核 | 🟡 | 设计方案初稿，接受修改意见 |
| 已冻结 | 🔵 | 设计方案确认，不再接受设计变更 |
| 编码中 | 🟢 | 已进入编码阶段，变更走变更流程 |

**变更流程**：
1. 提出变更申请
2. 评估影响范围（文档 + 代码）
3. 确认变更必要性
4. 执行变更并同步更新所有受影响文件

### 3.2 同步更新要求

**所有设计变更必须同步更新**：
- PRD文档（含版本记录）
- 设计方案文档
- 安全规范（如受影响）
- 测试案例（如已生成）

**版本记录格式**：

```markdown
| 版本 | 日期 | 变更内容 | 变更人 |
|------|------|----------|--------|
| v1.1 | 2026-05-10 | 新增XX功能 | AI/User |
```

## 四、编码阶段规范

### 4.1 环境前置检查

**每个Task开始前执行**：

```bash
# 检查清单
- [ ] Node.js版本 >= 18
- [ ] pnpm >= 8
- [ ] Docker（如需要）
- [ ] 数据库（如需要）
- [ ] 网络连接（依赖安装）
```

**环境不可用时**：提前告知用户，协商替代方案（如SQLite代替PostgreSQL）。

### 4.2 人工审核机制

> **设计和代码提交两个关键节点必须由用户人工审核通过后才能继续**

#### 审核关卡流程

```
设计完成                    代码完成
    │                          │
    ▼                          ▼
┌──────────┐              ┌──────────┐
│ AI提交设计 │              │ AI提交代码 │
│ 供审核    │              │ 供审核    │
└────┬─────┘              └────┬─────┘
     │                          │
     ▼                          ▼
┌──────────┐              ┌──────────┐
│ ⏸️ 暂停    │              │ ⏸️ 暂停    │
│ 等待审核  │              │ 等待审核  │
└────┬─────┘              └────┬─────┘
     │                          │
     ▼                          ▼
┌──────────┐              ┌──────────┐
│ 用户审核  │              │ 用户审核  │
│ ✅通过    │              │ ✅通过    │
│ ❌打回修改│              │ ❌打回修改│
└────┬─────┘              └────┬─────┘
     │                          │
     ▼                          ▼
  通过→继续开发              通过→git commit
  打回→修改后重新提交        打回→修改后重新提交
```

#### 审核节点说明

| 审核节点 | 触发时机 | 审核内容 | 通过后动作 |
|----------|----------|----------|------------|
| **设计审核** | 每个Task设计阶段完成后 | 接口设计、数据模型、页面原型、类型定义 | 进入前后端并行开发 |
| **代码审核** | 每个提交节点完成后 | 代码质量、功能完整性、测试通过率 | git commit + push |

#### AI执行规则

1. **设计阶段**：AI完成设计后主动提交给用户审核，**不自动进入开发**
2. **代码阶段**：AI完成一个提交节点的代码后主动提交给用户审核，**不自动commit**
3. **打回处理**：用户打回时，AI根据反馈修改后重新提交审核
4. **审核超时**：无超时机制，AI始终等待用户确认

#### 编码门禁

**严格执行**：设计审核 → 冻结 → 编码

**编码过程中不接受设计变更**，变更需走变更流程。

### 4.3 代码提交策略

> **每个任务节点完成后立即提交代码，确保随时可回滚**

#### 提交时机流程

```
任务节点完成
    │
    ▼
┌──────────────┐
│ 1. 自测通过   │  本地运行验证功能正常
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 2. 单元测试   │  运行该节点的单元测试，全部通过
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 3. 提交代码   │  git commit + push
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 4. 更新日报   │  PM记录该节点已完成
└──────────────┘
```

#### Git分支与提交规范

| 规范 | 说明 |
|------|------|
| **分支策略** | 每个Task一个分支：`task-0-framework`、`task-1-requirement`、`task-2-train`、`task-3-ai`、`task-4-integration` |
| **Commit前缀** | `feat:` 新功能 / `fix:` 修复 / `test:` 测试 / `refactor:` 重构 / `docs:` 文档 |
| **Commit格式** | `feat(task-1): 实现需求CRUD API` |
| **提交粒度** | 每个任务节点一次提交，不拆分过细也不过粗 |

#### 提交节点示例

| Task | 提交次数 | 示例Commit |
|------|----------|------------|
| Task 0 | 2次 | `feat(task-0): 初始化项目脚手架和基础配置` |
| Task 1 | 5次 | `feat(task-1): 实现需求CRUD API` |
| Task 2 | 5次 | `feat(task-2): 实现系统和成员管理API` |
| Task 3 | 3次 | `feat(task-3): 实现AI排期建议和依赖检查API` |
| Task 4 | 2次 | `test(task-4): 系统集成测试（端到端）` |
| **合计** | **17次** | 平均每天1次提交 |

## 五、沟通协作规范

### 5.1 沟通渠道选择

| 方式 | 适用场景 |
|------|----------|
| 对话 | 简单修改、快速确认 |
| 文件批注 | 具体位置的复杂修改意见 |
| 清单 | 批量确认、待办事项 |

### 5.2 进度可视化

**长时间操作（>30秒）需告知进度**：

```
[进度] 正在安装依赖... 预计2-3分钟
[进度] 已下载 150/381 个包...
```

### 5.3 上下文保持

**关键决策及时写入文档**：
- 设计决策 → 写入设计方案
- 权限规则 → 写入PRD
- 技术选型 → 写入技术选型文档

**新会话启动时提供**：
- 前序文档路径
- 关键决策摘要
- 当前任务状态

## 六、文档管理规范

### 6.1 文档组织策略

**方案A：单一主文档**
- 适用：小型项目、需求简单
- 文件：PRD.md（含所有章节）

**方案B：模块化多文件**
- 适用：中大型项目、需求复杂
- 文件：PRD.md + 详细设计/*.md

**选择后不再反复切换**。

### 6.2 文件命名规范

```
RT-{任务}-{功能}-{类型}_{日期}.md

示例：
RT-T0-基础框架与数据层-设计方案_20260510.md
RT-T1-需求池管理-设计方案_20260511.md
RT-SEC-安全规范_20260510.md
```

### 6.3 版本控制

**所有文档变更记录版本**：
- PRD：v1.0 → v1.1 → v1.2
- 设计方案：v1.0 → v1.1
- 协作规范：v1.0 → v1.1

## 七、质量保证规范

### 7.1 安全规范前置

**编码前必须输出安全规范**：
- 敏感信息管理
- 认证授权机制
- 常见漏洞防护
- 安全测试要求

### 7.2 检查清单

**提交前检查清单**：

```markdown
- [ ] 无明文密码/密钥/Token
- [ ] API接口已添加鉴权中间件
- [ ] 用户输入已做参数校验
- [ ] 错误响应未暴露内部信息
- [ ] 密码字段未返回给前端
- [ ] 数据查询已校验用户权限
- [ ] 无SQL拼接（仅使用参数化查询）
- [ ] 日志无敏感信息泄露
```

### 7.3 测试验证

**每个Task完成后验证**：
- [ ] 单元测试通过
- [ ] 接口测试通过
- [ ] 联调测试通过（如依赖其他模块）

## 八、问题处理规范

### 8.1 需求边界不明确

**现象**：先设计后删除、反复纳入/移除

**改进**：
1. 启动时明确排除清单
2. 使用"明确不做"清单
3. 变更时评估必要性

### 8.2 角色权限反复

**现象**：权限规则多次调整

**改进**：
1. 先画权限矩阵表
2. 一次性对齐所有角色
3. 冻结后不再调整

### 8.3 文档拆分摇摆

**现象**：先拆后合或先合后拆

**改进**：
1. 启动时约定文档组织策略
2. 选择后不再反复

## 九、AI协作规范（7条核心规则）

### 9.1 分步骤生成

复杂功能拆分成小任务，逐个生成，避免一次性输出大量内容导致用户难以审核。

### 9.2 先设计后编码

设计完成后必须等用户审核通过，再开始编码。严格执行设计冻结机制。

### 9.3 增量验证

每完成一个功能点立即测试验证，不堆积到后期。

### 9.4 保留上下文

关键决策写入文档，避免AI遗忘。新会话启动时提供前序文档和决策背景。

### 9.5 主动提交审核

设计和代码两个节点主动暂停，等待用户确认。不自动进入下一阶段。

### 9.6 同步更新PRD

所有设计变更、代码变更都必须同步更新PRD文档，并在PRD的版本记录中追加变更历史（版本号+日期+变更内容）。

### 9.7 过程文档归档

每个Task的设计方案、测试案例必须输出为独立文档（MD格式），作为过程文档交付。

## 十、AI协作经验总结

| 经验 | 说明 |
|------|------|
| **给AI足够的上下文** | 每次新会话提供前序文档和决策背景 |
| **分步骤验证** | 复杂功能先输出框架，确认后再补充细节 |
| **明确"不做"比明确"做"更重要** | 提前告知排除项，减少不必要内容 |
| **保持对话连续性** | 关键决策及时写入文档作为"记忆" |
| **批量确认优于逐条确认** | 清单形式一次性确认多个点 |
| **可视化优于文字描述** | 状态图、权限矩阵优先于纯文字 |
